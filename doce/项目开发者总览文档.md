## 2. 项目概览

### 2.1 项目定位

这是一个以“公会组织管理 + 主播账号管理 + 待办任务闭环”为核心的全栈系统，当前主要覆盖：

1. 组织树管理

- `dev`：同时启动前后端

## 5. 后端启动流程

`backend/src/server.ts` 的启动流程可以概括为：

1. 创建上传目录 `uploads/tasks`
2. 执行任务发放表结构兼容处理 `ensureTaskAssignmentSchemaCompatibility`
3. 启动临时任务收口定时器 `startTemporaryAssignmentReconciler`
4. 创建 Express app
5. 监听 `env.PORT`

### 5.1 临时任务自动收口

服务启动时会立即执行一次临时任务收口，并每分钟执行一次：

- 超过 `deadlineAt`
- `category = TEMPORARY`
- `status = active`
- `isActive = true`

则自动更新为：

- `status = ended`
- `isActive = false`
- `endedAt = now`

这意味着临时任务是“到点自动结束”的，而不是完全依赖手工关闭。

## 7. 统一认证、身份、权限模型

这是整个系统最重要的基础设计之一。

## 7.1 三层访问控制

系统不是单一“登录即有权限”的模型，而是三层：

1. 认证 `auth`
   - 证明你是谁
   - 通过 JWT 完成

2. 身份 `identity`
   - 证明你当前以什么身份操作
   - 同一用户可切换多个身份

3. 权限 `permission`
   - 当前身份是否有权限访问某资源

## 7.2 认证层

中间件：`backend/src/middleware/authRequired.ts`

逻辑：

1. 读取 `Authorization: Bearer <token>`
2. 用 `JWT_SECRET` 验证 token
3. 查用户是否存在且 `status = active`
4. 成功后挂载 `req.userId`

失败会返回：

- `UNAUTHORIZED`
- `账号不存在或已停用`
- `登录状态无效或已过期`

## 7.3 身份层

中间件：`backend/src/middleware/identityRequired.ts`

逻辑：

1. 从请求头 `X-Identity-Id` 读取当前身份
2. 校验该身份是否属于当前用户且状态为 `active`
3. 成功后挂载 `req.identity`

`req.identity` 关键字段包括：

- `id`
- `userId`
- `roleCode`
- `orgId`
- `anchorProfileId`
- `scopePath`
- `status`

## 7.4 权限层

中间件：`backend/src/middleware/permissionRequired.ts`

逻辑：

1. 根据当前 `identity.roleCode` 查询 `role_permissions`
2. 提取 `permissionCode`
3. 判断是否包含指定权限，或是否包含 `*`

### 7.4.1 关键含义

- 角色是粗粒度
- 权限码是细粒度
- 页面入口通常会同时做“角色限制 + 权限限制”
- 后端接口也会再次强校验，防止前端绕过

## 7.5 角色清单

前端类型中定义的角色：

- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`
- `ANCHOR`

其中：

- `DEV_ADMIN` 更像系统开发超级管理员
- `HQ_ADMIN` 总部级
- `BASE_ADMIN` 基地级
- `TEAM_ADMIN` 团队级
- `HALL_MANAGER` 厅级管理者
- `ANCHOR` 主播执行者

## 7.6 scopePath 的作用

`scopePath` 是很多业务模块的核心字段。

它用于表达当前身份的组织可见范围，例如：

- `/HQ001`
- `/HQ001/BASE001`
- `/HQ001/BASE001/TEAM001`

绝大多数“范围过滤”都是围绕它做：

- `startsWith(scopePath)` 代表当前范围内的下级组织
- 与祖先 / 后代路径比较代表上下级关系

组织、账号、任务、主播等多模块都依赖它。

## 9. 认证与飞书集成

## 9.1 手机号密码登录

接口：`POST /api/auth/login`

逻辑：

1. 根据手机号查询用户
2. `bcrypt.compare` 验证密码
3. 校验账号状态
4. 更新 `lastLoginAt`
5. 加载可用身份
6. 返回 token + user + identities

## 9.2 改密

接口：`POST /api/auth/change-password`

要求：

- 旧密码正确
- 新密码长度至少 8 位

## 9.3 身份切换

接口：

- `GET /api/identities`
- `POST /api/identities/switch`
- `GET /api/me/permissions`

前端通过 `authStore` 和 `identityStore` 维护：

- token
- 当前用户
- 所有 identities
- 当前 identity

## 13. HTTP 调用约定

`frontend/src/services/http.ts` 定义了统一请求层。

## 13.1 请求头自动注入

- `Authorization: Bearer <token>`
- `X-Identity-Id: <currentIdentity.id>`
- `Content-Type: application/json`

## 13.2 响应约定

前后端统一响应结构：

```ts
{
  success: boolean,
  data?: T,
  error?: {
    code: string,
    message: string,
    details?: unknown
  }
}
```

## 13.3 错误处理特点

- 如果响应不是合法 JSON，会直接抛出“服务响应不是有效 JSON”
- 如果 `success=false` 或 HTTP 非 2xx，也统一抛异常
- 业务异常信息以 `body.error.message` 为主

这使得前端业务层 API 使用相对简单。

## 20. 任务发放模块

后端：`backend/src/modules/task/assignment/*`

这是整个任务业务的核心编排层。

## 20.1 发放对象关键字段

`TaskAssignment` 承担以下职责：

- 绑定模板
- 定义任务类型
- 定义状态
- 定义生效时间
- 定义范围与排除项
- 定义创建者 / 创建身份 / 创建组织
- 临时任务下定义触达模式和主体维度

## 20.2 状态流转

### 日常任务
- `draft`
- `scheduled`
- `active`
- `ended`

### 临时任务
- `draft`
- `active`
- `ended`
- `deleted`

## 20.3 日常任务草稿

关键方法：

- `saveDailyDraft`
- `publishDailyDraft`
- `getDailyPublishPreview`

特点：

1. 可以复用同一创建者 / 组织 / scope 下的草稿
2. 发布时区分：
   - `immediate`
   - `next_midnight`
3. 立即发布会结束同范围旧 active 任务
4. 定时发布会占用同范围 scheduled 名额

## 20.4 临时任务草稿

关键方法：

- `saveTemporaryDraft`
- `getTemporaryPublishPreview`
- `publishTemporaryDraft`

特点：

1. 草稿允许反复保存
2. 发布时必须存在 audience
3. 必须设置截止时间 `deadlineAt`
4. 发布后立即生成主体记录
5. 到时由后台定时任务自动收口

## 20.5 临时任务三种模式

### `ACCOUNT`
按具体账号触达

### `ANCHOR`
按主播主体触达

### `MANAGER`
按管理组织主体触达

`MANAGER` 模式下会额外处理：

- 目标组织主体
- 可见身份集合
- 缺少管理者组织提示
- 多人对同一组织主体协作提交

## 20.6 删除与关闭逻辑

### 日常任务删除
- `scheduled` 的任务删除时，会复制原模板为草稿副本再删除 assignment
- 其他情况通常走“结束”而不是硬删

### 临时任务删除
- 若无 record 且仍是 draft，可硬删
- 否则标记为 `deleted`

### 关闭 / reopen
- 会更新 `status / endedAt / isActive`

## 21. 任务记录模块

后端：`backend/src/modules/task/record/*`

## 21.1 我的待办来源

接口：`GET /api/tasks/my-records`

来源包括两部分：

1. 日常任务：动态匹配后按需补建 record
2. 临时任务：查询对当前身份可见的 `TaskRecord`

## 21.2 日常 record 的特点

不是发布时全量生成，而是：

- 根据当前身份
- 根据当前时间
- 根据今日 / 昨日补录规则
- 动态选中 assignment
- 按需创建 record

这能减少无用数据量，但会提高运行时逻辑复杂度。

## 21.3 临时 record 的特点

发布临时任务时会生成记录，并通过 `TaskRecordIdentityLink` 定义哪些身份可见。

## 21.4 子任务提交

接口：`POST /api/tasks/item-records`

支持：

- 文本
- 选项
- 链接确认
- 附件
- 完成 / 未完成

更新后会重新计算：

- `doneItems`
- `status`
- `submittedAt`
- `lastSubmittedBy*`

## 21.5 记录总提交

接口：`POST /api/tasks/my-records/:id/submit`

前置要求：

- 必填项必须完成
- 当前记录仍可提交

## 21.6 MANAGER 模式协作特性

这是一个很有特点的实现。

当临时任务是：

- `category = TEMPORARY`
- `temporaryMode = MANAGER`
- `subjectType = ORG`

系统会把多人提交的内容以一种“行级快照文本”形式写入 `answerText`，格式类似：

- `[identityId|userId|createdAt] content`

之后再通过解析函数拆成 contribution 列表。

这意味着：

1. 组织主体任务支持多人协作
2. 贡献明细当前是“编码在 answerText 里的”
3. 如果以后要改造为结构化协作日志表，可以此为重构入口

## 21.7 豁免申请

接口：

- `POST /api/tasks/exemptions`
- `DELETE /api/tasks/exemptions/:taskRecordId`
- `GET /api/tasks/exemptions`
- `POST /api/tasks/exemptions/:id/review`

当前规则：

1. 仅日常任务可申请豁免
2. 已提交的记录不能申请豁免
3. 已存在 pending / approved 的豁免不能重复申请
4. 豁免审批通过后，不代表任务完成，只是记录审批结果

## 26. 飞书通知模块

后端有 `task/notify` 路由，前端 `notifyApi` 已经封装日常与临时两类通知预览 / 发送接口。

当前前端已知接口：

### 日常通知
- `GET /api/tasks/notify/daily-feishu/preview`
- `POST /api/tasks/notify/daily-feishu/send`

### 临时通知
- `GET /api/tasks/notify/temporary-feishu/preview`
- `POST /api/tasks/notify/temporary-feishu/send`

## 26.1 通知预览内容

从前端类型可以看出，预览会返回：

- 范围组织
- 前缀占位符
- 总人数
- pending / inProgress 数
- 已绑定 / 未绑定飞书人数
- 配置维度分组统计
- 未绑定用户列表

## 26.2 业务意义

飞书通知不是简单群发，而是：

1. 先按飞书企业配置分组
2. 识别 open_id 绑定情况
3. 输出发送前预览
4. 再执行正式发送

这适合实施与运营人员先确认再推送。

## 28. 典型业务流梳理

## 28.1 用户登录到进入业务页

1. 手机号密码登录或飞书登录
2. 获取 token、user、identities
3. 用户在身份页选择 identity
4. 前端拉取 `/me/permissions`
5. 进入 `AppLayout`
6. 根据角色 + 权限显示菜单

## 28.2 主播注册到成为正式主播

1. 公开注册页提交申请
2. 生成或复用账号
3. 创建 `AnchorRegistrationApplication`
4. 管理端审核
5. 审核通过后创建 `AnchorProfile`
6. 创建 / 激活 `ANCHOR` 身份
7. 账号激活，可登录系统

## 28.3 日常任务闭环

1. 管理员创建 / 维护日常模板
2. 保存日常草稿 assignment
3. 选择立即生效或次日生效
4. 用户进入我的待办时动态匹配 assignment
5. 自动补建当天 / 昨日补录 record
6. 用户逐项完成并提交
7. 管理端通过日常看板查看完成情况
8. 必要时申请 / 审批豁免
9. 飞书通知催办未完成对象

## 28.4 临时任务闭环

1. 管理员选择临时模板
2. 保存 temporary draft
3. 选择 mode、目标、排除项、截止时间
4. 预览发布范围
5. 发布后生成 task records
6. 记录通过 identity links 分发给可见身份
7. 执行者提交项记录或主体记录
8. 管理端通过 temporary dashboard 查看进度
9. 截止后系统自动收口

## 29. 代码层面的关键设计亮点

## 29.1 用户与身份分离

这让系统支持：

- 一人多岗
- 同一账号切换不同组织角色
- 主播与管理身份共存

## 29.2 组织路径 `path` 方案

通过物化路径实现：

- 快速上下级判断
- 范围过滤
- 组织迁移后的路径级联更新

这是该系统最重要的组织建模基础。

## 29.3 任务记录可见性链接表

`TaskRecordIdentityLink` 允许一个记录被多个身份看到，特别适用于：

- 管理式临时任务
- 协作提交
- 多角色查看同一主体任务

## 29.4 日常任务按需补建 record

避免了大规模预生成数据，但要求开发者清楚：

- 某些 record 不存在不是 bug
- 可能需要通过访问入口触发创建

## 29.5 临时任务自动收口

通过服务端定时扫描保障状态一致性，而不是依赖前端操作。

## 30. 维护与排查建议

## 30.1 菜单不可见时

优先检查：

1. 当前 token 是否存在
2. `currentIdentity` 是否存在
3. `/api/me/permissions` 是否返回预期权限
4. 路由是否被 `RoleProtected` 拦截
5. `PermissionGate` 的角色与权限条件是否命中

## 30.2 某用户看不到任务时

优先检查：

1. 当前身份 `roleCode`
2. `scopePath`
3. assignment 的 targets / exclusions
4. 对临时任务：`TaskRecordIdentityLink` 是否有该 identity
5. 对日常任务：是否命中 `pickDailyAssignment`
6. 是否超过补录窗口

## 30.3 组织迁移后数据异常时

优先检查：

1. `OrgUnit.path`
2. `UserIdentity.scopePath`
3. `AnchorProfile.hallOrgId`
4. 相关 `TaskRecord.subjectOrgId / subjectKey`
5. `syncOrganizationRelations` 是否覆盖到场景

## 30.4 飞书登录失败时

优先检查：

1. `FEISHU_REDIRECT_URI`
2. 对应 `FeishuEnterpriseConfig` 是否 active
3. `appId / appSecret` 是否完整
4. 选中的 `configId` 是否正确
5. open_id / union_id 是否已绑定到别的账号

## 30.5 临时任务发布失败时

优先检查：

1. 模板是否存在且未归档
2. `deadlineAt` 是否已设置
3. 目标 audience 是否为空
4. mode / roleCodes / targetUserIds / subjectOrgType 是否匹配
5. 是否缺少 manager 组织主体的管理者

## 31. 后续重构建议

以下不是当前 bug，而是从代码结构上看值得长期优化的方向。

## 31.1 report.routes.ts 体量偏大

报表路由已经非常长，建议拆成：

- daily dashboard service
- temporary dashboard service
- summary/report service
- serializers / mappers

## 31.2 organization/routes.ts 职责偏重

当前把：

- 树查询
- 创建编辑
- 迁移删除
- 飞书配置

都放在一个文件里，后续建议拆模块。

## 31.3 MANAGER 协作记录结构化

目前贡献明细编码在 `answerText` 中，适合尽快评估是否升级为独立表，例如：

- `task_item_contributions`

这样会更利于：

- 搜索
- 统计
- 审计
- 前端展示

## 31.4 权限缓存或批量加载优化

当前权限多次从数据库读取，随着规模增大可考虑：

- Redis 缓存角色权限
- 在登录或切换身份后缓存到短期 session

## 31.5 文件存储升级

若未来进入正式生产，建议把本地 `uploads` 升级为对象存储。

## 33. 总结

当前项目已经不是一个简单 CRUD 系统，而是一个具备以下特征的中等复杂度业务系统：

- 多组织层级
- 多身份切换
- 基于角色 + 权限 + scopePath 的访问控制
- 主播注册审核链路
- 飞书登录绑定与企业配置
- 日常任务闭环
- 临时任务多模式闭环
- 报表下钻与飞书通知

如果用一句话概括这个项目的架构核心，可以表述为：

**以组织树为范围底座，以用户身份为访问载体，以任务发放与任务记录为业务核心，以飞书为外部协同通道的公会任务管理平台。**
