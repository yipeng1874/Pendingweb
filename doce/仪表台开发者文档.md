## 1. 功能概述

仪表台（Cockpit）是本系统 PC 端的核心数据看板页面，路由为 `/tasks/cockpit`。  
面向有管理权限的用户（BASE_ADMIN 及以上），集中展示：

| 模块 | 说明 |
|---|---|
| 顶部 Hero | 使用天数 + 基地切换器 + 上传入口（星星按钮） |
| 历史待办完成率 | 昨天 / 近3天 / 近7天 / 本月（可自定义日期）的投放完成率环形图 |
| 基地直播间空余 | 按「基地→场地→房间类型→团队」四级展示直播间占用 |
| 主播数量统计 | 基地维度的主播总数/7天内新增/20天内新增（v9 已移除趋势图） |
| 厅/主播运营 KPI | 正式厅、训练厅、近30天主播流失、人均音浪 |
| 情况 | 今日投放/完成/进行中/未开始/豁免 |
| 过程指标 | 按团队+厅+日期的完成率趋势折线图，参与团队配置（服务端共享），单条删除 |
| 留存率看板 | 按团队+月份的留存率上传与展示 |
| 员工流失看板 | 按团队+日期的员工流失上传与展示 |

**已搬迁**：
- 「我的待办汇总」→ 路由 `/tasks/dashboard`（TaskDashboardPage）

## 5. 过程指标（v9 重点）

### 5.1 架构设计

```
前端 ProcessMetricCard
  ↓ scopeOrgId
  ↓ 并行请求：
  ├── GET /process-metric/config   → 读取参与团队配置（服务端存储，同一基地所有用户共享）
  ├── GET /process-metric/by-date  → 读取过程指标数据
  └── fetchOrgTree()               → 获取组织树（用于团队列表）

删除：
  上传弹窗中点击删除按钮 → DELETE /process-metric
    ↓ 用位置（idx）在 dateEntries 里找回原始厅名
    ↓ 用原始厅名调后端删除
    ↓ 删除成功 → 本地更新 dateEntries（图表实时刷新）
```

### 5.2 参与团队配置（服务端共享）

**之前的问题**：`participatingTeamIds` 存在 React 本地 state，A 账号配置的团队 B 看不到。

**v9 解决方案**：`process_metric_configs` 表，按 `baseOrgId` 主键存储 `teamIds` JSON 数组。

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/process-metric/config?scopeOrgId=xxx` | 读取该基地的参与团队 ID 列表 |
| PUT | `/process-metric/config?scopeOrgId=xxx` | 保存参与团队 ID 列表（全量覆盖） |

**前端行为**：
- 首次加载：fetch config，如果服务端有配置则使用；如果为空则自动检测有数据的团队并保存到服务端
- 每次切换参与团队：`toggleParticipatingTeam` 立即调 PUT 保存
- 全选/全不选：一次性保存到服务端

### 5.3 删除功能

**场景**：A 团队有多个厅，想删除某个厅的数据。

**实现**：

| 方法 | 路径 | 参数 |
|---|---|---|
| DELETE | `/process-metric` | `?scopeOrgId=xxx&teamOrgId=xxx&hallName=xxx&recordDate=xxx` |

**前端逻辑**：
1. 用户点击删除按钮
2. 用 **idx 位置** 在 `dateEntries` 中找到**原始厅名**（防止用户改过名导致匹配失败）
3. 用原始厅名调 DELETE 接口
4. 成功 → 本地更新 dateEntries，图表实时刷新
5. 404 → 忽略（说明数据库本来就没有）

**权限**：TEAM_ADMIN 和 HALL_MANAGER 无权删除（返回 403）

### 5.4 全部接口

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/process-metric/upsert?scopeOrgId=xxx` | 覆盖写入单条 |
| POST | `/process-metric/upsert-batch?scopeOrgId=xxx` | 批量覆盖写入 |
| GET | `/process-metric/by-date?scopeOrgId=xxx&days=6` | 按日期查询（最近 N 天，含团队、厅分组） |
| GET | `/process-metric/latest-halls?scopeOrgId=xxx&teamOrgId=xxx` | 某团队最近一次的厅列表 |
| GET | `/process-metric/config?scopeOrgId=xxx` | 读取参与团队配置 |
| PUT | `/process-metric/config?scopeOrgId=xxx` | 保存参与团队配置 |
| DELETE | `/process-metric?scopeOrgId=xxx&teamOrgId=xxx&hallName=xxx&recordDate=xxx` | 删除单条记录 |

### 11.1 服务器环境

| 项目 | 要求 |
|---|---|
| CPU | 2 核+（推荐 4 核） |
| 内存 | 4 GB+（推荐 8 GB） |
| 磁盘 | 50 GB+ |
| Node.js | 18.x LTS |
| MySQL | 8.0+ |
| Nginx | 1.24+ |
| PM2 | latest（`npm i -g pm2`） |

### 11.2 端口分配

| 服务 | 端口 | 说明 |
|---|---|---|
| MySQL | 3306 | 数据库 |
| 后端 API | **4000** | Express（PM2 守护） |
| Nginx PC | **8080** | `frontend/dist` + 反代 `/api/` → 4000 |
| Nginx H5 | **8081** | `todo-h5/dist` + 反代 `/api/` → 4000 |

### 11.4 标准部署流程

#### Step 1：备份数据库

```bash
D:\Pendingweb\auto-backup.bat
```

#### Step 2：拉取代码 + 安装依赖

```bash
cd D:\Pendingweb
git pull origin main
npm install
```

#### Step 3：数据库迁移（v9 新增表）

```bash
# 执行建表脚本（新增 process_metric_configs 表）
cd D:\Pendingweb\backend
npx tsx src/migrations/migrate-process-metric-config.mjs

# 或直接用 Prisma（含所有迁移）
# npx prisma migrate deploy
npx prisma generate
cd ..
```

#### Step 4：构建前后端

```bash
npm run build -w frontend
cd todo-h5 && npm run build && cd ..
npm run build -w backend
```

#### Step 5：重载 Nginx + 重启后端

```bash
start-nginx.bat
pm2 restart anchor-todo-api
pm2 logs anchor-todo-api --lines 30
```

#### Step 6：验证

| 检查项 | 操作 |
|---|---|
| 后端健康检查 | `http://localhost:4000/api/health` |
| PC 前端 | `http://localhost:8080/` |
| 过程指标 | 打开仪表台 → 确认配置同步、删除功能正常 |
| 主播统计 | 确认趋势图已移除，仅剩 3 张统计卡片 |

## 13. 开发时新增模型流程

1. 修改 `backend/prisma/schema.prisma`
2. 执行：
   ```bash
   cd backend
   npx prisma migrate dev --name add_your_model_name
   ```
3. 提交迁移文件到 Git
4. 推送后生产环境执行第 11.4 节流程
5. 如果表无需 DDL（如 JSON 列扩展），只需 `npx prisma generate`
