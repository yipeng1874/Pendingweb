## 1. 功能概述

仪表台（Cockpit）是本系统 PC 端的核心数据看板页面，路由为 `/tasks/cockpit`。  
面向有管理权限的用户（BASE_ADMIN 及以上），集中展示：

| 模块 | 说明 |
|---|---|
| 顶部 Hero | 使用天数 + 基地切换器 + 上传入口（星星按钮） |
| 历史待办完成率 | 昨天 / 近3天 / 近7天 / 本月（可自定义日期）的投放完成率环形图 |
| 基地直播间空余 | 覆盖式快照：总数量 / 已使用 / 办公室 / 空余 + 进度条 |
| 主播数量趋势 | 基地维度的主播线上/线下分布、入职时间分布及各运营分组统计 |
| 厅/主播运营 KPI | 正式厅、训练厅、近30天主播流失、线下人均音浪/线上人均音浪/人均音浪（含浮层明细和趋势） |
| 今日主播日常任务概览 | 今日投放/完成/进行中/未开始/豁免 |

**已搬迁**：
- 「我的待办汇总」→ 路由 `/tasks/dashboard`（TaskDashboardPage），位于标题下方

model HallDailySummary {

**业务说明**：时序数据，按 baseOrgId + recordDate 唯一。上传用 upsert，同日覆盖。

### 4.3 AnchorDailySummary（主播每日快照）

```prisma
model AnchorDailySummary {
  id            String   @id @default(uuid())
  baseOrgId     String   @map("base_org_id")
  baseOrgName   String   @map("base_org_name")
  recordDate    String   @map("record_date")
  uploadedBy    String   @map("uploaded_by")
  uploaderName  String   @map("uploader_name")
  totalCount    Int      @map("total_count")
  onlineCount   Int      @map("online_count")
  offlineCount  Int      @map("offline_count")
  within7Days   Int      @map("within_7_days")
  within20Days  Int      @default(0) @map("within_20_days")
  dailyNew      Int      @default(0) @map("daily_new")
  operatorStats Json     @map("operator_stats")
  rawRowCount   Int      @map("raw_row_count")
  rawAnchors    Json?    @map("raw_anchors")     // 【v6 新增】
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  @@unique([baseOrgId, recordDate])
  @@map("anchor_daily_summaries")
}
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **时序数据**（每天一条，按 baseOrgId + recordDate 唯一） |
| **唯一约束** | `@@unique([baseOrgId, recordDate])` |
| **录入方式** | Excel 上传（主播信息表），需指定 `recordDate` |
| **趋势支持** | 支持 N 日趋势图（默认 7 天），含线上/线下堆叠柱 + 总数折线 + 当日新增折线 |

**新增字段 `rawAnchors`（v6）**：

| 维度 | 说明 |
|---|---|
| **存储格式** | `[{ "joinDate": "2026-07-01", "isOnline": true }, ...]` |
| **用途** | 保留每条主播的入职日期和类型原始明细，供前端查看时动态试用期过滤 |
| **写入时机** | Excel 上传时逐行构建 |
| **可空** | 是（历史数据为 NULL，旧数据无法做试用期过滤） |

**试用期过滤机制**：趋势接口 `GET /anchor-summary/trend` 新增 `probationDays` 参数（如 `5`/`20`/`30`）。传入后对每条 record 的 `rawAnchors` 做遍历重算：`入职时间 < probationDays 天` 的主播不进入 `totalCount/onlineCount/offlineCount`。详见第 4.7 节。

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/anchor-summary/upload?scopeOrgId=xxx` | FormData（file + recordDate） |
| GET | `/anchor-summary/latest?scopeOrgId=xxx` | 最新一条记录或 null |
| GET | `/anchor-summary/trend?scopeOrgId=xxx&days=7&probationDays=5` | 趋势数据，probationDays 可选 |

### 4.4 LiveRoomCapacity（基地直播间空余）

```prisma
model LiveRoomCapacity {
  id           String   @id @default(uuid())
  baseOrgId    String   @unique @map("base_org_id")
  baseOrgName  String   @map("base_org_name")
  siteDetails  Json     @map("site_details")
  updatedBy    String   @map("updated_by")
  updaterName  String   @map("updater_name")
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@map("live_room_capacities")
}
```

**siteDetails JSON 结构**：

```json
[
  {
    "siteId": "uuid",
    "siteName": "场地一",
    "rooms": [
      { "typeName": "直播间", "used": 10, "total": 20 },
      { "typeName": "办公室", "used": 3, "total": 20 },
      { "typeName": "化妆间", "used": 2, "total": 5 }
    ]
  }
]
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **覆盖式快照**（每次填写覆盖上一条，只保留最新数据） |
| **唯一约束** | `@unique([baseOrgId])`（每个基地一条记录） |
| **录入方式** | 表单填写（非 Excel 上传），支持多场地、多房间类型 |
| **多场地支持** | `siteDetails` 数组中每条为一个场地，场地名来自 `LiveRoomSite` 表 |
| **房间类型** | 每个场地 `rooms` 数组支持自定义类型（直播间、办公室、化妆间等） |
| **空余计算** | 前端计算：`spareCount = total - used`（按行聚合） |
| **不需要日期** | 不携带 recordDate，它就是最新快照 |

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/live-room-capacity/upsert?scopeOrgId=xxx` | body 含 `siteDetails` 数组 |
| GET | `/live-room-capacity/latest?scopeOrgId=xxx` | 返回最新一条记录（含 siteDetails），展示时按场地分组 |
| GET | `/live-room-capacity/sites?scopeOrgId=xxx` | 获取该基地的所有场地列表（用于表单选择） |

### 4.4b LiveRoomSite（基地直播间场地）

```prisma
model LiveRoomSite {
  id           String   @id @default(uuid())
  baseOrgId    String   @map("base_org_id")
  baseOrgName  String   @map("base_org_name")
  name         String                                                // 场地名称（如"场地一"），可自定义
  sort         Int      @default(0)                                  // 排序
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@unique([baseOrgId, name])                                        // 同基地下场地名不重复
  @@index([baseOrgId])
  @@map("live_room_sites")
}
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **独立实体表**，每个基地可创建多个场地 |
| **唯一约束** | `@@unique([baseOrgId, name])`（同基地下场地名不重复） |
| **用途** | 定义该基地有哪些场地（如"场地一""场地二""备用场地"），供直播间空余录入时选择 |
| **管理方式** | 前端界面可 CRUD 场地，LiveRoomCapacity 的 siteDetails 引用 siteId 关联 |

### 4.5 【新增】AnchorAvgWaveDaily（人均音浪每日快照）

```prisma
model AnchorAvgWaveDaily {
  id           String   @id @default(uuid())
  baseOrgId    String   @map("base_org_id")
  baseOrgName  String   @map("base_org_name")
  recordDate   String   @map("record_date")
  waveType     String   @default("online") @map("wave_type")  // "online" | "offline" | "total"
  avgWaveValue Float    @map("avg_wave_value")
  totalWave    Float    @default(0) @map("total_wave")
  anchorCount  Int      @default(0) @map("anchor_count")
  uploadedBy   String   @map("uploaded_by")
  uploaderName String   @map("uploader_name")
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@unique([baseOrgId, recordDate, waveType])
  @@map("anchor_avg_wave_dailies")
}
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **时序数据**（每天一条，按 waveType 区分线上/线下/合计） |
| **唯一约束** | `@@unique([baseOrgId, recordDate, waveType])` |
| **录入方式** | 表单填写（数值录入 tab），需指定日期 |
| **线下** | `waveType = "offline"`，即"线下人均音浪"（KPI 卡片主展示） |
| **线上** | `waveType = "online"`，即"线上人均音浪" |
| **合计** | `waveType = "total"`，即"人均音浪"（合计值，独立录入） |
| **趋势支持** | 支持 7 日趋势折线图（三条曲线） |

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/anchor-avg-wave/upsert?scopeOrgId=xxx` | body: `recordDate / avgWaveValue / waveType`（waveType: "online" | "offline" | "total"）[totalWave / anchorCount] |
| GET | `/anchor-avg-wave/latest?scopeOrgId=xxx` | 返回 `{ online: ... , offline: ... , total: ... }` |
| GET | `/anchor-avg-wave/trend?scopeOrgId=xxx&days=7` | 返回 `{ online: { points, latest, prevDay, change }, offline: { ... }, total: { ... } }` |

### 4.7 【v6 新增】试用期动态过滤机制

**业务需求**：查看趋势图时可选择试用期天数（5/10/15/20/25/30 天），入职未满试用期的主播不进入 `totalCount/onlineCount/offlineCount` 统计。

**实现原理**：

```
┌─ 上传时 ───────────────────────┐       ┌─ 查看时 ─────────────────────────────┐
│ 逐行解析 Excel                 │       │ GET /anchor-summary/trend?            │
│ → push {joinDate, isOnline}     │       │   scopeOrgId=xxx&days=7&               │
│ → 存入 rawAnchors JSON 列      │       │   probationDays=5                     │
└──────────────────────────────┘       │                                        │
                                       │ for each record:                       │
                                       │   for each anchor in rawAnchors:       │
                                       │     if joinDate exists AND             │
                                       │        (recordDate - joinDate) <       │
                                       │         probationDays:                 │
                                       │         → 跳过（不计入）               │
                                       │     else:                              │
                                       │         → totalCount++ / online++ /     │
                                       │           offline++                    │
                                       │ 返回 probationExcluded 字段            │
                                       └────────────────────────────────────────┘
```

**关键约束**：
- 试用期过滤**仅在查看时动态计算**，不上传到数据库
- 旧数据（`rawAnchors` 为 NULL）无法做试用期过滤，需重新上传 Excel 才会写入原始明细
- `within7Days` / `within20Days` / `dailyNew` 指标不受试用期影响（它们是独立的入职时长统计）
- 试用期过滤不包含 `operatorStats`（运营分组明细暂不做动态重算）

**前端交互**：

| 元素 | 说明 |
|---|---|
| 选择器 | 标题行右侧下拉框，选项：无试用期 / 5天 / 10天 / 15天 / 20天 / 25天 / 30天 |
| 触发 | onChange 时自动调用 API，传入 probationDays 参数 |
| 提示 | 选择试用期后，图表底部显示琥珀色提示条：`试用期 N 天内入职的 X 人未计入主播总数/线上/线下` |

**新增 API 参数**：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `probationDays` | `number`（query） | `0` | `0` = 不启用过滤，`>0` = 过滤入职 < N 天的主播 |

## 5. 统一上传弹窗

### 5.1 设计理念

所有数据上传集中在**一个弹窗**、**三个 tab**：

```
┌── 上传数据 ──────────────────────────────────┐
│ [厅数据/流失表] [主播数据表] [数值录入]   [×]  │
├──────────────────────────────────────────────┤
│ 当前 tab 内容                                 │
└──────────────────────────────────────────────┘
```

| Tab | 说明 | 对应 API | 对应数据模型 |
|---|---|---|---|
| 厅数据 / 流失表 | Excel 上传，自动解析「厅个数」+「主播流失」 | `dataOverviewApi.upload` | HallDailySummary + AnchorLossDailySummary |
| 主播数据表 | Excel 上传，解析「主播信息表」 | `anchorSummaryApi.upload` | AnchorDailySummary |
| 数值录入 | 表单填写（直播间空余 + 线下人均音浪 + 线上人均音浪 + 人均音浪） | `liveRoomCapacityApi.upsert` + `anchorAvgWaveApi.upsert` | LiveRoomCapacity + AnchorAvgWaveDaily |

### 5.2 入口设计

- **主要入口**：Hero 左侧星星按钮（半隐藏），hover 时呼吸动画 + tooltip，双击打开弹窗
- **Hero 文字**：单行显示「您已陪伴千广成长系统 N 天 · 成长协同」
- **基地切换器**：Hero 右侧半透明 select

### 5.3 联动刷新

- 厅数据/流失表上传成功 → 刷新 `hallTrend` / `lossTrend`
- 主播数据表上传成功 → 触发 `window.dispatchEvent(new CustomEvent("anchor-summary-refresh"))` → AnchorSummaryCard 自动刷新
- 数值录入 → 刷新 `roomCapacity` / `avgWaveTrend`

- **只有** `hallType === "正式厅"` 或 `hallType === "训练厅"` 的行才算数据行
- 其他行（汇总行、表头行、空行）自动跳过

### 7.2 主播流失 Sheet

**列探测**：
- 遍历前 3 行的键名和值
- **优先精确匹配** `"流失时间"`（避免匹配到衍生列）
- 退化匹配：`v.includes("流失时间")`
- 取「所属运营」列

**数据行识别**：`parseDateStr()` 返回 `null` 的行自动跳过

### 7.3 parseDateStr 日期解析

```typescript
// 字符串：智能识别三种格式
//   YYYY-MM-DD（n1 > 31 → 年-月-日）
//   M/D/YYYY（n3 > 31 → 月-日-年）
//   M/D/YY（全部 ≤ 31 → 月-日-年，2位年 +2000）
// 数字：Excel 日期序列号转 Date
```

**关键注意**：不依赖 `toISOString()`（避免 UTC 时区偏移导致日期差一天）。

### 7.4 【v6 新增】主播信息表 rawAnchors 存储

上传「主播数据表」时，除了原有汇总统计外，还额外构建 `rawAnchors` 数组存入数据库：

```typescript
const rawAnchors: { joinDate: string | null; isOnline: boolean }[] = [];

for (const row of rows) {
  const joinDate = parseExcelDate(row["入职日期"] / row["加入时间"]);
  const isOnline = String(row["主播类型"]).trim() === "线上";

  rawAnchors.push({
    joinDate: joinDate ? joinDate.toISOString().slice(0, 10) : null,
    isOnline,
  });
  // ... 原有汇总统计不变
}

// upsert 时同时存入 rawAnchors
```

**必要列**：`主播昵称`、`所属基地`、`所属运营`、`主播类型`（必选），`入职日期`/`加入时间`/`入职时间`/`加入日期`（可选，但若无此列则 rawAnchors 的 joinDate 全为 null）。

## 9. 生产环境部署与数据库更新流程

> **核心原则**：所有数据库 schema 变更（加表、加列、改约束等）通过 `backend/prisma/schema.prisma` + Prisma 迁移机制管理，禁止手动在生产数据库执行 DDL。

---

### 9.1 Prisma 迁移机制概览

| 命令 | 用途 | 何时使用 |
|---|---|---|
| `prisma db push` | 直接将 schema 同步到数据库（不生成迁移文件） | **开发阶段**快速验证、**简单变更**（加列、加 `@default`） |
| `prisma migrate dev --name xxx` | 生成迁移文件 + 应用到开发数据库 | **开发阶段**正式变更，迁移文件提交到 Git |
| `prisma migrate deploy` | 应用已有迁移文件到数据库（不检查 schema 漂移） | **生产环境**唯一推荐方式 |
| `prisma migrate resolve` | 解决失败的迁移记录 | 迁移中途失败时修复 |
| `prisma generate` | 重新生成 Prisma Client（类型定义） | schema 变更后必须执行 |

---

### 9.2 标准上线流程（含数据库变更）

#### 开发电脑（第一步：修改 schema + 生成迁移）

```bash
# 1. 修改 backend/prisma/schema.prisma（加模型 / 加字段 / 改约束）

# 2. 生成迁移文件
cd backend
npx prisma migrate dev --name add_your_change_name
# ↓ 会在 backend/prisma/migrations/ 下生成新目录（含 SQL 文件）
# ↓ 同时自动执行 prisma generate

# 3. 检查生成的 SQL 迁移文件是否合理
cat backend/prisma/migrations/<最新目录>/migration.sql
```

#### 开发电脑（第二步：提交代码 + 迁移文件）

```bash
# 4. 提交 schema + 迁移文件到 Git
git add backend/prisma/schema.prisma
git add backend/prisma/migrations/
git commit -m "feat: add your_model_name / alter your_table_name"
git push
```

**重要**：`prisma/migrations/` 目录中的 SQL 文件**必须提交到 Git**，它们是生产环境执行迁移的唯一依据。

#### 生产环境 / 公司电脑（第三步：拉取 + 部署）

```bash
# 5. 拉取最新代码
cd C:\deploy\source
git pull

# 6. 【必做】备份数据库
db-backup.bat

# 7. 安装依赖（如 package.json 有变化）
npm install

# 8. 应用数据库迁移（核心步骤）
cd backend
npx prisma migrate deploy
# ↓ 自动执行 migrations/ 中所有未应用的 SQL
# ↓ 自动执行 prisma generate

# 9. 如迁移文件不存在（用了 prisma db push）
#    替代方案：
npx prisma db push
npx prisma generate

# 10. 构建后端
cd C:\deploy\source
npm run build -w backend

# 11. 重启后端服务
pm2 restart anchor-todo-api

# 12. 查看启动日志确认无报错
pm2 logs anchor-todo-api --lines 30
```

#### 生产环境（第四步：验证）

| 检查项 | 命令/方式 |
|---|---|
| 后端健康检查 | `curl http://localhost:4000/api/health` |
| PM2 服务状态 | `pm2 list`（status 应为 `online`） |
| 关键 API 响应 | 登录、仪表台数据加载 |
| 前端页面 | 清缓存后访问，确认新功能可用 |

### 9.3 三种变更场景及对应操作

#### 场景 A：纯加字段 / 纯加 `@default` / 纯改注释

**无破坏性**，最简单的变更。

开发电脑：
```bash
cd backend
npx prisma migrate dev --name add_xxx_column  # 生成迁移文件
git add backend/prisma/
git commit -m "feat: add xxx column to yyy"
git push
```

生产环境：
```bash
git pull
db-backup.bat                          # 备份
cd backend && npx prisma migrate deploy # 执行迁移（同时 generate）
cd .. && npm run build -w backend       # 构建
pm2 restart anchor-todo-api             # 重启
```

#### 场景 B：加新表 / 加 `@unique` / 加外键

**有风险，需确认已有数据不违反新约束**。

开发电脑：同上，但生成迁移后检查 SQL 确认不会因数据冲突而失败。

生产环境：同上，但额外注意：
```bash
# 如果 migrate deploy 报唯一约束冲突
# → 先清理重复数据，再重新执行
npx prisma migrate deploy
```

#### 场景 C：结构迁移（改列类型 / 拆表 / 数据迁移）

**高风险，推荐用两步法或自定义迁移脚本**。

**两步法示例**（本次 LiveRoomCapacity 结构升级）：
- Step 1：`prisma migrate dev --name` 只做 DDL（加新列 / 加新表，保留旧列）
- Step 2：用自定义迁移脚本做数据迁移（`src/migrations/migrate-live-room-capacity.ts`）
- Step 3：再起一个 migration 删旧列

或使用 `backend/src/migrations/` 中的自定义脚本：
```bash
# 生产环境执行
cd backend

# 方案一：直连数据库跑 SQL
mysql -u root -p pendingweb < src/migrations/migrate-live-room-capacity.sql

# 方案二：用 Prisma 跑迁移脚本
npx tsx src/migrations/migrate-live-room-capacity.ts

# 无论哪种，最后必须
npx prisma generate
cd .. && npm run build -w backend
pm2 restart anchor-todo-api
```

### 9.6 v7 数据库变更详情（LiveRoomCapacity 结构升级）

**变更说明**：`LiveRoomCapacity` 从简单三字段（`totalCount / liveRoomUsed / officeUsed`）升级为多场地多房间类型结构。

**旧结构**（已废弃）：
```
live_room_capacities
├── total_count        INT     ← 已删除
├── live_room_used     INT     ← 已删除
├── office_used        INT     ← 已删除
└── base_org_id        VARCHAR UNIQUE
```

**新结构**：
```
live_room_sites               ← 新表
├── id, base_org_id, name, sort
└── UNIQUE(base_org_id, name)

live_room_capacities          ← 结构变更
├── site_details        JSON  ← 新列（替代三个旧列）
│   [{
│     siteId, siteName,
│     rooms: [{typeName, used, total}, ...]
│   }]
└── base_org_id         VARCHAR UNIQUE
```

**迁移 SQL 概要**（完整脚本见 `backend/src/migrations/migrate-live-room-capacity.sql`）：

```sql
-- 1. 建新表
CREATE TABLE live_room_sites (...);

-- 2. 为已有基地创建默认场地
INSERT INTO live_room_sites SELECT ... FROM live_room_capacities;

-- 3. 加新列
ALTER TABLE live_room_capacities ADD COLUMN site_details JSON;

-- 4. 数据迁移
UPDATE live_room_capacities SET site_details = JSON_ARRAY(...);

-- 5. 删旧列
ALTER TABLE live_room_capacities
  DROP COLUMN total_count, DROP COLUMN live_room_used, DROP COLUMN office_used;
```

**生产环境执行步骤**：

```bash
# 方式一：用 prisma db push（推荐，自动处理建表 + 删列）
cd backend
npx prisma db push
# 然后执行数据迁移
npx tsx src/migrations/migrate-live-room-capacity.mjs
npx prisma generate

# 方式二：用 raw SQL（需先确认 prisma db push 已完成）
mysql -u root -p pendingweb < src/migrations/migrate-live-room-capacity.sql
npx prisma generate

# 最后构建 + 重启
cd ..
npm run build -w backend
pm2 restart anchor-todo-api
```

**影响评估**：

| 维度 | 说明 |
|---|---|
| **DDL 风险** | 中 — 涉及删列，建议先备份数据库 |
| **兼容性** | **不向后兼容** — 旧前端调用旧字段会报错 |
| **数据安全** | 旧数据已迁移到 siteDetails，不丢失 |
| **回滚方案** | 恢复备份 → 回滚代码 → 重新部署 |

---

### 9.7 常见迁移错误处理

| 错误信息 | 原因 | 处理方式 |
|---|---|---|
| `Table 'xxx' doesn't exist` | 未执行迁移 | 执行 `prisma migrate deploy` |
| `The table 'xxx' already exists` | 重复迁移或手动建表冲突 | 检查 `_prisma_migrations` 表 |
| `P3009: migrate found failed migrations` | 上次迁移中途失败 | `prisma migrate resolve` |
| `Prisma Client is not generated` | Client 未重新生成 | `npx prisma generate` 后 rebuild |
| `Duplicate entry for key` | 唯一约束冲突（新加约束前已有重复数据） | 先清理重复数据再部署 |
| `Can't DROP column; check that it exists` | 列已被删除（重复执行迁移） | 跳过，不影响 |
| `Unknown column 'xxx' in schema.prisma` | schema 与数据库不同步 | 确认 `prisma db push` 已执行 |
