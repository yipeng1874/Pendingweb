## 1. 功能概述

仪表台（Cockpit）是本系统 PC 端的核心数据看板页面，路由为 `/tasks/cockpit`。  
面向有管理权限的用户（BASE_ADMIN 及以上），集中展示：

| 模块 | 说明 |
|---|---|
| 顶部 Hero | 使用天数 + 基地切换器 + 上传入口（星星按钮） |
| 历史待办完成率 | 昨天 / 近3天 / 近7天 / 本月（可自定义日期）的投放完成率环形图 |
| 基地直播间空余 | 按「基地→场地→房间类型→团队」四级展示直播间占用；悬停查看团队明细和整体占比 |
| 主播数量趋势 | 基地维度的主播线上/线下分布、入职时间分布及各运营分组统计 |
| 厅/主播运营 KPI | 正式厅、训练厅、近30天主播流失、线下人均音浪/线上人均音浪/人均音浪（含浮层明细和趋势） |
| 今日主播日常任务概览 | 今日投放/完成/进行中/未开始/豁免 |

**已搬迁**：
- 「我的待办汇总」→ 路由 `/tasks/dashboard`（TaskDashboardPage），位于标题下方

### 4.3 AnchorDailySummary（主播每日快照）

```prisma
model AnchorDailySummary {
  id            String   @id @default(uuid())
  baseOrgId     String   @map("base_org_id")
  baseOrgName   String   @map("base_org_name")
  recordDate    String   @map("record_date")
  uploadedBy    String   @map("uploaded_by")
  uploaderName  String   @map("uploader_name")
  totalCount    Int      @map("total_count")
  onlineCount   Int      @map("online_count")
  offlineCount  Int      @map("offline_count")
  within7Days   Int      @map("within_7_days")
  within20Days  Int      @default(0) @map("within_20_days")
  dailyNew      Int      @default(0) @map("daily_new")
  operatorStats Json     @map("operator_stats")
  rawRowCount   Int      @map("raw_row_count")
  rawAnchors    Json?    @map("raw_anchors")     // v6 新增
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  @@unique([baseOrgId, recordDate])
  @@map("anchor_daily_summaries")
}
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **时序数据**（每天一条，按 baseOrgId + recordDate 唯一） |
| **唯一约束** | `@@unique([baseOrgId, recordDate])` |
| **录入方式** | Excel 上传（主播信息表），需指定 `recordDate` |
| **趋势支持** | 支持 N 日趋势图（默认 7 天），含线上/线下堆叠柱 + 总数折线 + 当日新增折线 |

**新增字段 `rawAnchors`（v6）**：

| 维度 | 说明 |
|---|---|
| **存储格式** | `[{ "joinDate": "2026-07-01", "isOnline": true }, ...]` |
| **用途** | 保留每条主播的入职日期和类型原始明细，供前端查看时动态试用期过滤 |
| **写入时机** | Excel 上传时逐行构建 |
| **可空** | 是（历史数据为 NULL，旧数据无法做试用期过滤） |

**试用期过滤机制**：趋势接口 `GET /anchor-summary/trend` 新增 `probationDays` 参数（如 `5`/`20`/`30`）。传入后对每条 record 的 `rawAnchors` 做遍历重算：`入职时间 < probationDays 天` 的主播不进入 `totalCount/onlineCount/offlineCount`。

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/anchor-summary/upload?scopeOrgId=xxx` | FormData（file + recordDate） |
| GET | `/anchor-summary/latest?scopeOrgId=xxx` | 最新一条记录或 null |
| GET | `/anchor-summary/trend?scopeOrgId=xxx&days=7&probationDays=5` | 趋势数据，probationDays 可选 |

### 4.4 LiveRoomCapacity（基地直播间空余）

```prisma
model LiveRoomCapacity {
  id           String   @id @default(uuid())
  baseOrgId    String   @unique @map("base_org_id")
  baseOrgName  String   @map("base_org_name")
  siteDetails  Json     @map("site_details")
  updatedBy    String   @map("updated_by")
  updaterName  String   @map("updater_name")
  createdAt    DateTime @default(now()) @map("created_at")
  updatedAt    DateTime @updatedAt @map("updated_at")

  @@map("live_room_capacities")
}
```

**siteDetails JSON 结构（v8 含团队分配）**：

```json
[
  {
    "siteId": "uuid",
    "siteName": "南区",
    "rooms": [
      {
        "typeName": "1228",
        "used": 130,
        "total": 180,
        "allocations": [
          { "orgId": "team-uuid-1", "orgName": "一队", "used": 30 },
          { "orgId": "team-uuid-2", "orgName": "二队", "used": 50 },
          { "orgId": "team-uuid-3", "orgName": "三队", "used": 50 }
        ]
      }
    ]
  }
]
```

**业务说明**：

| 维度 | 说明 |
|---|---|
| **数据性质** | **覆盖式快照**（每次填写覆盖上一条，只保留最新数据） |
| **唯一约束** | `@unique([baseOrgId])`（每个基地一条记录） |
| **录入方式** | 表单填写（非 Excel 上传），支持多场地、多房间类型、按团队分配 |
| **多场地支持** | `siteDetails` 数组中每条为一个场地，场地名来自 `LiveRoomSite` 表 |
| **房间类型** | 每个场地 `rooms` 数组支持自定义类型（直播间、办公室、1228 等） |
| **团队分配** | `allocations` 数组关联组织架构中的 TEAM 节点，每个团队的 `used` 独立输入 |
| **已用计算** | 有 `allocations` 时 `used = sum(allocations[].used)`，自动合计；无时兼容旧 `used` 字段 |
| **不需要日期** | 不携带 recordDate，它就是最新快照 |

**前端交互（v8 新增）**：

| 交互点 | 说明 |
|---|---|
| **展示区** | 每个房间类型行显示进度条 + 已用/总数 + 百分比。悬停该行显示该类型的团队分配明细（团队名 + 占用数 + 百分比） |
| **底部汇总悬停** | 悬停「总/已用/空余」行显示该场地跨所有类型的团队整体占比（聚合所有 rooms 的 allocations 按 orgId 求和） |
| **录入表单** | 房间类型行：类型名称 → 总数 → 团队占用(▼按钮) → 删除。展开后显示当前基地下所有 TEAM，每个团队一个输入框，已用自动合计。分配总和超过总数时红色警告 |

**接口**：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/live-room-capacity/upsert?scopeOrgId=xxx` | body 含 `siteDetails` 数组（含可选 `allocations`） |
| GET | `/live-room-capacity/latest?scopeOrgId=xxx` | 返回最新一条记录（含 siteDetails + allocations），展示时按场地分组 |
| GET | `/live-room-sites?scopeOrgId=xxx` | 获取该基地的所有场地列表 |
| POST | `/live-room-sites?scopeOrgId=xxx` | 创建新场地 |
| PUT | `/live-room-sites/:id` | 修改场地名/排序 |
| DELETE | `/live-room-sites/:id` | 删除场地 |

### 4.5 LiveRoomSite（基地直播间场地）

```prisma
model LiveRoomSite {
  id           String   @id @default(uuid())
  baseOrgId    String   @map("base_org_id")
```

**业务说明**：

| **用途** | 定义该基地有哪些场地，供直播间空余录入时选择 |
| **管理方式** | 前端界面可 CRUD 场地（重命名、删除），LiveRoomCapacity 的 siteDetails 引用 siteId 关联 |

## 5. 前端类型定义

### 5.1 RoomAllocation（v8 新增）

```typescript
// frontend/src/services/task.ts

export type RoomAllocation = {
  orgId: string;             // 组织架构 ID（TEAM 类型）
  orgName: string;           // 团队名称
  used: number;              // 该团队占用数
};

export type RoomTypeDetail = {
  typeName: string;          // 自定义房间类型名
  used: number;              // 已使用（有 allocations 时等于其 sum）
  total: number;             // 总数
  allocations?: RoomAllocation[]; // v8 新增：按团队的占用明细
};
```

### 5.2 数据流

```
录入时：
  用户展开团队分配 → 填写各团队占用数 → 前端实时合计 → 提交时 used=sum(allocations.used)

展示时：
  后端返回 siteDetails（含 allocations） → 前端 getUsed() 优先从 allocations 求和
  → 进度条用 used 计算百分比
  → 悬停行显示该类型团队明细
  → 悬停底部汇总显示场地整体占比（跨类型聚合）
```

### 9.5 Prisma 迁移机制

| 命令 | 用途 | 何时使用 |
|---|---|---|
| `prisma db push` | 直接将 schema 同步到数据库（不生成迁移文件） | 开发阶段快速验证、简单变更 |
| `prisma migrate dev --name xxx` | 生成迁移文件 + 应用到开发数据库 | 开发阶段正式变更，迁移文件提交到 Git |
| `prisma migrate deploy` | 应用已有迁移文件到数据库（不检查 schema 漂移） | **生产环境唯一推荐方式** |
| `prisma migrate resolve` | 解决失败的迁移记录 | 迁移中途失败时修复 |
| `prisma generate` | 重新生成 Prisma Client（类型定义） | schema 变更后必须执行 |

> **v8 说明**：本次 `LiveRoomCapacity` 的 `siteDetails` JSON 列已包含 `allocations` 数组，**无需 DDL 变更**。只需 `prisma generate` 重新生成 Client 即可。

### 9.8 常见问题排查

| 问题 | 排查步骤 |
|---|---|
| **后端启动失败** | `pm2 logs anchor-todo-api` 查看日志；检查 MySQL 是否运行、端口 4000 是否被占用 |
| **前端 404** | 确认 `frontend/dist/` 和 `todo-h5/dist/` 已生成；Nginx 是否运行（`tasklist /fi "imagename eq nginx.exe"`） |
| **API 请求 502** | 后端服务是否在线（`pm2 list`）；Nginx `proxy_pass` 指向 `127.0.0.1:4000` |
| **数据库迁移失败** | 检查 `_prisma_migrations` 表；`prisma migrate resolve` 修复；或手动备份后 `prisma db push` |
| **CORS 错误** | 确认通过 Nginx 代理访问（同一端口），不要直连 4000 |
| **上传功能异常** | 确认 `backend/uploads/` 目录存在且可写 |
| **新建团队不显示** | 需同时更新组织架构（`orgUnit` 表），直播间空余录入表单从 `fetchOrgTree()` 读取 |
