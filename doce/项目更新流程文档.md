# 项目更新流程文档

## 最近一次重要回归修复说明（2026-06）

本次对“日常任务发布链路”做过一次回归修复，后续更新此模块时必须重点回归以下三项：

1. 日常任务能否从草稿成功发布为 `active` 或 `scheduled`
2. 新发布的 `immediate` 日常任务是否会自动结束旧 `active` 任务
3. 发布成功后前端是否返回正确页面（`/tasks/issue/daily?scopeOrgId=...`）

### 本次回归的真实问题
- 发布请求成功但任务仍停留在草稿态
- 新任务发布后旧任务未结束，列表中同时出现两个进行中任务
- 发布成功后前端跳到了模板库，而不是日常任务发放页
- 草稿归属与发布校验被放宽后，出现“发布草稿与实际生效任务不一致”的风险

### 当前稳定约束
- 日常任务草稿复用依赖：`createdBy + createdByOrgId + ownerScopePath`
- `ownerScopePath` 对旧任务自动结束、待生效切换、草稿复用都非常关键
- 不能随意把日常任务的发布权限判断改成“只要 target 命中基地即可”
- `endOtherActiveDailyAssignments` 不能在 `ownerScopePath` 为空时直接返回
- “创建并配置排除”属于快捷链路，必须显式使用 `forceNewDraft`，禁止回退复用旧 `daily draft assignment`
- 普通“保存当前修改 / 下一步”与快捷“创建并配置排除”必须保持两套不同语义：
  - 普通流程：允许复用当前草稿
  - 快捷流程：必须从刚保存的模板草稿强继承出新的发放草稿

### 建议的最小回归测试
每次修改日常任务发布相关代码后，至少手测：

1. 选择一个基地，新建草稿并发布 `immediate`
2. 确认该草稿状态从 `draft` 变为 `active`
3. 再新建第二个草稿并发布 `immediate`
4. 确认第一个任务变为 `ended`，第二个任务是唯一 `active`
5. 确认页面发布成功后回到日常任务发放页，而不是模板库


## 1. 文档目标

本文档用于指导当前 `D:\Pendingweb` 项目后续版本更新时的标准操作流程。

适用场景：

- 开发电脑完成代码修改
- 代码通过 Git 推送到仓库
- 公司电脑作为正式运行环境
- FRP 地址保持不变
- 需要稳定更新 PC、H5 与后端

本文档重点说明：

1. 开发电脑要做什么
2. Git 应该推什么，不该推什么
3. 公司电脑更新时要做什么
4. 哪些情况需要重启后端
5. 哪些情况需要 reload Nginx
6. 如何做更新后的检查

---

## 2. 更新的基本原则

### 2.1 Git 只管理源码和配置

应该推送到 Git 的内容：

- `backend/src`
- `backend/prisma`
- `frontend/src`
- `todo-h5/src`
- `package.json`
- `vite.config.ts`
- `tsconfig.json`
- 文档与脚本

不应该推送到 Git 的内容：

- `node_modules`
- `dist`
- `.env`
- 上传文件目录
- 日志和缓存文件

当前项目已经通过 `.gitignore` 做了基础约束。

---

## 2.2 正式环境不跑开发命令

正式环境更新后不建议使用：

- `npm run dev`
- `vite`
- `tsx watch`

正式环境应使用：

- build 产物
- Nginx 托管前端
- PM2 守护后端

---

## 3. 开发电脑上的更新流程

## 3.1 修改代码

在开发电脑完成以下修改之一：

- 后端逻辑
- PC 前端页面
- H5 页面
- 配置脚本
- 文档

---

## 3.2 本地测试

建议至少验证：

1. 项目能正常启动
2. 主要页面能打开
3. 关键功能能正常操作
4. 无明显控制台报错

如果改动较大，建议额外检查：

- 登录
- 身份切换
- 上传图片
- 提醒功能
- 任务提交流程

---

## 3.3 提交 Git

示例：

```bash
git add .
git commit -m "优化 H5 界面与路由懒加载"
git push
```

说明：

- 建议 commit message 说明本次改动目的
- 不要把真实 `.env`、`dist`、`node_modules` 提交上去

---

## 4. 公司电脑上的更新流程

以下流程适用于：

- 公司电脑已部署好项目
- Git 仓库可正常访问
- 后端已由 PM2 启动
- Nginx 已配置完成

---

## 4.1 拉取最新代码

进入项目源码目录，例如：

```bash
cd C:\deploy\source
```

拉取最新代码：

```bash
git pull
```

---

## 4.2 如有依赖变化，安装依赖

如果本次更新涉及 `package.json` 改动，建议执行：

```bash
npm install
```

如果 H5 是独立依赖管理，也建议进入 H5 目录执行：

```bash
cd todo-h5
npm install
cd ..
```

如果只是普通页面改动，且依赖没有变化，这一步通常可跳过。

---

## 4.3 重新构建

### 1. 构建后端

```bash
npm run build -w backend
```

### 2. 构建 PC 前端

```bash
npm run build -w frontend
```

### 3. 构建 H5

```bash
cd todo-h5
npm run build
cd ..
```

---

## 4.4 更新部署目录

### PC

把新的：

- `frontend/dist`

同步到正式静态目录，例如：

- `C:/deploy/pc-web`

### H5

把新的：

- `todo-h5/dist`

同步到正式静态目录，例如：

- `C:/deploy/h5-web`

### 后端

如果后端直接在源码目录 build 后运行，则无需额外复制。

如果后端部署目录独立，则需要把新的：

- `backend/dist`
- 必要时连同 `prisma`

同步到正式后端目录。

---

## 4.5 重启后端服务

如果本次更新包含后端改动，执行：

```bash
pm2 restart anchor-todo-api
```

查看状态：

```bash
pm2 list
```

查看日志：

```bash
pm2 logs anchor-todo-api
```

如果本次只改了前端页面，通常不需要重启后端。

---

## 4.6 是否需要重载 Nginx

大多数情况下：

## 不需要

因为普通更新通常只是：

- 前端页面变了
- H5 页面变了
- 后端逻辑变了

这些更新不一定会改变 Nginx 规则。

---

## 4.7 哪些情况需要 reload Nginx

只有以下情况之一，才通常需要执行：

```bash
nginx -s reload
```

### 需要 reload 的情况

1. 改了 Nginx 配置文件
2. 改了 PC 静态目录
3. 改了 H5 静态目录
4. 改了 `/api` 代理规则
5. 改了 `/uploads` 映射
6. 改了 gzip 配置
7. 改了缓存策略
8. 改了路由回退规则

如果只是页面代码更新，通常不用 reload Nginx。

---

## 5. 按改动类型区分更新动作

## 5.1 只改 PC 前端

需要做：

1. `git pull`
2. `npm run build -w frontend`
3. 更新 `pc-web` 目录

通常不需要：

- 重启后端
- reload Nginx

---

## 5.2 只改 H5

需要做：

1. `git pull`
2. `cd todo-h5 && npm run build`
3. 更新 `h5-web` 目录

通常不需要：

- 重启后端
- reload Nginx

---

## 5.3 只改后端

需要做：

1. `git pull`
2. `npm run build -w backend`
3. `pm2 restart anchor-todo-api`

通常不需要：

- reload Nginx

---

## 5.4 改了数据库结构

需要做：

1. `git pull`
2. 执行数据库相关操作
3. 重新 build 后端
4. 重启后端

注意：

- 这类更新前建议先备份数据库
- 如果涉及 Prisma schema 变化，要按实际数据库变更流程谨慎执行

---

## 5.5 改了 Nginx 配置

需要做：

1. 更新 Nginx 配置文件
2. 执行：

```bash
nginx -s reload
```

如果配置存在语法问题，应先修复再 reload。

---

## 6. 更新后的检查流程

每次更新后建议至少做一次冒烟测试。

## 6.1 页面访问检查

确认：

- PC 首页能打开
- H5 首页能打开
- 刷新子路由不 404
- 静态资源正常加载

---

## 6.2 后端接口检查

确认：

- `/api/health` 正常
- 登录接口正常
- 常用业务接口正常

---

## 6.3 关键功能检查

建议至少检查：

### PC

- 登录
- 身份切换
- 组织 / 账号页能打开
- 待办看板能打开

### H5

- 登录
- 待办列表
- 详情页
- 提醒页

### 公共

- 图片上传可用
- 上传图片可访问
- 提交功能正常

---

## 7. 如果更新失败，优先排查哪些

## 7.1 前端页面没更新

优先检查：

1. 是否重新 build 了前端/H5
2. 是否正确覆盖了 `dist`
3. 浏览器缓存是否仍在使用旧资源
4. Nginx 是否仍指向旧目录

---

## 7.2 后端接口异常

优先检查：

1. 是否重新 build backend
2. 是否重启 PM2 服务
3. `.env` 是否正确
4. PM2 日志是否报错

---

## 7.3 上传图片访问异常

优先检查：

1. 上传目录是否存在
2. Nginx `/uploads` 映射是否正确
3. 文件权限是否正常

---

## 7.4 飞书登录异常

优先检查：

1. FRP 地址是否可访问
2. 回调地址是否变化
3. 后端飞书配置是否正确

---

## 8. 建议的最简更新流程

如果是一次普通版本更新，可按下面最简流程执行：

### 开发电脑

1. 改代码
2. 测试
3. `git commit`
4. `git push`

### 公司电脑

1. `git pull`
2. 需要时 `npm install`
3. build：backend / frontend / todo-h5
4. 更新静态目录
5. 如有后端改动则 `pm2 restart`
6. 如有 Nginx 规则改动则 `nginx -s reload`
7. 做冒烟测试

---

## 9. 最终建议

为了保证后续更新稳定，建议长期坚持：

1. Git 只管源码与配置模板
2. `dist` 不进仓库
3. `.env` 不进仓库
4. 上传目录单独管理
5. 更新后做最小化冒烟测试
6. 数据库结构更新前先备份

这样后续无论是你自己，还是同事接手，更新流程都会更清晰、更稳定。
