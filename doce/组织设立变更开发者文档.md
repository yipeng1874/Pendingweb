本文档用于说明“组织设立变更”模块的前后端实现、数据模型、接口语义、权限机制、分级加载逻辑、维护方式和改造注意点，方便：

- 后续开发人员快速接手

- 厅抖音号 / UID 唯一约束
- 组织路径 `path` 与范围联动

## 2. 模块概述

“组织设立变更”模块是系统组织域的核心后台模块，负责维护全平台组织树结构。

当前组织层级固定为四层：

1. `HQ`：总部
2. `BASE`：基地
3. `TEAM`：团队
4. `HALL`：厅

页面主要能力：

1. 展示当前身份可见范围内的组织树
2. 查看组织档案详情
3. 创建符合层级规则的下级组织
4. 对厅组织录入业务信息
5. 批量导入厅组织
6. 编辑组织信息
7. 暂停 / 恢复组织
8. 迁移组织到新的合法上级
9. 删除空组织

这个模块不仅是“组织资料维护页面”，还是很多下游业务的上游主数据来源，影响：
- 账号权限范围
- 主播归属
- 任务记录主体
- 组织树展示
- 身份 scopePath 计算

因此该模块属于高敏核心模块。

## 4. 功能说明

### 4.1 左侧组织树

页面左侧展示当前身份可见范围内的组织树。

能力包括：
- 选中节点
- 默认展开当前身份路径
- 全部折叠 / 全部展开
- 根据选中组织刷新右侧详情区

说明：
- 当前“组织设立变更”页面仍采用一次性加载完整可见组织树的方式
- 即页面初始化直接请求 `/orgs/tree`
- 尚未像“组织账号管理”那样改造成按父节点懒加载
- 这是当前有意保留的实现策略，便于组织迁移、批量建厅、删除可行性判断等全局操作共用同一份完整树数据

### 4.2 组织详情区

选中组织后，右侧展示该组织档案，包括：
- 组织编码
- 层级路径
- 负责人
- 联系电话
- 厅抖音号
- 厅抖音 UID
- 运营经纪人
- 备注
- 当前状态

其中：
- 非厅组织通常只使用基础字段
- 厅组织使用完整业务字段集

### 4.3 新建下级组织

根据当前选中组织，页面自动推导允许创建的下级类型：

| 当前组织 | 可创建下级 |
|---|---|
| HQ | BASE |
| BASE | TEAM |
| TEAM | HALL |
| HALL | 无 |

其中：
- `TEAM` 的组织编码由系统自动生成
- `HALL` 的组织编码由 `HALL-${douyinUid}` 自动生成

### 4.4 批量新建厅

仅当当前选中组织为 `TEAM` 时显示。

支持：
- 上传 CSV
- 解析表头
- 预览厅名称 / 编码 / 负责人 / 抖音 UID
- 批量调用后端创建厅组织

### 4.5 编辑组织档案

支持修改：
- `name`
- `principalName`
- `contactPhone`
- `brokerName`
- `remark`
- 厅组织还支持维护 `douyinNo` / `douyinUid`

注意：
- 非厅组织禁止维护 `douyinNo` / `douyinUid`
- 厅组织编辑时会触发唯一性校验

### 4.6 暂停 / 恢复组织

支持对选中组织做：
- 暂停：`status = paused`
- 恢复：`status = active`

### 4.7 迁移组织

支持将组织迁移到符合层级规则的新上级下，例如：
- 基地 -> 总部
- 团队 -> 基地
- 厅 -> 团队

并联动更新：
- 当前组织路径 `path`
- 子孙组织路径 `path`
- 用户身份 `scopePath`
- 厅组织相关任务主体标识（如 UID 变化时）

### 4.8 删除组织

仅支持删除“空组织”，即：
- 没有下级组织
- 没有绑定身份
- 没有主播档案
- 总部不可删除

## 5. 页面实现说明

### 5.1 页面状态设计

`OrganizationPage.tsx` 维护的主要状态有：

1. `orgs`：组织列表
2. `selected`：当前选中组织
3. `form`：创建下级组织表单
4. `editForm`：编辑组织表单
5. `isEditing`：是否处于编辑态
6. `moveParentId`：迁移目标上级组织 ID
7. `message / error`：操作反馈
8. `collapsedIds`：组织树折叠状态
9. `batchRows`：批量导入厅的解析结果

### 5.1.1 当前加载策略说明

页面初始化时：
1. 调用 `/orgs/tree`
2. 获取当前身份可见范围内的完整组织平铺列表
3. 前端用 `buildOrgTree` 构建左侧树
4. 再基于完整组织集计算：
   - 默认展开路径
   - 默认选中组织
   - 迁移候选组织
   - 是否存在子级
   - 批量建厅后的回显定位

这意味着“组织设立变更”当前不是懒加载树页面，而是“完整树编辑页”。
在当前业务下这样做是合理的，因为该页面更偏组织维护后台，而不是高频大列表浏览页。

### 5.2 表单模式

页面有两套表单：

#### 5.2.1 新建表单 `form`
用于新建下级组织。

#### 5.2.2 编辑表单 `editForm`
用于编辑当前组织档案。

两套数据分离，避免“创建态”和“编辑态”互相污染。

### 5.3 组织类型驱动页面行为

前端通过：
- `nextTypeMap`
- `parentTypeMap`

来推导：
- 当前组织能创建什么下级
- 当前组织能迁移到哪类父组织下

### 5.4 厅组织特殊表单规则

厅组织有额外必填字段：
- 负责人
- 联系电话
- 厅抖音号
- 厅抖音 UID
- 运营经纪人
- 备注

前端通过 `requiredHallKeys` + `isHallFormValid` 做提交前校验。

### 5.5 批量导入 CSV 逻辑

页面内置了：
- `parseCsvLine`
- `parseBatchHallText`

用于把 CSV 文本解析为结构化行数据。

兼容的列名映射包括：
- `组织名称` / `厅名称` / `名称` -> `name`
- `厅抖音号` / `抖音号` -> `douyinNo`
- `厅抖音UID` / `抖音UID` -> `douyinUid`

这使得实施侧上传不同口径的 CSV 时仍可识别。

## 6. 后端实现说明

### 6.1 接口入口

组织模块接口统一在：
- `backend/src/modules/organization/routes.ts`

全局先挂载：
- `authRequired`
- `identityRequired`

然后每个接口再叠加权限控制。

### 6.2 核心辅助函数

#### 6.2.1 `text`
统一把输入转成去首尾空格的字符串。

#### 6.2.2 `normalizeDouyinNo`
去除抖音号中的空白字符。

#### 6.2.3 `normalizeDouyinUid`
去除 UID 中的空白字符。

#### 6.2.4 `makeHallOrgCode`
厅组织编码生成规则：

```text
HALL-${douyinUid}
```

#### 6.2.5 `generateTeamOrgCode`
团队编码生成规则：
- 读取当前基地下已存在团队编码
- 生成类似 `基地编码 + A1 ~ Z9`

例如：
- `BASE001A1`
- `BASE001A2`
- ...
- `BASE001Z9`

#### 6.2.6 `resolveOrgCode`
根据组织类型决定编码生成方式：
- `TEAM`：系统生成
- `HALL`：由 `douyinUid` 推导
- 其他层级：使用手工填写的 `orgCode`

#### 6.2.7 `assertHallIdentityUnique`
检查厅组织的：
- `douyinNo`
- `douyinUid`

是否与已存在厅组织重复。

#### 6.2.8 `syncOrganizationRelations`
当厅组织 UID 或组织路径发生变化时，联动同步：
- `userIdentity.scopePath`
- 厅下主播账号关联身份路径
- 任务记录 `subjectKey`
- 子孙组织路径

这是组织变更后保持下游数据一致性的关键函数。

## 7. 业务规则

### 7.1 固定层级规则

组织层级固定如下：

```text
HQ -> BASE -> TEAM -> HALL
```

不允许：
- BASE 直接挂 TEAM 之外的其他层级
- HALL 直接挂 BASE
- TEAM 挂到 TEAM 下

### 7.2 组织编码规则

#### HQ / BASE
- 手工填写 `orgCode`

#### TEAM
- 系统自动生成
- 基于所属 BASE 编码 + `A1 ~ Z9`

#### HALL
- 系统自动生成
- 编码规则：`HALL-${douyinUid}`

### 7.3 厅字段唯一规则

当前已经按你的要求落了业务约束：

1. `douyinNo` 唯一
2. `douyinUid` 唯一
3. 新建、批量新建、编辑过程都校验
4. 数据库层也设唯一约束

### 7.4 批量导入规则

批量新建厅时校验：
- 行内必填字段是否完整
- 表内抖音号是否重复
- 表内 UID 是否重复
- 数据库中抖音号是否已存在
- 数据库中 UID 是否已存在
- 推导出的组织编码是否已存在

### 7.5 删除规则

删除组织的前提：
- 不是 `HQ`
- 没有子组织
- 没有 `UserIdentity`
- 没有 `AnchorProfile`

### 7.6 迁移规则

迁移组织时必须满足：
- 新父组织存在
- 不能迁移到自己或自己下级
- 层级必须匹配
- 总部不可迁移

### 7.7 编辑规则

- 非厅组织不能编辑 `douyinNo / douyinUid`
- 厅组织编辑 UID 后，`orgCode` 和 `path` 会联动变化
- UID 若冲突会报错
- 抖音号若冲突会报错

## 8. 权限与界面可见性

### 8.1 页面访问权限

前端路由限制在：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`

并且必须具有：
- `org:view`

因此：
- `HALL_MANAGER` 不能进入此页面
- `ANCHOR` 不能进入此页面

### 8.2 接口权限矩阵

| 接口 | 权限 | 说明 |
|---|---|---|
| GET `/orgs/tree` | `org:view` | 查看组织树 |
| POST `/orgs` | `org:create` | 创建组织 |
| POST `/orgs/halls/batch` | `org:create` | 批量创建厅 |
| PATCH `/orgs/:id` | `org:update` | 编辑组织 |
| POST `/orgs/:id/pause` | `org:pause` | 暂停组织 |
| POST `/orgs/:id/restore` | `org:restore` | 恢复组织 |
| POST `/orgs/:id/move` | `org:update` | 迁移组织 |
| DELETE `/orgs/:id` | `org:update` | 删除组织 |

### 8.3 默认角色权限情况

基于 `sync-role-permissions.ts`，组织模块相关权限大致如下：

| 角色 | org:view | org:create | org:update | org:pause | org:restore |
|---|---:|---:|---:|---:|---:|
| DEV_ADMIN | 是 | 是 | 是 | 是 | 是 |
| HQ_ADMIN | 是 | 是 | 是 | 是 | 是 |
| BASE_ADMIN | 是 | 是 | 是 | 否 | 否 |
| TEAM_ADMIN | 是 | 是 | 是 | 否 | 否 |
| HALL_MANAGER | 是 | 否 | 否 | 否 | 否 |
| ANCHOR | 否 | 否 | 否 | 否 | 否 |

注意：
- 页面路由没有对 `HALL_MANAGER` 放行，即使它有 `org:view` 也无法进入页面
- `pause/restore` 权限只给了 `DEV_ADMIN/HQ_ADMIN`

## 9. API 接口释义

以下默认省略网关前缀 `/api`。

### 9.1 GET `/orgs/tree`

#### 用途
加载当前身份可见范围内的组织树。

#### 权限
- `org:view`

#### 返回
平铺组织列表，前端自行构建树。

#### 特殊逻辑
- 若当前身份不是 `DEV_ADMIN`，则按 `scopePath` 过滤

---

### 9.2 POST `/orgs`

#### 用途
创建组织。

#### 请求体主要字段
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| orgType | string | 是 | 组织类型 |
| parentId | string | 否 | 上级组织 ID |
| name | string | 是 | 组织名称 |
| orgCode | string | 否 | HQ/BASE 等需手填 |
| principalName | string | 厅必填 | 负责人 |
| contactPhone | string | 厅必填 | 联系电话 |
| douyinNo | string | 厅必填 | 厅抖音号 |
| douyinUid | string | 厅必填 | 厅抖音 UID |
| brokerName | string | 厅必填 | 运营经纪人 |
| remark | string | 厅必填 | 备注 |

#### 关键逻辑
1. 校验组织名称
2. 校验父子层级合法性
3. 厅组织校验必填字段
4. 厅组织校验抖音号/UID 唯一
5. TEAM/HALL 自动生成编码
6. 在事务内创建组织
7. 若 TEAM 编码冲突，自动重试最多 3 次

#### 并发优化说明
TEAM 创建时做了冲突重试，避免并发下自动编码撞车立即失败。

---

### 9.3 POST `/orgs/halls/batch`

#### 用途
批量创建厅组织。

#### 请求体
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| parentId | string | 是 | 团队 ID |
| rows | array | 是 | 厅数据列表 |

#### 每行字段
- `name`
- `principalName`
- `contactPhone`
- `douyinNo`
- `douyinUid`
- `brokerName`
- `remark`

#### 后端校验
1. 必须选中 TEAM 作为父组织
2. rows 不能为空
3. 表内必填字段校验
4. 表内 `douyinNo / douyinUid` 不能重复
5. 库内 `douyinNo / douyinUid / orgCode` 不能重复
6. 通过事务批量创建

---

### 9.4 PATCH `/orgs/:id`

#### 用途
编辑组织档案。

#### 允许修改字段
- `name`
- `principalName`
- `contactPhone`
- `brokerName`
- `remark`
- 厅组织额外允许 `douyinNo`
- 厅组织额外允许 `douyinUid`

#### 特殊逻辑
1. 若是厅：
   - `douyinNo` 不能为空
   - `douyinUid` 不能为空
   - 两者都要做唯一校验
2. 若 UID 变化：
   - `orgCode` 自动变化
   - `path` 自动变化
   - 子孙路径自动变化
   - 身份范围 / 任务主体同步变化
3. 非厅组织若传 `douyinNo / douyinUid` 会报错

---

### 9.5 POST `/orgs/:id/pause`

#### 用途
暂停组织。

#### 权限
- `org:pause`

#### 处理方式
直接把 `status` 改成 `paused`。

---

### 9.6 POST `/orgs/:id/restore`

#### 用途
恢复组织。

#### 权限
- `org:restore`

#### 处理方式
直接把 `status` 改成 `active`。

---

### 9.7 POST `/orgs/:id/move`

#### 用途
迁移组织到新的上级。

#### 请求体
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| parentId | string | 是 | 新父组织 ID |

#### 校验点
1. 原组织和新父组织必须存在
2. 总部不能迁移
3. 不能迁移到自己或自己的下级下
4. 新父层级必须匹配

#### 联动更新
- 当前组织 `parentId`
- 当前组织 `path`
- 当前组织 `depth`
- 子孙组织 `path/depth`
- 身份 `scopePath`
- 厅关联数据路径

---

### 9.8 DELETE `/orgs/:id`

#### 用途
删除组织。

#### 权限
- `org:update`

#### 删除条件
1. 组织存在
2. 不是 HQ
3. 没有下级组织
4. 没有身份数据
5. 没有主播档案

#### 返回
- `{ deleted: true }`

## 11. 页面数据加载与分级传输逻辑

> 这里的“懒加载分级传输”指按用户操作步骤逐层取数，而不是一次把所有数据全部打包传给前端。

### 11.1 总体策略

组织设立变更页面采用“主树 + 按选择加载详情上下文”的分层加载方式：

1. 先加载组织树
2. 选中组织后在前端切换详情区域
3. 创建/编辑/迁移/删除后重新拉组织树
4. 批量导入只在用户上传文件后进行本地解析
5. 提交批量导入时才把结构化数据发给后端

### 11.2 第一层：组织树加载

页面初始化调用：
- `GET /orgs/tree`

特点：
- 返回当前身份可见范围内的组织平铺列表
- 前端本地构造成树
- 同时根据当前身份决定默认展开节点和默认选中节点

### 11.3 第二层：选中组织后的详情切换

当前实现中，组织详情不是单独请求接口，而是：
- 从 `orgs` 列表中直接找到当前选中组织
- 用该对象填充右侧详情区与编辑表单

好处：
- 少一次详情接口调用
- 组织详情切换速度快

限制：
- 如果未来组织字段越来越重，建议拆出 `GET /orgs/:id`

### 11.4 第三层：批量导入本地预处理

批量导入不是直接把文件原样传给后端，而是：
1. 前端读取 CSV 文件文本
2. 本地解析为 `BatchHallRow[]`
3. 在页面预览部分关键字段
4. 用户确认后才提交到 `/orgs/halls/batch`

这种方式的优点：
- 减少后端文件上传处理复杂度
- 能在提交前让用户看见数据预览
- 有利于前端先做字段级纠错

### 11.5 第四层：操作后刷新

组织创建、编辑、迁移、暂停、恢复、删除、批量创建完成后，都会重新调用：
- `/orgs/tree`

这是当前页面的数据一致性策略：
- 不做局部 patch
- 直接用整棵树刷新保证最终一致性

### 11.6 当前不是分页式加载

当前组织模块没有分页：
- `/orgs/tree` 一次返回当前可见范围内全部组织
- 右侧详情不单独分页
- 批量导入是一次性提交

在中大型组织树下，如果节点数非常多，可能出现：
- 组织树首屏加载变慢
- 创建/编辑后整树刷新开销大

### 11.7 后续可优化方向

1. 组织树支持按节点展开异步加载子节点
2. 增加 `GET /orgs/:id`
3. 增加“局部刷新”代替全量树刷新
4. 批量导入改为服务端异步任务方式

## 12. 并发与一致性处理

### 12.1 TEAM 编码并发问题

团队组织编码原本存在“先查后建”的并发竞争窗口。

目前已做优化：
1. 在事务中重新计算编码
2. 如果 `org_code` 唯一冲突，自动重试最多 3 次

因此在多人同时给同一基地创建 TEAM 时，成功率更高。

### 12.2 HALL 唯一冲突处理

厅组织创建/编辑时，除了前置查询外，还依赖数据库唯一索引兜底：
- `douyin_no`
- `douyin_uid`
- `org_code`

若并发下出现同值提交，数据库会返回唯一冲突，后端转成人类可读错误。

### 12.3 路径联动一致性

迁移组织、修改厅 UID 时，会在事务内同步更新：
- 当前组织
- 子组织路径
- 身份范围
- 任务记录主体键

这样能避免“组织改了，权限范围没改”的脏数据。

## 13. 维护指南

### 13.1 新维护者优先理解的 4 个概念

1. `orgType`：组织层级类型
2. `orgCode`：组织业务编码
3. `path`：组织树路径，也是权限范围关键字段
4. `scopePath`：身份的数据可见范围

### 13.2 常见改动场景

#### 场景 A：新增组织字段
需要同时修改：
1. `schema.prisma`
2. `frontend/src/types.ts`
3. `OrganizationPage.tsx`
4. `backend/src/modules/organization/routes.ts`
5. 数据库迁移脚本

#### 场景 B：增加新的组织层级
这是高风险改造，需要同步改：
1. 前端 `nextTypeMap`
2. 前端 `parentTypeMap`
3. 前端 `orgTypeMeta`
4. 后端 `parentTypeMap`
5. 后端 `resolveOrgCode`
6. 迁移规则
7. 删除规则
8. 权限矩阵
9. 路径深度逻辑
10. 下游使用 `orgType` 的任务/账号/主播模块

#### 场景 C：修改厅编码规则
需要同步改：
1. `makeHallOrgCode`
2. `previewHallOrgCode`
3. 批量导入预览逻辑
4. 组织 UID 编辑逻辑
5. 任何依赖 `HALL-${uid}` 的下游逻辑

#### 场景 D：放开 HALL_MANAGER 页面访问
需要修改：
1. `frontend/src/App.tsx` 页面路由放行
2. 权限校验是否允许编辑/新建/迁移/删除
3. 后端 scopePath 操作范围是否需要补加强校验

### 13.3 排障建议

#### 问题 1：新建团队报编码冲突
先检查：
- 是否并发提交
- 是否同一基地下编码池已接近耗尽
- 数据库是否已存在手工脏数据

#### 问题 2：新建厅提示抖音号或 UID 重复
检查：
- 是否表单里有空格导致看起来不同实际相同
- 是否历史数据已占用
- 是否批量导入文件表内重复

#### 问题 3：组织迁移后权限异常
检查：
- `org_units.path`
- `user_identities.scope_path`
- 当前身份是否重新拉取
- 是否存在下游缓存

#### 问题 4：删除组织失败
检查：
- 是否仍有子组织
- 是否还有用户身份
- 是否还有主播档案

### 13.4 推荐补充的审计能力

建议后续为以下操作记录审计日志：
1. 创建组织
2. 编辑组织
3. 修改厅抖音号/UID
4. 暂停/恢复组织
5. 迁移组织
6. 删除组织
7. 批量导入厅

因为这些操作都会影响下游权限和业务归属。

## 14. 风险点与改进建议

### 14.1 当前设计优点

1. 层级规则清晰固定
2. TEAM/HALL 自动编码减少人工出错
3. 厅抖音号 / UID 已双重唯一约束
4. 批量导入有表内 + 库内双重校验
5. 组织迁移和 UID 修改具备联动同步机制

### 14.2 当前可改进点

1. **删除说明文案与后端规则不完全一致**
   - 前端文案写到“没有任务目标”
   - 后端实际仅校验子组织/身份/主播

2. **写接口未见统一 scopePath 二次校验**
   - 查询树按 scopePath 过滤
   - 但创建/编辑/迁移/删除是否完全限制在当前身份范围内，需要后续补强更稳妥

3. **组织树每次操作后整树刷新**
   - 数据量大时性能一般

4. **批量导入目前不做更多格式校验**
   - 如手机号格式、特殊字符长度等

5. **暂停组织只是状态标记**
   - 当前并未看到更多业务冻结联动

### 14.3 推荐优化优先级

高优先级：
1. 写接口补 scopePath 范围校验
2. 组织树改局部刷新/按节点加载
3. 补充组织操作审计日志

中优先级：
1. 删除规则与文案统一
2. 批量导入增加格式校验
3. 组织详情拆独立接口

低优先级：
1. 导入模板下载
2. 批量迁移/批量暂停
3. 组织操作历史时间线

## 16. 维护结论

“组织设立变更”模块本质上是系统组织主数据管理中枢。

它同时承担：
- 组织树定义者
- 权限范围来源者
- 厅业务主信息维护者
- 下游账号/主播/任务的组织归属基础

因此维护该模块时，必须同时考虑四类影响：

1. **组织结构是否正确**
2. **编码是否唯一且可追踪**
3. **路径是否与权限范围一致**
4. **组织变更是否已同步影响下游数据**

如果未来继续扩展，建议优先投入在：
- 审计日志
- 范围安全校验
- 局部刷新与树性能优化
- 删除规则与业务规则统一

这会显著提升该模块的长期可维护性与安全性。
