## 2. 项目定位

`todo-h5` 是一个独立于 PC 前端的移动端 H5 项目，用于承接“我的待办”相关核心能力。

当前设计原则：

1. 独立开发，不直接复用 PC 页面组件
2. 尽量复用现有后端 API、认证机制与业务口径
3. 以手机端交互为主，页面结构和视觉风格做移动化收敛
4. 优先保障“执行闭环”，而不是只做只读展示

当前 H5 已覆盖：

- 登录
- 身份选择
- 我的待办列表
- 任务详情与子任务执行
- 个人提醒

## 4. 路由结构

H5 路由入口在：

- `todo-h5/src/App.tsx`

当前页面路由如下：

| 路径 | 页面 | 说明 |
|---|---|---|
| `/login` | `LoginPage` | 登录页 |
| `/identity` | `IdentityPage` | 身份选择页 |
| `/todos` | `TodoListPage` | 我的待办列表 |
| `/todos/:id` | `TodoDetailPage` | 任务详情 |
| `/reminders` | `ReminderPage` | 个人提醒 |

路由保护规则：

### 4.1 `ProtectedRoute`
用于要求必须登录。

当前作用于：
- `/identity`

### 4.2 `IdentityRequiredRoute`
用于要求：
- 已登录
- 已选择当前身份

当前作用于：
- `/todos`
- `/todos/:id`
- `/reminders`

这意味着 H5 的业务主流程是：

1. 登录
2. 选择身份
3. 进入待办主流程

## 5. 页面职责说明

## 5.1 登录页 `LoginPage.tsx`

当前支持两种登录方式：

1. 手机号 + 密码登录
2. 飞书登录

### 页面职责
- 承接账号登录
- 获取飞书登录所需组织/团队/企业配置数据
- 登录成功后写入用户、token、身份列表
- 根据返回结果跳转到身份页

### 当前界面特征
- 已做移动端收敛
- 顶部文案精简
- 登录卡片与飞书登录卡片采用统一 card 规范

### 主要接口
- `authApi.login(phone, password)`
- `authApi.getFeishuBaseOptions()`
- `authApi.getFeishuTeamOptions(baseOrgId)`
- `authApi.getFeishuConfigs(baseOrgId, teamOrgId)`
- `authApi.completeFeishuLogin(code, state)`

## 5.2 身份选择页 `IdentityPage.tsx`

### 页面职责
- 展示当前账号下的全部身份
- 选择一个身份作为当前上下文
- 写入 `currentIdentity`
- 跳转到 `/todos`

### 当前展示内容
每个身份卡片显示：
- 主显示名：主播昵称 / 组织名称 / 角色编码
- 角色编码
- 组织名称与组织类型
- “进入”标签

### 状态来源
来自 `useAuthStore`：
- `identities`
- `user`
- `setCurrentIdentity`
- `logout`

---

## 5.3 我的待办列表页 `TodoListPage.tsx`

### 页面职责
- 拉取当前身份下的待办记录
- 拉取当前用户的个人提醒概要
- 提供日常任务 / 临时任务切换
- 提供临时任务子分类切换
- 进入任务详情页
- 进入个人提醒页

### 当前信息架构
页面分为几块：

1. 顶部身份区
   - 当前角色
   - 当前组织 / 主播名
   - 刷新 / 身份切换 / 退出

2. 待办主切换区
   - 日常任务
   - 临时任务

3. 临时任务子切换区
   - 待开始
   - 进行中

4. 个人提醒入口区
   - 进行中数量
   - 重要数量
   - 紧急 / 逾期标记

5. 待办卡片列表

### 列表卡片当前展示内容
- 任务类型标签
- 状态标签
- 标题
- 进度
- 主体
- 截止时间
- 发布人

### 当前交互原则
- 列表页只做“浏览 + 进入”
- 不在列表内直接展开复杂编辑器
- 复杂执行逻辑统一放在详情页

### 主要接口
- `taskApi.getMyRecords()`
- `taskApi.getReminders("active")`

## 5.4 任务详情页 `TodoDetailPage.tsx`

### 页面职责
- 拉取单条任务详情
- 展示任务头信息
- 按未完成 / 已完成拆分子任务
- 承接 6 种子任务类型的执行交互
- 承接整条任务提交

### 顶部信息区当前展示
- 任务标题
- 任务说明
- 当前状态
- 当前进度
- 主体
- 截止时间
- 发布人

### 子任务分组
当前拆分为：
- 待处理子任务
- 已完成子任务

### 支持的子任务类型
当前已支持：
- `QA`
- `SINGLE_CHOICE`
- `MULTI_CHOICE`
- `FILL_BLANK`
- `LINK`
- `ATTACHMENT`

### 不同子任务的交互方式

#### QA
- 文本输入框
- 按钮提交

#### SINGLE_CHOICE
- 点击选项即提交

#### MULTI_CHOICE
- 先选择多个选项
- 再点击确认提交

#### FILL_BLANK
- 一键确认完成

#### LINK
- 先打开外链
- 再点击“已完成学习并确认”

#### ATTACHMENT
- 上传图片
- 支持预览
- 上传后确认提交

### 上传能力说明
上传接口仍然复用当前后端上传逻辑：
- 使用 `fetch` + `FormData`
- 需带 token 与 `X-Identity-Id`
- 当前前端限制：单张图片 ≤ 1MB

### 底部提交区
页面底部使用粘性提交条：
- 展示是否可提交
- 展示提交提示
- 支持提交整条任务

### 主要接口
- `taskApi.getRecord(id)`
- `taskApi.submitItemRecord(...)`
- `taskApi.submitRecord(id)`
- `taskApi.upload(taskItemRecordId, file)`

## 5.5 个人提醒页 `ReminderPage.tsx`

### 页面职责
- 查看当前用户的个人提醒
- 区分进行中 / 已逾期
- 新建提醒
- 删除提醒
- 标记提醒完成

### 当前页面结构
1. 顶部导航区
   - 返回
   - 刷新
   - 新建提醒

2. Tab 区
   - 进行中
   - 已逾期

3. 提醒卡片列表

4. 新建提醒弹窗

### 当前提醒卡片展示内容
- 是否重要
- 状态（进行中 / 紧急 / 已逾期 / 已完成）
- 标题
- 备注
- 截止时间
- 创建时间
- 标记完成按钮
- 删除按钮

### 当前新建提醒交互
采用弹窗式，而不是页面内展开。

### 当前日期时间交互
日期和时间选择器采用 H5 抽屉式面板：
- 日期：底部弹出月历选择
- 时间：底部弹出小时 / 分钟双列选择

组件位置：
- `todo-h5/src/components/MiniDateTimePickers.tsx`

### 当前提醒状态口径
后端正式状态仍然是：
- `active`
- `done`

前端另外推导：
- `overdue`
- `urgent`

判断逻辑：
- 已逾期：`remindEnd < now` 且不是 done
- 紧急：剩余时间 ≤ 3 天 且不是 done

### 主要接口
- `taskApi.getReminders(status)`
- `taskApi.createReminder(data)`
- `taskApi.deleteReminder(id)`
- `taskApi.markReminderDone(id)`

## 6. 全局样式与视觉规范

全局样式文件：
- `todo-h5/src/styles.css`

当前 H5 已经过多轮视觉收敛，整体原则是：

1. 偏工具型，不做营销页风格
2. 圆角、阴影、渐变已整体收敛
3. 优先压缩手机端纵向空间
4. 强调执行效率，而不是视觉装饰

### 当前统一规则

#### 6.1 页面容器
- 统一使用 `.page-shell` + `.mobile-page`
- 最大宽度控制在移动端视口友好范围

#### 6.2 卡片
- 统一使用 `.card`
- 重点块可叠加 `.card-strong`
- 卡片圆角、边框、阴影统一由全局管理

#### 6.3 标题体系
- 页面主标题：`.hero-title` / `.topbar-title`
- 卡片标题：`.card-title`
- 辅助文案：`.card-subtitle` / `.section-note`

#### 6.4 按钮体系
- 统一使用 `.btn`
- 主按钮：`.btn-primary`
- 次按钮：`.btn-secondary`
- 幽灵按钮：`.btn-ghost`
- 图标按钮：`.icon-btn`

#### 6.5 标签体系
- 统一使用 `.tag`
- 颜色类：
  - `.tag-blue`
  - `.tag-green`
  - `.tag-slate`
  - `.tag-red`
  - `.tag-purple`

#### 6.6 分段切换
- 统一使用 `.segmented`
- 适合 tab、二级切换、二选一状态选择

#### 6.7 弹窗 / 抽屉
- 遮罩：`.modal-backdrop`
- 面板：`.modal-sheet`

- `todo-h5/src/stores/auth.ts`

当前 store 主要负责：
- `token`
- `user`
- `identities`
- `currentIdentity`
- 登录写入
- 身份切换
- 登出清理

### 使用原则

#### 与认证相关的全局状态放 store
例如：
- token
- 用户信息
- 当前身份

#### 页面局部状态留在页面内部
例如：
- loading
- tab 当前值
- 表单值
- 弹窗开关

## 12. 开发注意事项

### 12.1 不要直接复用 PC 页面组件
PC 页面组件更偏桌面端结构，H5 应继续坚持独立承接。

### 12.2 保持后端口径一致
尤其注意：
- 提醒的 `overdue` 不是正式数据库状态
- 日常任务和临时任务展示口径不同
- 管理式任务更关注“谁完成了哪项”

### 12.3 手机端优先压纵向空间
不要轻易新增：
- 冗长说明
- 大块统计
- 多层嵌套卡片
- 复杂行内编辑

### 12.4 修改样式优先从 `styles.css` 统一收口
如果只是局部页面微调，才在页面里加少量 `style`。
尽量避免再次回到“每页各写一套尺寸”的状态。

## 13. 结论

当前 `todo-h5` 已经不是原型页，而是一套具备实际业务闭环的移动端待办界面。

它的特点是：

- 业务口径继承现有系统
- 交互方式面向手机端重构
- 已覆盖待办执行闭环与个人提醒闭环
- UI 已完成基础收敛并进入规范化阶段

如果后续继续开发，建议以本文档为基础，优先推进：

1. 统一卡片结构
2. 统一页面头部模式
3. 完成提醒编辑闭环
4. 逐步抽象通用组件

这样 H5 端会更稳定，也更适合长期维护。
