# 组织账号管理开发者文档

## 1. 文档目标

本文档面向后续开发、维护、排障、重构以及 AI 辅助阅读场景，系统性说明“组织账号管理”模块的前后端实现、接口、数据库结构、权限模型、数据加载方式、维护建议与常见风险点。

适用对象：
- 新接手项目的前端开发
- 新接手项目的后端开发
- 测试 / 运维 / 产品技术支持
- 需要让 AI 理解该模块并生成维护建议的使用者

---

## 2. 功能概述

“组织账号管理”模块用于对系统中的管理账号进行组织维度与账号维度的统一管理，核心目标包括：

1. 按组织查看已有管理账号
2. 为指定账号授予管理身份
3. 按账号查看其所有管理身份
4. 启用 / 停用单条身份
5. 启用 / 禁用整个账号
6. 强制删除账号或身份（仅开发管理员）
7. 控制账号授权范围，避免同一账号在同一组织链内重复拥有冲突的管理权限

该模块不负责：
- 主播注册审核
- 主播资料录入
- 登录鉴权本身
- 组织树维护本身

但它依赖这些模块的输出：
- 组织树
- 用户账号
- 身份体系
- 主播档案
- 权限表

## 4. 模块功能拆解

### 4.1 页面入口

组织账号管理页面在“组织管理”分组下，路径为：

- 前端路由：`/accounts`
- 页面组件：`AccountsPage`

### 4.2 页面分为两种视图

#### 4.2.1 按组织管理（by-org）

用途：
- 从组织树选择一个组织
- 查看该组织直属管理账号
- 打开账号抽屉查看详情
- 为该组织新增管理员
- 对账号做禁用/启用/删除
- 对身份做停用/启用

当前业务规则已经明确为“直属管理员模式”：
- 选择 `河北基地` 时，只看 `orgId = 河北基地` 的管理身份
- 不展示 `河北基地` 下团队、厅的管理身份
- 也就是说，左侧组织树决定的是“当前组织节点”，不是“整个子树范围”

#### 4.2.2 按账号全局管理（by-account）

用途：
- 输入手机号或昵称搜索账号
- 查看该账号名下所有管理身份
- 逐条启用/停用身份

### 4.3 角色授权规则

当前管理角色与组织层级绑定关系如下：

| 组织层级 | 可授权角色 |
|---|---|
| HQ | HQ_ADMIN |
| BASE | BASE_ADMIN |
| TEAM | TEAM_ADMIN |
| HALL | HALL_MANAGER |

即：
- 总部组织只能授总部管理
- 基地只能授基地运营
- 团队只能授团队运营
- 厅只能授厅管理

这套约束由后端强制校验，前端仅做显示层过滤。

### 4.4 管理范围冲突规则

同一个账号在同一组织链上不能同时存在多个生效中的管理身份。

例如：
- 已有 `BASE_ADMIN@基地A`
- 不能再授予 `TEAM_ADMIN@基地A/团队B`
- 也不能再授予 `HQ_ADMIN@总部`（若总部路径覆盖该基地路径）

系统会将这类情况识别为“组织链范围冲突”。

## 5. 前端实现说明

### 5.1 主页面状态结构

主页面 `AccountsPage.tsx` 维护以下核心状态：

1. 视图模式：`by-org / by-account`
2. 组织树数据：`orgs`
3. 当前组织账号列表：`accounts`
4. 当前选中组织：`selectedOrgId`
5. 当前选中账号：`activeAccountId`
6. 全局搜索结果：`globalAccounts`
7. 全局选中账号：`selectedGlobalAccountId`
8. 授权弹窗开关：`showGrantModal`
9. 二次确认弹窗：`confirmation`
10. 成功/失败反馈：`message / error`

### 5.2 按组织视图的页面流程

#### 5.2.1 初始化

页面加载后：
1. 调用 `/orgs/children`
2. 只加载当前身份下首屏所需的组织层级节点
3. 根据当前身份决定默认展开节点，但**不再默认选中任何组织**
4. `collapsedIds` 仅在首屏组织树完成初始化时按默认展开规则计算一次
5. 右侧组织账号列表、详情抽屉和“新增管理员”操作都要求用户先手动选择左侧组织后再继续

#### 5.2.2 组织切换

切换左侧组织树节点时：
1. 更新 `selectedOrgId`
2. 主动展开当前选中节点的祖先链路，保证当前路径始终可见
3. 若当前节点尚未加载下级，则调用 `/orgs/children?parentId=xxx` 懒加载子节点
4. 调用 `/accounts?orgId=xxx&page=1&pageSize=20`
5. 后端只返回该组织直属管理身份对应的账号列表
6. 前端只保留 `identity.orgId === 当前选中 orgId` 且层级角色合法的身份

#### 5.2.3 账号详情抽屉

点击表格行会打开右侧抽屉，展示：
- 账号基础信息
- 绑定主播资料（如果有）
- 身份列表
- 账号控制按钮

当前抽屉详情已经改为独立接口加载：
1. 列表行点击后先用当前行基础数据打开抽屉
2. 再调用 `/accounts/:id/detail`
3. 抽屉内展示独立详情结果，并在加载时显示 loading 提示

#### 5.2.4 授权弹窗

点击“新增管理员”后：
1. 打开 `GrantModal`
2. 选择组织
3. 选择角色
4. 搜索账号
5. 确认授权
6. 成功后刷新当前视图与全局搜索视图

### 5.3 按账号视图的页面流程

1. 输入手机号/昵称
2. 调用 `/accounts/search?keyword=...`
3. 返回账号列表
4. 用户选择某个账号
5. 展示该账号的全部管理身份
6. 可逐条停用/启用身份

### 5.4 交互反馈机制

页面统一使用：
- 顶部绿色成功提示
- 顶部红色错误提示
- 危险操作二次确认弹窗

危险操作包括：
- 停用身份
- 启用身份
- 禁用账号
- 启用账号
- 删除账号

## 6. 后端处理逻辑

### 6.1 模块总入口

后端组织账号管理统一定义在：
- `backend/src/modules/account/routes.ts`

路由统一挂载以下中间件：
1. `authRequired`
2. `identityRequired`
3. 接口级别的 `permissionRequired`

这意味着每个接口都要求：
- 已登录
- 当前账号已经选择一个身份
- 当前身份具备对应权限

### 6.2 核心后端逻辑点

#### 6.2.1 组织范围搜索

系统会根据当前身份的 `scopePath` 决定可搜索和可查看的数据范围。

关键逻辑：
- `DEV_ADMIN` 可查看全量
- 其他身份默认只能查看 `scopePath` 以内的组织/账号/身份数据

#### 6.2.2 主播档案补充信息

账号列表接口会额外通过 `resolvePrimaryAnchorProfiles` 为账号补上主播档案，便于前端展示：
- 主播昵称
- 抖音号
- 抖音 UID

#### 6.2.3 管理身份冲突检查

授权身份、重新启用身份时都会调用：
- `findManagementScopeConflicts`

用途：
- 阻止同一账号在同一组织链上拥有重复/重叠的管理范围

#### 6.2.4 身份授予逻辑

后端通过 `grantIdentity` 执行授权：
- 若完全相同的身份已存在，则恢复成 `active`
- 若不存在，则新建身份记录

#### 6.2.5 账号禁用逻辑

当禁用账号时：
- `users.status` 会变为 `disabled`
- 所有当前生效身份会被同步停用
- `expiredAt` 写入当前时间

#### 6.2.6 主播身份联动逻辑

若身份角色为 `ANCHOR`：
- 启用时：绑定 `AnchorProfile.boundUserId`
- 停用时：解绑 `AnchorProfile.boundUserId`

虽然“组织账号管理”页面本身不直接授主播身份，但后端模块本身同时兼容主播身份处理。

## 7. 页面权限与角色矩阵

### 7.1 页面访问权限

前端路由要求：
- 角色必须属于：`DEV_ADMIN / HQ_ADMIN / BASE_ADMIN / TEAM_ADMIN`
- 且必须具备：`account:view`

也就是说：
- `HALL_MANAGER` 默认不能进入“组织账号管理”页面
- `ANCHOR` 不能进入

### 7.2 接口权限矩阵

| 接口 | 权限 | 说明 |
|---|---|---|
| GET `/accounts` | `account:view` | 按组织加载账号列表 |
| GET `/accounts/search` | `account:view` | 全局搜索账号 |
| POST `/accounts/:id/identities` | `identity:grant` | 授权身份 |
| PATCH `/accounts/:id/status` | `account:create` | 启用/禁用账号 |
| POST `/accounts/:id/reset-password` | `account:create` | 重置密码 |
| PATCH `/accounts/identities/:id/status` | `identity:grant` | 启用/停用身份 |
| DELETE `/accounts/:id` | DEV_ADMIN 特判 | 强制删除账号 |
| DELETE `/accounts/identities/:id` | DEV_ADMIN 特判 | 强制删除身份 |

### 7.3 角色默认权限

基于 `sync-role-permissions.ts`，该模块当前角色权限如下：

| 角色 | account:view | account:create | identity:grant |
|---|---:|---:|---:|
| DEV_ADMIN | 是 | 是 | 是 |
| HQ_ADMIN | 是 | 是 | 是 |
| BASE_ADMIN | 是 | 是 | 是 |
| TEAM_ADMIN | 是 | 是 | 是 |
| HALL_MANAGER | 否 | 否 | 否 |
| ANCHOR | 否 | 否 | 否 |

### 7.4 特殊权限说明

即便接口没有显式 `permissionRequired`，仍可能存在角色硬编码限制，例如：
- 强制删除账号：仅 `DEV_ADMIN`
- 强制删除身份：仅 `DEV_ADMIN`

因此后续新增接口时，必须同时检查：
1. 权限码是否正确
2. 是否还存在角色级别特判

## 8. API 接口释义

以下接口均以 `/api` 为网关前缀，文档中省略 `/api`。

### 8.1 GET `/orgs/tree`

#### 用途
加载当前身份可见范围内的完整组织树平铺列表。

#### 请求参数
无。

#### 权限
- `org:view`

#### 返回
- 当前身份可见范围内的组织树平铺列表

#### 说明
该接口目前仍被“组织设立变更”等页面使用；组织账号管理页已不再依赖该接口首屏加载全树。

---

### 8.2 GET `/orgs/children`

#### 用途
按父节点懒加载组织树下一层节点，供“组织账号管理”左侧组织树使用。

#### 查询参数
| 参数 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| parentId | string | 否 | 父组织 ID；为空时表示加载当前身份首屏层级 |

#### 权限
- `org:view`

#### 返回
- 当前身份可见范围内、指定父节点下一级的平铺节点列表
- 每个节点附带 `hasChildren` / `childCount`

#### 当前首屏返回规则
- `DEV_ADMIN / HQ_ADMIN`：首屏返回 `HQ / BASE`
- `BASE_ADMIN`：首屏返回 `TEAM`
- `TEAM_ADMIN`：首屏返回 `HALL`
- `HALL_MANAGER`：首屏返回当前厅

---

### 8.3 GET `/accounts`

#### 用途
按组织查询管理账号列表。

#### 查询参数
| 参数 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| orgId | string | 否 | 组织 ID |
| keyword | string | 否 | 账号昵称/手机号/抖音号/UID 模糊搜索 |
| roleCode | string | 否 | 按角色筛选身份 |
| page | number | 否 | 页码，默认 1 |
| pageSize | number | 否 | 每页条数，默认 20，最大 100 |

#### 权限
- `account:view`

#### 后端逻辑
1. 根据当前身份 scopePath 做范围过滤
2. 根据 `orgId` 只查询拥有该组织直属身份的账号
3. 按 keyword 做模糊搜索
4. 查询匹配用户并关联身份、组织、主播档案
5. 补全主主播档案信息
6. 返回分页结果 `{ items, total, page, pageSize }`

#### 备注
当前模块采用“直属管理员模式”，不是组织子树范围模式。
前端收到后，还会再做一次显示层过滤，只保留：
- 管理身份
- `identity.orgId === 当前选中 orgId`
- 与该组织层级匹配的角色

---

### 8.4 GET `/accounts/search`

#### 用途
全局搜索账号，用于授权弹窗或按账号管理视图。

#### 查询参数
| 参数 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| keyword | string | 否 | 手机号/昵称/抖音号/UID |
| ids | string/string[] | 否 | 指定账号 ID 列表 |
| scopeOrgId | string | 否 | 将搜索范围限定到某个组织路径 |
| page | number | 否 | 页码，默认 1 |
| pageSize | number | 否 | 每页条数，默认 20，最大 100 |

#### 权限
- `account:view`

#### 特殊逻辑
- 若传 `scopeOrgId`，会先解析为组织路径
- 非开发管理员只能在自己 scopePath 内搜索
- 无关键字且无 ids 时直接返回空分页结果
- 非 `ids` 查询时支持分页返回 `{ items, total, page, pageSize }`

---

### 8.5 GET `/accounts/:id/detail`

#### 用途
按需加载单个账号详情，用于右侧账号抽屉。

#### 路径参数
| 参数 | 说明 |
|---|---|
| id | 用户 ID |

#### 权限
- `account:view`

#### 后端逻辑
1. 查询单个用户及全部身份
2. 按当前身份 scopePath 校验是否可见
3. 补全主主播档案
4. 返回账号详情对象

#### 说明
该接口用于降低列表接口负载，抽屉展示不再完全依赖列表行数据。

---

### 8.6 POST `/accounts/:id/identities`

#### 用途
为账号授予身份。

#### 路径参数
| 参数 | 说明 |
|---|---|
| id | 用户 ID |

#### 请求体
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| roleCode | string | 是 | 角色代码 |
| orgId | string | 否 | 管理身份对应组织 ID |
| anchorProfileId | string | 否 | 主播身份对应档案 ID |

#### 权限
- `identity:grant`

#### 后端关键校验
1. 用户必须存在
2. 管理身份必须选组织
3. 角色和组织层级必须匹配
4. 同一组织链上不能已有其他生效管理身份
5. 主播身份必须指定主播档案

#### 冲突错误
当发现组织链冲突时，返回：
- `ORG_SCOPE_CONFLICT`
- 并附带 `details`

前端会把 `details` 转成人类可读的冲突信息。

---

### 8.5 PATCH `/accounts/:id/status`

#### 用途
启用或禁用整个账号。

#### 请求体
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| status | string | 是 | `active` 或 `disabled` |

#### 权限
- `account:create`

#### 后端逻辑
- 更新 `users.status`
- 若禁用账号，则同步停用该账号全部 active 身份

---

### 8.6 POST `/accounts/:id/reset-password`

#### 用途
重置账号密码。

#### 权限
- `account:create`

#### 后端逻辑
- 将密码重置为 `123456`
- `mustChangePassword = false`
- 同时将账号状态改为 `active`

#### 注意
该能力当前不在“组织账号管理”前端页面显式暴露，但后端接口已存在。

---

### 8.7 PATCH `/accounts/identities/:id/status`

#### 用途
启用或停用单条身份。

#### 请求体
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| status | string | 是 | `active` 或 `disabled` |

#### 权限
- `identity:grant`

#### 特殊逻辑
- 若重新启用的是管理身份，会再次校验组织链冲突
- 若是主播身份，会同步绑定/解绑主播档案

---

### 8.8 DELETE `/accounts/:id`

#### 用途
强制删除账号。

#### 权限
- 仅 `DEV_ADMIN`

#### 后端逻辑
1. 查询用户和全部身份
2. 若身份中含主播身份，则先解绑主播档案
3. 删除全部身份
4. 删除账号

## 10. 页面数据加载与“懒加载分级传输”逻辑

> 这里的“懒加载分级传输”不是前端 React lazy/suspense 代码分包，而是页面按使用阶段逐步拉取数据、避免一次性拉全量明细的分层传输策略。

### 10.1 总体策略

本模块采用“分阶段加载”而不是“一次返回所有细节”：

1. 先懒加载组织树首屏节点
2. 用户展开节点时，再按父节点拉取下一层组织
3. 用户选择组织后，再加载该组织直属管理账号列表分页数据
4. 用户点击某账号后，再单独请求账号详情接口
5. 用户进行全局搜索时，才触发全局账号检索
6. 授权弹窗打开后，只有用户主动搜索时才加载候选账号

### 10.2 加载分层说明

#### 第一层：组织树首屏

页面初始化先加载：
- `/orgs/children`

特点：
- 是页面主导航骨架
- 只返回当前身份首屏层级所需节点
- 返回平铺结构，由前端构造成树

#### 第二层：组织树子节点懒加载

只有用户展开某个节点时才调用：
- `/orgs/children?parentId=...`

特点：
- 仅拉取当前节点下一层
- 节点携带 `hasChildren / childCount`
- 避免一次性下发全树
- 懒加载只负责补充节点数据，不再覆写用户已经形成的折叠状态

#### 第三层：组织下账号列表

当 `selectedOrgId` 或分页变化时调用：
- `/accounts?orgId=...&page=...&pageSize=20`

特点：
- 按组织直属身份缩小结果集
- 避免一次性拉整个系统账号
- 是典型“主列表按组织懒加载 + 分页加载”

#### 第四层：账号详情展示

点击账号后打开抽屉，并额外调用：
- `/accounts/:id/detail`

原因：
- 列表接口只承担列表所需字段
- 抽屉详情独立加载，避免列表越做越重
- 同时支持抽屉 loading 态

#### 第五层：全局账号搜索

当用户切换到“按账号全局管理”并主动搜索时，才调用：
- `/accounts/search?keyword=...&page=...&pageSize=20`

特点：
- 按需触发
- 支持分页
- 属于典型的“搜索型懒加载”

#### 第五层：授权弹窗搜索

打开授权弹窗本身不加载账号；只有输入关键字并点击搜索时才调用：
- `/accounts/search?keyword=...`

这可避免授权弹窗打开时自动扫描全库。

补充说明：
- `/accounts/search` 当前返回分页对象 `{ items, total, page, pageSize }`
- 前端统一通过 `accountApi.searchAccounts()` / `getAccountsByIds()` 解包 `items`
- 授权弹窗若直接把返回值当数组使用，会导致运行时白屏

### 10.3 后端在数据传输上的处理逻辑

#### 10.3.1 范围收缩

后端优先通过 `scopePath` 做数据范围收缩，减少返回结果：
- 组织树只返回可见组织
- 账号搜索只返回可见范围账号

#### 10.3.2 条件搜索

查询按条件拼接：
- orgId
- roleCode
- keyword
- ids
- scopeOrgId

这意味着接口本身是“条件化按需加载”的，不是粗暴全量读取。

#### 10.3.3 分页式返回

当前组织账号管理已经使用分页返回：
- `/accounts`：支持 `page/pageSize`
- `/accounts/search`：支持 `page/pageSize`

默认前端使用：
- `pageSize = 20`

### 10.4 当前加载策略总结

当前模块已经具备以下分层能力：
- 组织树按父节点懒加载
- 组织账号列表分页加载
- 全局账号搜索分页加载
- 抽屉详情独立按需加载
- 搜索输入前端防抖

### 10.5 后续若继续扩展

建议优先方向：

1. 组织树节点级 loading 指示
2. 抽屉详情缓存
3. 全局账号视图也统一接入抽屉详情交互
4. 搜索请求去重与并发控制

## 11. 维护指南

### 11.1 新开发者接手建议

第一步：读代码路径
- 先看 `AccountsPage.tsx`

3. `AccountsPage.tsx` 表格列
4. `AccountDrawer.tsx` 详情展示

#### 场景 C：放宽或收紧授权冲突规则

#### 场景 D：让 HALL_MANAGER 也能进入组织账号管理

### 11.3 排障建议

#### 问题 1：页面空白或跳转回去
先检查：
- 是否已登录
- 是否已选择身份
- 当前身份是否有 `account:view`
- 当前角色是否允许进入 `/accounts`

#### 问题 2：授权失败，提示组织链冲突
检查：
- 该账号是否已有 active 管理身份
- 这些身份的 `scopePath` 是否覆盖目标组织
- 是否本应先停用旧身份再授权

#### 问题 3：搜索不到账号
检查：
- 是否被 `scopePath` 限制
- 是否账号没有任何可见身份
- 是否搜索词过短或输入错误
- 是否在授权弹窗中忘记点击搜索按钮

#### 问题 4：禁用账号后仍显示身份
检查：
- 前端是否刷新视图
- 后端是否成功更新 `user_identities.status`
- 是否查看的是缓存结果

当前模块适合后续补充以下能力：

## 12. 风险点与改进建议

### 12.1 当前已知设计特点

1. 列表详情不单独请求，依赖当前列表已返回的数据
2. 授权冲突规则较强，避免了同组织链多重身份叠加
3. 强制删除只允许开发管理员
4. 权限和角色放行是“双层控制”：前端路由 + 后端权限

### 12.2 当前可优化点

1. **重置密码前端未暴露**
   - 接口有，页面没有操作入口

2. **删除/禁用/授权未写审计日志**
   - 需要时应补 `audit_logs`

3. **强制删除接口存在角色硬编码特判**
   - 维护时必须同时关注权限码与角色判断

4. **组织树 loading 仍是区域级提示**
   - 还不是节点级 loading

5. **全局账号管理视图尚未统一使用右侧抽屉详情交互**
   - 当前主要仍是列表 + 下方详情表格

### 12.3 推荐改造优先级

高优先级：
1. 关键操作审计日志化
2. 删除/重置密码前端显式管理与确认增强
3. 组织树节点级 loading 与搜索请求去重

中优先级：
1. 全局账号视图接入统一抽屉详情
2. 搜索结果增加更多筛选项（角色、组织层级、状态）
3. 增加账号创建入口（若未来需要）

低优先级：
1. 表格列可配置
2. 批量授权/批量停用
3. 操作历史时间线

## 14. 维护结论

“组织账号管理”模块本质上是一个：

- 以组织树为边界
- 以 User / UserIdentity 为核心实体
- 以 scopePath 为数据域控制手段
- 以组织链冲突校验为主要业务约束
- 以前端双视图（按组织 / 按账号）承载操作入口

的后台管理模块。

维护该模块时，必须始终同时考虑以下四件事：

1. **角色和组织层级是否匹配**
2. **scopePath 范围是否正确**
3. **身份冲突规则是否被破坏**
4. **账号状态与身份状态是否联动一致**

如果未来要做大规模扩展，建议优先做：
- 分页
- 审计日志
- 更清晰的权限矩阵
- 更细粒度的详情接口

根因总结：
- 页面原先把“默认展开规则”同时用于“首次初始化”和“后续交互过程”
- 但默认展开只适合首屏初始化，不适合在用户已开始操作后继续重放

当前修复后的规则：
1. `collapsedIds` 只在组织树首次加载时初始化一次
2. 后续懒加载子节点时，不再重置或补写用户当前折叠状态
3. 点击某个组织节点时，会主动展开该节点祖先链路
4. 用户手动展开 / 折叠的结果，优先级高于默认展开策略

本次修复带来的产品收益：
- 点击“xx厅”后不会再误以为组织树异常折叠
- 组织定位更稳定，账号查看路径更清晰
- 左侧树交互符合“点击=选中、箭头=折叠/展开”的常规认知

后续维护注意事项：
- 若继续调整账号管理页的组织树行为，不要在每次 `orgs` 更新时重新按默认规则写入 `collapsedIds`
- 若新增更多懒加载层级，必须保证当前选中节点祖先路径可见
- 若需要调整默认展开体验，只改“初始化逻辑”，不要改“交互期状态保持逻辑”
