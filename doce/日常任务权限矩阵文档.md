# 日常任务权限矩阵文档

## 1. 文档目标

本文档用于整理当前项目中“日常任务”相关页面、接口、操作按钮和业务动作的权限矩阵，方便：

- 前端开发判断哪些入口该显示、哪些该隐藏
- 后端开发判断哪些接口需要额外权限保护
- 测试人员按身份验证可见性和操作范围
- 产品、实施、运维统一理解各身份边界
- AI 工具在后续维护时避免误放开或误收紧权限

本文档只讨论日常任务，不覆盖临时任务。

---

## 2. 涉及身份

当前日常任务链路中涉及的主要身份：

- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`
- `ANCHOR`

---

## 3. 总体权限结论

## 3.1 发布权限
允许进入和维护“任务发布 -> 日常任务”的身份：

- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`

不允许：
- `TEAM_ADMIN`
- `HALL_MANAGER`
- `ANCHOR`

## 3.2 看板权限
允许进入“任务看板 -> 日常任务看板”的身份：

- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

不允许：
- `ANCHOR`

## 3.3 我的待办权限
允许进入“我的待办”并执行自己的日常任务的身份：

- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`
- `ANCHOR`

说明：
- “我的待办”是个人执行入口，不是管理入口
- 主播身份也需要看到自己的日常任务

## 6. 页面级操作权限矩阵

## 6.1 日常任务发布页
文件：`frontend/src/pages/tasks/manage/DailyIssuePage.tsx`
文件：`frontend/src/pages/tasks/manage/components/DailyTaskWizard.tsx`

### 可执行动作
- 查看基地下草稿 / 待生效 / 生效中 / 已结束任务
- 新建日常任务草稿
- 编辑草稿
- 删除草稿
- 设置排除组织 / 主播
- 预览发布
- 正式发布
- 发送飞书通知

### 允许身份
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`

### 备注
虽然页面内部还会按权限开关一些按钮，但从业务规则上，这整个页面就是发布管理页，只对总公司 / 基地管理开放。

## 6.2 日常任务看板页
文件：`frontend/src/pages/tasks/dashboard/DailyTaskDashboardPage.tsx`

### 可执行动作
- 查询日常任务看板
- 切换日期
- 切换基地（仅高权限）
- 查看团队 / 厅 / 主播明细
- 查看主播子任务详情
- 提交豁免审核
- 取消 / 回撤豁免
- 发送飞书通知

### 允许身份
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

### 特殊说明
#### `DEV_ADMIN / HQ_ADMIN`
- 可以选择基地查看

#### `BASE_ADMIN`
- 一般锁定在所属基地

#### `TEAM_ADMIN`
- 可查看自己范围内的团队 / 厅 / 主播数据

#### `HALL_MANAGER`
- 可查看自己厅范围内数据
- 不能越权看其他范围

#### `ANCHOR`
- 前端不显示入口
- 路由也不允许进入

## 6.3 我的待办页
文件：`frontend/src/pages/tasks/dashboard/TaskDashboardPage.tsx`
文件：`frontend/src/pages/tasks/my-todos/TaskRecordCard.tsx`

### 可执行动作
- 查看今天的日常任务
- 查看昨天未完成补录任务
- 填写子任务
- 上传附件
- 提交任务
- 提交补录
- 申请豁免

### 允许身份
- 所有已登录身份，但仅能操作自己当前身份可见的记录

主要文件：`backend/src/modules/task/record/record.controller.ts`
文件：`backend/src/modules/task/record/record.service.ts`

### 接口
- `GET /api/tasks/my-records`
- `GET /api/tasks/my-records/:id`
- `POST /api/tasks/item-records`
- `POST /api/tasks/my-records/:id/submit`
- `POST /api/tasks/exemptions`
- `GET /api/tasks/exemptions`
- `POST /api/tasks/exemptions/:id/review`

### 权限口径
#### 个人执行接口
- 面向所有已登录身份
- 但 record 必须是“当前用户或当前身份可见”的记录

#### 豁免审核接口
- 仅管理身份可审核
- 审核范围还受 `scopePath` 限制

审核角色判定见：
- `canReviewDailyExemption`

允许审核身份：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

禁止审核身份：
- `ANCHOR`

## 9. 当前已落地的前端屏蔽规则

本轮已确认并落地：

1. `任务发布 -> 日常任务`
   - 非 `DEV_ADMIN / HQ_ADMIN / BASE_ADMIN` 前端隐藏
   - 路由也已保护

2. `任务看板 -> 日常任务看板`
   - `ANCHOR` 前端隐藏
   - 路由也已保护

3. 如果某一级菜单下没有任何可见子项：
   - 整个一级菜单分组应直接隐藏
   - 避免出现“子项没了但一级还在”的问题

---

## 10. 维护注意事项

## 10.1 前端隐藏不等于后端安全
任何入口隐藏都只能算用户体验控制，不能替代后端权限校验。

## 10.2 角色与权限码要同时看
当前项目很多地方是“双条件”：
- 角色命中
- 权限码命中

维护时不要只看其中之一。

## 10.3 日常任务的发布权限不要被看板权限带偏
当前业务明确：
- 看板可开放到团队 / 厅管理
- 发布只开放到总公司 / 基地管理

两者不是同一套权限模型，不能直接复制。

## 10.4 主播不能看日常任务看板，但必须能看我的待办
这条边界很关键：
- 主播没有管理视角
- 但一定要保留个人执行入口
