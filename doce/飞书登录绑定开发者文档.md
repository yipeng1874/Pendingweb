# 飞书登录绑定开发者文档

## 1. 文档目标

本文档用于系统性说明当前项目中的"飞书企业配置 + 飞书登录 + 飞书绑定/解绑 + 回调处理"整条链路，方便：

- 新接手项目的前端开发快速定位飞书相关页面与状态流转
- 新接手项目的后端开发快速理解鉴权接口、配置模型与绑定约束
- 测试 / 运维 / 实施人员理解飞书登录与绑定失败时的排查路径
- AI 工具在阅读代码时快速建立完整上下文

本文覆盖范围包括：
- 飞书企业配置数据模型
- 登录页飞书登录流程（含一键重登）
- 飞书免登中转页（FeishuEntryPage）自动登录流程
- 个人设置页飞书绑定 / 解绑流程
- 飞书授权回调页处理逻辑
- 后端飞书鉴权接口与组织配置接口
- 当前权限、菜单可见性、环境变量与常见错误

## 2. 功能概述

当前系统中的飞书能力分为两层：

### 2.1 组织侧：飞书企业配置

由组织管理人员维护"某个基地 / 团队"对应的飞书企业开放平台配置，核心字段包括：
- 飞书企业名称 `name`
- App ID `appId`
- App Secret `appSecret`
- 所属基地 `baseOrgId`
- 所属团队 `teamOrgId`
- 配置状态 `status`

### 2.2 用户侧：飞书登录与绑定

用户侧当前提供四类能力：

1. **飞书免登（PC 客户端工作台入口）**
   - 飞书开放平台配置"网页应用 → 桌面端主页"指向 `/feishu-entry?appId=cli_xxx`
   - 用户在飞书 PC 客户端工作台点击应用图标，直接跳转该 URL
   - `FeishuEntryPage` 自动调用 `tt.requestAccess` 获取 code，完成无感登录
   - 登录成功后写入 `localStorage.feishu_entry_app_id` 并跳转 `/tasks/cockpit`

2. **一键重登（退出后兜底）**
   - 用户在系统内手动退出后跳转到 `/login`
   - 若当前在飞书 PC 客户端内且 `localStorage.feishu_entry_app_id` 有值
   - 登录页飞书 Tab 会显示"一键飞书登录"按钮（手动选择区默认折叠）
   - 点击即重定向回 `/feishu-entry?appId=xxx`，重走免登流程，无需重新选择组织

3. **Web 飞书登录（手动选择组织）**
   - 在登录页手动选择"基地 -> 团队 -> 飞书企业"
   - 点击"使用飞书登录"按钮后跳转飞书授权页（浏览器 OAuth 重定向流）
   - 回调页拿到授权码后完成系统登录
   - 一键重登失败时可展开此区域作为兜底

4. **飞书绑定 / 解绑**
   - 用户先使用账号密码登录系统
   - 在"个人账号管理"页选择归属后手动发起绑定
   - 回调页完成当前系统账号与飞书账号的绑定
   - 已绑定用户可在设置页直接解绑

## 3. 当前产品规则

### 3.1 飞书免登是自动发起，Web 飞书登录不是

**飞书免登（FeishuEntryPage）**：
- 用户从飞书 PC 客户端工作台进入，页面自动调用 `tt.requestAccess` 获取 code
- 无需任何用户操作，全流程自动完成
- 非飞书环境则自动跳转 `/login`

**Web 飞书登录（LoginPage 手动选择区）**：
- 须先选择基地 → 团队 → 飞书企业
- 最后由用户手动点击"使用飞书登录"按钮
- 页面不会自动跳飞书、不会自动猜配置

**一键重登（LoginPage 一键按钮）**：
- 仅在飞书 PC 客户端内、且 `localStorage.feishu_entry_app_id` 有值时显示
- 点击后直接跳 `/feishu-entry?appId=xxx`，重走自动免登流程
- 本质是退出后重走免登的兜底快捷方式，无需重新选择组织

### 3.2 飞书绑定依赖已登录系统账号

飞书绑定不是匿名操作，必须先登录系统账号。

绑定场景下，前端会：
- 从当前登录态拿到 JWT token
- 临时存入 `sessionStorage.feishu_bind_token`
- 跳转到 `/api/auth/feishu/login?action=bind&token=...&configId=...`
- 回调页再恢复 token 并调用绑定接口

### 3.3 飞书配置与组织主数据强绑定

一条飞书企业配置必须绑定到：
- 一个 `BASE`
- 一个隶属于该 `BASE` 的 `TEAM`

因此飞书登录和绑定并不是"全局单一企业配置"模型，而是：
- 配置按组织归属分桶
- 用户在发起登录 / 绑定前必须显式选择当前业务归属

### 3.4 团队管理不开放飞书企业配置页

当前前端与路由规则已经统一为：
- `DEV_ADMIN / HQ_ADMIN / BASE_ADMIN` 可以看到并访问 `/organization/feishu-configs`
- `TEAM_ADMIN` 不显示该菜单，也无法直接访问该路由

## 5. 页面与路由入口

### 5.1 飞书免登中转页

- 页面：`FeishuEntryPage`
- 路由：`/feishu-entry`
- 入口：飞书 PC 客户端工作台应用主页（飞书后台配置 URL 为 `http(s)://域名/feishu-entry?appId=cli_xxx`）
- 作用：自动调用 `tt.requestAccess` 获取 code，完成无感登录后跳转 `/tasks/cockpit`
- 特殊处理：
  - 非飞书环境直接跳 `/login`
  - 登录成功后写入 `localStorage.feishu_entry_app_id` 供退出后一键重登使用
  - `FEISHU_UNBOUND / USER_NOT_FOUND` 时跳 `/login?feishuUnbound=1`

### 5.2 登录页

- 页面：`LoginPage`
- 路由：`/login`
- 作用：账号密码登录、Web 飞书手动登录、飞书一键重登入口

### 5.3 回调页

- 页面：`FeishuCallbackPage`
- 路由：`/pc/auth/callback`
- 作用：接收飞书 OAuth 授权结果（仅 Web 手动登录 / 绑定流程使用），完成登录或绑定
- 注意：飞书免登和一键重登走 JSAPI 流程，**不经过此回调页**

### 5.4 个人设置页

- 页面：`SettingsPage`
- 路由：`/settings`
- 作用：已登录用户修改密码、绑定 / 解绑飞书账号

### 5.5 飞书企业配置页

- 页面：`FeishuConfigPage`
- 路由：`/organization/feishu-configs`
- 作用：维护基地 / 团队维度的飞书企业配置

### 5.6 路由保护规则

当前 `App.tsx` 中的关键路由保护如下：

- `/feishu-entry`
  - 公共路由，无需登录
  - 页面内部检测飞书环境，非飞书环境自动跳 `/login`

- `/organization/feishu-configs`
  - 角色：`DEV_ADMIN / HQ_ADMIN / BASE_ADMIN`
  - 权限：`org:view`

- `/settings`
  - 只要求已登录进入主应用布局
  - 绑定 / 解绑接口本身再依赖 `authRequired`

- `/pc/auth/callback`
  - 公共路由，不要求先登录
  - 但绑定场景回调内部会校验 token 是否存在

## 6. 数据模型说明

### 6.1 `User` 中的飞书相关字段

`schema.prisma` 中 `User` 当前包含以下飞书字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `feishuOpenId` | `String?` | 飞书账号 open_id，唯一 |
| `feishuUnionId` | `String?` | 飞书账号 union_id，唯一 |
| `feishuConfigId` | `String?` | 当前绑定的飞书企业配置 ID |
| `feishuName` | `String?` | 飞书返回的昵称 |
| `feishuAvatarUrl` | `String?` | 飞书头像 |
| `feishuBoundAt` | `DateTime?` | 绑定时间 |

说明：
- `open_id` 和 `union_id` 都被设计为唯一值
- 登录与绑定匹配时，会按 `feishuConfigId + open_id/union_id` 查找用户
- 登录成功后会补刷 `feishuName / feishuAvatarUrl`

### 6.2 `FeishuEnterpriseConfig` 模型

`FeishuEnterpriseConfig` 当前字段包括：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `String` | 主键 |
| `name` | `String` | 飞书企业名称 |
| `appId` | `String` | 飞书开放平台 App ID |
| `appSecret` | `String` | 飞书开放平台 App Secret |
| `baseOrgId` | `String` | 归属基地 |
| `teamOrgId` | `String` | 归属团队 |
| `status` | `OrgStatus` | `active / paused` |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

当前约束：
- `appId` 全局唯一
- `(teamOrgId, name)` 联合唯一
- `teamOrgId` 必须隶属于 `baseOrgId`

### 6.3 `OrgUnit` 与飞书配置关系

`OrgUnit` 中存在两条反向关系：
- `feishuBaseConfigs`
- `feishuTeamConfigs`

这意味着：
- 飞书企业配置并不是独立域模型
- 它本质上属于组织域的延伸配置

## 7. 前端实现说明

### 7.1 飞书免登中转页 `FeishuEntryPage.tsx`

飞书 PC 客户端工作台的直接入口，完整自动登录流程：

1. 从 URL query 读取 `appId`
2. 调 `GET /api/auth/feishu/app-ids` 找到对应 `configId`
3. 调 `getFeishuAuthCode(appId)` 获取临时 code（`tt.requestAccess`）
4. `POST /api/auth/feishu/app-login` 换取 token
5. 登录成功 → 写入 `localStorage.feishu_entry_app_id = appId` → 跳转 `/tasks/cockpit`
6. 失败 → 显示错误信息，提供"手动登录"按钮跳 `/login`
7. `FEISHU_UNBOUND / USER_NOT_FOUND` → 跳 `/login?feishuUnbound=1`

核心状态：`status: "loading" | "error"`，使用 `useRef(ran)` 防止 StrictMode 双调用。

### 7.2 登录页 `LoginPage.tsx`

登录页当前有两个 tab：
- `account`：账号密码登录
- `feishu`：飞书登录（含一键重登 + Web 手动登录）

飞书 Tab 核心状态：
- `savedAppId`：从 `localStorage.feishu_entry_app_id` 读取
- `canOneClick`：`inFeishu && !!savedAppId`，同时满足才激活一键登录
- `orgSelectorOpen`：手动选择区折叠状态，`canOneClick` 时默认折叠
- `baseOptions / teamOptions / configOptions`
- `selectedBaseId / selectedTeamId / selectedConfigId`
- `feishuLoading`

**一键重登流程**（`canOneClick = true` 时）：
1. 页面进入飞书 Tab，自动切到飞书 Tab
2. 显示"一键飞书登录"按钮，手动选择区默认折叠
3. 用户点击 → `window.location.href = /feishu-entry?appId=xxx`
4. 重走 `FeishuEntryPage` 免登流程

**Web 手动登录流程**（`canOneClick = false` 或展开手动区）：
1. 页面初始化请求 `/auth/feishu/base-options`
2. 选择基地后请求 `/auth/feishu/team-options`
3. 选择团队后请求 `/auth/feishu/configs`
4. 选择飞书企业后，点击"使用飞书登录"
5. 页面跳转到 `/api/auth/feishu/login?action=login&configId=...`

折叠控件说明：
- `canOneClick` 时显示"一键登录失败请点击此处 ▼"（加粗加大）
- 无一键登录时显示"选择组织归属 ▼"（普通样式）
- 点击展开/折叠手动选择区

### 7.2 设置页 `SettingsPage.tsx`

设置页除了密码修改，还承载飞书绑定和解绑。

关键状态：
- `boundConfig`
- `selectedBaseId / selectedTeamId / selectedConfigId`
- `bindLoading / unbindLoading`
- `bindMessage / bindError`
- `freshUser`

当前行为：
1. 页面初始化请求 `/me` 获取最新用户信息
2. 同时请求 `/auth/feishu/base-options`
3. 如果用户已有 `feishuConfigId`，则通过 `/auth/feishu/configs?configId=...` 回填当前绑定配置
4. 未绑定用户可手动选择归属并发起绑定
5. 已绑定用户可直接解绑

绑定时的特殊处理：
- 前端把当前 JWT token 暂存到 `sessionStorage.feishu_bind_token`
- 这样即使回调页刷新，也能尝试恢复绑定所需 token

### 7.3 回调页 `FeishuCallbackPage.tsx`

回调页负责把飞书授权结果收口处理掉。

关键逻辑：
- 从 URL 读取 `code` 和 `state`
- 解析 `state.action`，判断是 `login` 还是 `bind`
- `bind` 场景会尝试从：
  1. `state.token`
  2. `sessionStorage.feishu_bind_token`
  3. `useAuthStore.getState().token`
  三层恢复绑定 token
- 使用 `inflightFeishuCallbacks` 去重同一授权回调请求

成功后行为：
- `bind`：提示成功并跳回 `/settings`
- `login`：写入 token / user / identities，然后根据身份数跳转 `/organization` 或 `/identity`

失败后行为：
- 绑定失败优先返回设置页
- 登录失败优先返回登录页
- 会对"授权码已使用 / 已过期 / 网络异常"等做友好文案转换

### 7.4 飞书企业配置页 `FeishuConfigPage.tsx`

该页面是组织侧飞书能力的上游入口。

当前页面能力：
- 按基地筛选配置
- 按团队筛选配置
- 按状态筛选 `active / paused`
- 新增配置
- 编辑配置
- 启停配置
- 对新增 / 编辑操作增加二次确认

当前实现特点：
- 列表不再一上来拉全量配置
- 必须先选基地才加载配置列表
- 统一展示回调地址 `${window.location.origin}/pc/auth/callback`
- 明确提示后端环境变量 `FEISHU_REDIRECT_URI` 也必须保持一致
- 点击"新增配置 / 保存修改"后不会直接落库，而是先弹出确认窗
- 确认窗会展示企业名称、基地、团队、App ID，帮助操作者二次核对
- 编辑场景下 `App Secret` 留空代表保留原密钥，填写则代表本次同步更新密钥

### 7.6 `shared/utils/feishu.ts` 的当前地位

飞书 App / H5 SDK 工具函数：
- `isInFeishuApp()`：检测飞书环境（UA 包含 lark/feishu）
- `getFeishuAuthCode(appId)`：调 `tt.requestAccess` 获取授权 code
- `initH5sdk(appId)`：移动端 H5 JSSDK 初始化
- `doAuthorize()`：内部执行授权 JSAPI

当前地位：
- **`FeishuEntryPage` 已经正式使用** `isInFeishuApp` 和 `getFeishuAuthCode`，是主链路核心依赖
- **`LoginPage`** 使用 `isInFeishuApp` 判断是否显示飞书 Tab 和一键登录按钮
- `initH5sdk` 目前属于移动端 H5 预留能力，PC 客户端 Electron 环境直接跳过

## 8. 后端实现说明

### 8.1 核心辅助函数

`auth/routes.ts` 中的关键辅助函数包括：

- `getFeishuConfigDelegate()`
  - 安全获取 `feishuEnterpriseConfig` Prisma delegate

- `getFeishuConfigById(configId)`
  - 只返回 `status = active` 的配置

- `requireFeishuConfig(config, res)`
  - 校验配置是否存在、是否有 `appId/appSecret`、是否已配置 `FEISHU_REDIRECT_URI`

- `feishuState(action, configId, token?)`
  - 构造发给飞书的 `state` 字符串，包含：
    - `action`
    - `configId`
    - `nonce`
    - 绑定场景下可选 `token`

- `parseFeishuState(raw)`
  - 回调时解析 `state`

- `getFeishuAppAccessToken(config)`
  - 通过 `appId/appSecret` 向飞书换取 `app_access_token`

- `exchangeFeishuCode(config, code)`
  - 用授权码换取用户 `open_id / union_id / name / avatar_url`

- `exchangeFeishuJsapiCode(config, code)`
  - 为 App / H5 SDK 预留的 code 交换方法

### 8.2 登录完成逻辑

`POST /auth/feishu/complete-login` 的核心步骤：

1. 校验 `code`
2. 解析 `state` 并拿到 `configId`
3. 读取对应的飞书企业配置
4. 向飞书换取用户资料
5. 用以下条件查找已绑定用户：
   - `feishuConfigId = 当前配置`
   - `feishuUnionId = union_id` 或 `feishuOpenId = open_id`
6. 若没找到用户，则报"该飞书账号尚未绑定系统账号"
7. 校验系统账号状态必须为 `active`
8. 更新 `lastLoginAt / feishuName / feishuAvatarUrl`
9. 查询当前用户所有 active 身份
10. 返回 JWT token + user + identities

### 8.3 绑定逻辑

`POST /auth/feishu/bind` 的核心步骤：

1. 要求 `authRequired`
2. 校验 `code` 与 `state.configId`
3. 根据飞书授权码换取 `open_id / union_id`
4. 检查是否已有其他系统账号在同一 `configId` 下占用了该飞书账号
5. 若已被其他用户占用，则报 `FEISHU_ALREADY_BOUND`
6. 更新当前登录用户：
   - `feishuConfigId`
   - `feishuOpenId`
   - `feishuUnionId`
   - `feishuName`
   - `feishuAvatarUrl`
   - `feishuBoundAt`
7. 返回 `{ bound: true }`

### 8.4 解绑逻辑

`DELETE /auth/feishu/bind` 很直接：

- 清空当前用户的所有飞书相关字段
- 返回 `{ bound: false }`

这意味着解绑后：
- 该系统账号不再与任何飞书账号关联
- 该飞书账号无法再通过当前绑定记录自动登录

### 8.5 App 内免登接口（正式使用）

以下接口当前已被正式使用：

- `GET /auth/feishu/app-ids`
  - 返回所有飞书企业配置的 `{ configId, appId }` 列表
  - `FeishuEntryPage` 用此接口将 URL 中的 `appId` 转换为 `configId`

- `POST /auth/feishu/app-login`
  - 接收 `{ code, configId }`，使用 JSAPI code 完成登录
  - 与 `complete-login` 逻辑相似，但 code 交换接口不同（`exchangeFeishuJsapiCode`）

- `GET /auth/feishu/jssdk-config`
  - 为移动端 H5 JSSDK 签名提供配置（Electron PC 端不需要）

- `POST /auth/feishu/app-bind`
  - 预留：飞书 App 内 code 绑定（当前设置页未接入）

## 10. 关键流程说明

### 10.1 飞书 PC 客户端免登流程（正式主链路）

```text
飞书 PC 客户端工作台
  -> 用户点击应用图标
  -> 跳转 /feishu-entry?appId=cli_xxx
  -> FeishuEntryPage 调 GET /auth/feishu/app-ids -> 匹配 configId
  -> 调 tt.requestAccess(appId) 获取 code
  -> POST /auth/feishu/app-login { code, configId }
  -> 后端查找已绑定用户并返回 token / identities
  -> 前端写入登录态 + localStorage.feishu_entry_app_id
  -> 跳转 /tasks/cockpit
```

### 10.2 退出后一键重登流程

```text
用户在系统内手动退出
  -> navigate /login
  -> LoginPage 读取 localStorage.feishu_entry_app_id
  -> canOneClick = true（在飞书 PC 客户端内）
  -> 飞书 Tab 自动激活，显示"一键飞书登录"按钮，手动选择区折叠
  -> 用户点击"一键飞书登录"
  -> window.location.href = /feishu-entry?appId=xxx
  -> 重走免登流程（同 10.1）
```

### 10.3 Web 飞书登录流程（手动选择组织）

```text
登录页 LoginPage
  -> 选择基地 -> 选择团队 -> 选择飞书企业
  -> 点击"使用飞书登录"
  -> 跳转 /api/auth/feishu/login?action=login&configId=...
  -> 飞书授权页
  -> 回调 /pc/auth/callback?code=...&state=...
  -> FeishuCallbackPage 调 /auth/feishu/complete-login
  -> 后端查找已绑定用户并返回 token / identities
  -> 前端写入登录态并跳转组织页或身份页
```

### 10.4 Web 飞书绑定流程

```text
用户先账号密码登录
  -> 进入 /settings
  -> 选择基地 / 团队 / 飞书企业
  -> 保存 token 到 sessionStorage.feishu_bind_token
  -> 跳转 /api/auth/feishu/login?action=bind&token=...&configId=...
  -> 飞书授权页
  -> 回调 /pc/auth/callback?code=...&state=...
  -> FeishuCallbackPage 恢复 token
  -> 调 /auth/feishu/bind
  -> 绑定成功后跳回 /settings
```

### 10.5 解绑流程

```text
SettingsPage
  -> 用户点击解绑
  -> 调 DELETE /auth/feishu/bind
  -> 后端清空用户飞书字段
  -> 前端刷新 freshUser / boundConfig / 提示信息
```

## 12. 环境变量与部署要求

### 12.1 必要环境变量

| 变量 | 说明 |
|---|---|

### 12.2 回调地址一致性要求

当前前端展示的统一回调地址为：

如果三者不一致，常见结果包括：
- 飞书授权后无法回调
- `FEISHU_REDIRECT_NOT_CONFIGURED`
- 回调地址白名单校验失败

要保证飞书链路可用，还需要：
- Prisma Client 已包含 `FeishuEnterpriseConfig` 模型
- 数据库中已存在 active 的飞书企业配置
- 对应 `appId / appSecret` 已正确录入

## 13. 常见错误与排障建议

### 13.1 登录页看不到飞书配置选项

优先检查：
- 是否存在 active 的 `FeishuEnterpriseConfig`
- `/auth/feishu/base-options` 是否返回空
- 所选基地下是否存在 active 配置对应的团队

### 13.2 飞书授权后提示未绑定

典型原因：
- 当前飞书账号尚未绑定任何系统账号
- 登录场景下只支持"已绑定飞书账号直接登录"

正确处理方式：
- 先用手机号密码登录系统
- 到 `/settings` 页完成飞书绑定
- 再使用飞书登录

### 13.3 绑定时报"已被其他系统账号绑定"

检查点：
- 数据库中是否已有其他用户占用了相同 `feishuOpenId` 或 `feishuUnionId`
- 是否使用了相同 `feishuConfigId`
- 是否测试过程中把同一个飞书账号绑定到了多个测试用户

### 13.4 回调页提示绑定会话已过期

典型原因：
- 当前浏览器丢失了 `sessionStorage.feishu_bind_token`
- 回调页刷新时没有从 `state.token` 或 store 中恢复到 token
- 用户未登录就发起了绑定流程

### 13.5 飞书授权码已失效 / 已使用

当前回调页已对以下情况做友好提示：
- code has been used
- authorization_code expired / invalid
- Failed to fetch

处理建议：
- 返回上一步重新发起授权
- 不要反复刷新旧回调 URL

### 13.6 飞书配置页新增 / 编辑异常

先确认当前交互是否符合预期：
- 点击"新增配置 / 保存修改"后，页面会先弹出二次确认窗
- 只有点击确认窗里的"确认新增 / 确认保存"后，才会真正调用接口
- 如果只是关闭确认窗或点击取消，不会写入数据库

确认已经真正提交后，如仍失败，优先检查：
- `FEISHU_REDIRECT_URI` 是否配置
- `appId` 是否重复
- 所选团队是否确实隶属于所选基地
- Prisma Client 是否已生成包含 `FeishuEnterpriseConfig`
- 编辑场景若未填写 `App Secret`，当前实现会保留原密钥，这属于正常行为，不是保存失败

## 14. 当前设计特点与维护建议

### 14.1 当前设计特点

1. **PC 客户端工作台走全自动免登**，用户无感知，体验最优
2. **退出后一键重登兜底**：`localStorage.feishu_entry_app_id` 记录上次 appId，退出后一键回到免登流程，无需重新选择组织
3. **Web 手动登录保留为兜底**：一键登录失败或换企业时，可展开手动选择区
4. 回调页统一收口处理 Web OAuth 登录与绑定两类动作（免登不走此页）
5. 绑定场景用 `state + sessionStorage + authStore` 三层兜底恢复 token
6. 飞书企业配置强绑定基地 / 团队，不是全局唯一租户配置
7. 菜单可见性、路由保护、接口权限当前已基本一致
8. 飞书企业配置的新增 / 编辑已经加入二次确认，优先降低后台误触风险

### 14.2 当前已知局限

1. **第一次登录必须走 PC 客户端工作台或 Web 手动选择**
   - 一键重登依赖 `localStorage.feishu_entry_app_id`，首次进入系统前该值为空

2. **一键重登后 appId 切换场景无感知**
   - 若用户在多企业下切换，localStorage 中只保留最近一次的 appId
   - 仍可通过"一键登录失败请点击此处"展开手动选择兜底

3. **绑定冲突依赖用户表唯一字段兜底**
   - 维护时需要同时关注配置维度与用户唯一约束

4. **飞书配置对组织主数据依赖很强**
   - 组织迁移、停用、菜单权限变更都可能影响飞书链路

### 14.3 维护建议

后续若继续维护，请优先同时检查：

1. `LoginPage.tsx`
2. `SettingsPage.tsx`
3. `FeishuCallbackPage.tsx`
4. `auth/routes.ts`
5. `FeishuConfigPage.tsx`
6. `organization/routes.ts`
7. `schema.prisma`

尤其注意以下四件事：

1. **回调地址是否一致**
2. **配置状态是否 active**
3. **绑定唯一性是否被破坏**
4. **前端菜单可见性和路由保护是否同步**

## 15. 维护结论

当前项目中的飞书能力，本质上是一个：

- 以组织主数据为配置来源
- 以飞书企业配置为租户选择器
- **以 `FeishuEntryPage + JSAPI` 为 PC 客户端主登录链路**
- **以 `localStorage.feishu_entry_app_id` 为退出后免重选的兜底记忆**
- 以回调页为 Web OAuth 流的统一收口点
- 以 `User.feishu*` 字段为绑定结果存储
- 以 JWT 登录态和绑定 token 为会话基础

的完整飞书登录与组织协同体系。
