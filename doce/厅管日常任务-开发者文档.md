# 厅管日常任务 — 开发者文档

> 模块路径：`backend/src/modules/task/hall-daily/`  
> 数据库模型前缀：`HallTask*`  
> API 基础路径：`/api/hall-daily`

---

## 一、业务概述

厅管日常任务是独立于主播任务体系的另一套任务体系，以**团队（TEAM）**为管理中心，由团队管理员维护任务模板并向团队下各个**直播厅（HALL）**发布任务。厅管每天在所属厅的任务记录中完成各题目的作答，系统自动为每个生效中的直播厅每日生成一条记录（幂等创建）。

---

## 二、数据模型

### 2.1 模型关系图

```
HallTaskTemplate（模板）
  └─ HallTaskItem[]（题目）
       └─ HallTaskItemOption[]（选项，单选/多选题使用）

HallTaskAssignment（发布任务）
  ├─ HallTaskAssignmentTarget[]（目标厅，指向 OrgUnit.HALL）
  └─ HallTaskRecord[]（执行记录，每厅每日一条）
       ├─ HallTaskLeaveRequest[]（请假申请，执行人申请、管理员审批）
       └─ HallTaskItemRecord[]（题目作答记录，每题一条）
            └─ HallTaskItemAttachment[]（附件，仅 ATTACHMENT 题型使用）
```

### 2.2 HallTaskTemplate（任务模板）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String(UUID) | 主键 |
| `title` | String | 模板标题 |
| `description` | String? | 描述（Text） |
| `teamOrgId` | String | 所属团队 OrgUnit.id（orgType = TEAM） |
| `createdBy` | String | 创建者 userId |
| `version` | Int | 版本号（默认 1，目前为历史保留字段） |
| `status` | HallTaskTemplateStatus | 状态：`draft` / `archived` |
| `createdAt` | DateTime | 创建时间 |
| `updatedAt` | DateTime | 更新时间 |

**状态枚举 `HallTaskTemplateStatus`**

| 值 | 含义 |
|---|---|
| `draft` | 草稿（默认，可编辑、可发布） |
| `archived` | 已归档（不可再发布新任务） |

> 注意：模板没有 `published` 状态，`draft` 模板可以直接发布任务，只有手动归档后才无法使用。

---

### 2.3 HallTaskItem（题目）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String(UUID) | 主键 |
| `templateId` | String | 所属模板 |
| `sortOrder` | Int | 排序（升序展示） |
| `itemType` | HallTaskItemType | 题目类型 |
| `title` | String | 题目标题 |
| `isRequired` | Boolean | 是否必填（默认 true） |
| `linkUrl` | String? | 仅 LINK 类型使用，规范化后的 HTTP/HTTPS 地址 |

**题目类型枚举 `HallTaskItemType`**

| 值 | 说明 |
|---|---|
| `QA` | 问答 |
| `SINGLE_CHOICE` | 单选 |
| `MULTI_CHOICE` | 多选 |
| `FILL_BLANK` | 填空（前端展示为"待办确认"，勾选即完成） |
| `LINK` | 学习链接（需填写跳转地址，点击确认后才能提交） |
| `ATTACHMENT` | 附件上传（图片，单张 ≤ 1MB，支持拖拽上传） |

---

### 2.4 HallTaskAssignment（发布任务）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String(UUID) | 主键 |
| `templateId` | String | 所用模板 |
| `teamOrgId` | String | 所属团队 |
| `status` | HallTaskAssignmentStatus | 任务状态 |
| `effectMode` | HallTaskEffectMode | 生效方式 |
| `effectiveAt` | DateTime? | 实际生效时间 |
| `publishedAt` | DateTime? | 发布时间 |
| `endedAt` | DateTime? | 结束时间 |
| `createdBy` | String | 创建者 userId |
| `createdByOrgId` | String | 创建者所在 org |

**状态枚举 `HallTaskAssignmentStatus`**

| 值 | 含义 | 可操作 |
|---|---|---|
| `draft` | 草稿 | 可编辑、可发布、可删除 |
| `scheduled` | 待生效（次日零点激活） | 可结束、可删除（取消） |
| `active` | 生效中 | 可结束 |
| `ended` | 已结束 | 只读，保留完成记录 |
| `deleted` | 已删除（软删，目前由物理删除代替） | — |

**生效方式枚举 `HallTaskEffectMode`**

| 值 | 含义 |
|---|---|
| `immediate` | 立即生效（发布后状态直接变为 `active`，同时将同团队其他 `active` 任务改为 `ended`） |
| `next_midnight` | 次日零点生效（发布后状态变为 `scheduled`；旧 `active` 任务**保持不变**，等零点定时器激活新任务时再一并结束旧任务） |

---

### 2.5 HallTaskRecord（执行记录）

每个生效任务的目标厅，每个北京日期自动创建一条记录（幂等）。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String(UUID) | 主键 |
| `assignmentId` | String | 所属发布任务 |
| `hallOrgId` | String | 执行厅 OrgUnit.id（orgType = HALL） |
| `recordDate` | String | 北京日期，格式 `YYYY-MM-DD` |
| `status` | HallTaskRecordStatus | 记录状态 |
| `totalItems` | Int | 题目总数 |
| `doneItems` | Int | 已完成题目数 |
| `submittedAt` | DateTime? | 提交时间 |
| `submittedBy` | String? | 提交者 userId |
| `leaveRequests` | HallTaskLeaveRequest[] | 关联的请假申请记录 |

**唯一约束**：`(assignmentId, hallOrgId, recordDate)` — 同一发布任务同一厅同一天只有一条记录。

**记录状态枚举 `HallTaskRecordStatus`**

| 值 | 含义 |
|---|---|
| `pending` | 待开始 |
| `in_progress` | 进行中 |
| `submitted` | 已提交 |
| `overdue` | 已逾期 |

### 2.6 HallTaskLeaveRequest（请假申请）

> 表名：`hall_task_leave_requests`  
> 用途：记录厅管针对某条每日任务记录发起的请假申请，以及管理员审批结果。  
> 设计原则：请假是审批流程，不直接混入 `HallTaskRecord.status`；看板展示状态由 `HallTaskRecord` + 当前有效请假申请动态合并计算。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | String(UUID) | 主键 |
| `taskRecordId` | String | 对应的 `HallTaskRecord.id` |
| `applicantUserId` | String | 申请人 userId |
| `applicantName` | String? | 申请人展示名，优先取用户昵称，兜底手机号 |
| `reason` | String | 请假原因（Text，必填） |
| `status` | HallTaskLeaveStatus | 请假状态 |
| `reviewedBy` | String? | 审批人 userId |
| `reviewedAt` | DateTime? | 审批时间 |
| `reviewComment` | String? | 审批意见（Text） |
| `createdAt` | DateTime | 申请时间 |
| `updatedAt` | DateTime | 更新时间 |

**索引**：

| 索引字段 | 用途 |
|---|---|
| `taskRecordId` | 查询某条任务记录关联的请假申请 |
| `status` | 管理员看板统计待审批/已请假数量 |
| `applicantUserId` | 查询某用户发起的请假申请 |

**状态枚举 `HallTaskLeaveStatus`**

| 值 | 含义 |
|---|---|
| `pending` | 待审批 |
| `approved` | 已同意 |
| `rejected` | 已拒绝 |
| `cancelled` | 已撤回 |

**业务约束**：

- 同一条 `HallTaskRecord` 同一时间只允许存在一个 `pending` 请假申请。
- 已提交的任务不允许申请请假。
- 任务日次日 16:00 后不允许新申请或撤回请假。
- 管理员可以在补录截止后继续审批已经提交的 `pending` 请假申请。
- 已批准请假后，该任务禁止继续作答和提交。
- 一条任务记录可以保留多条历史请假申请，例如第一次被拒绝后再次申请。

## 三、权限体系

### 3.1 允许维护的角色

```
DEV_ADMIN  — 开发者超管
HQ_ADMIN   — 总部管理员
BASE_ADMIN — 基地管理员
TEAM_ADMIN — 团队管理员（核心角色）
```

所有接口均先调用 `ensureHallDailyRole()` 校验角色，非以上角色直接返回 403。

### 3.2 团队范围校验

每次操作都会调用 `resolveTeamOrg(teamOrgId, scopePath, roleCode)`：

- 校验 teamOrgId 对应的 OrgUnit 存在且状态为 `active`
- 校验该组织类型必须为 `TEAM`
- `DEV_ADMIN` 跳过范围校验；其他角色要求操作的团队路径在自己的 `scopePath` 管辖范围内

### 3.3 接口权限要求

| 接口类型 | 所需 Permission |
|---|---|
| 查看列表/详情/预览 | `task:assignment:view` |
| 新建/编辑/删除/复制/归档/发布/结束 | `task:template:manage` |
| 厅管填报（执行层） | `task:record:submit` |
| 厅管日常请假审批 | `task:report:view` |

### 3.4 厅管日常请假权限

| 操作 | 所需权限 | 角色/范围限制 |
|---|---|---|
| 申请请假 | `task:record:submit` | 当前用户必须拥有该厅活跃 `HALL_MANAGER` 身份 |
| 撤回请假 | `task:record:submit` | 只能撤回本人发起且状态为 `pending` 的申请 |
| 同意/拒绝请假 | `task:report:view` | `DEV_ADMIN` / `HQ_ADMIN` / `BASE_ADMIN` / `TEAM_ADMIN`，且必须通过 `scopePath` 范围校验 |

审批范围规则：

- `DEV_ADMIN` 可审批所有厅。
- `HQ_ADMIN` / `BASE_ADMIN` / `TEAM_ADMIN` 只能审批自己 `scopePath` 管辖范围内的厅。
- `HALL_MANAGER` 只能申请和撤回自己的请假，不能审批厅管日常请假。

### 3.5 豁免审批权限（主播任务模块）

> 本节描述的是**主播任务**豁免审批权限，与厅管日常任务模块本身无关，但影响厅管/团队管理角色在任务管理看板中的操作能力。

**`task:exemption:review`（审批豁免）** 权限自 2026-06-26 起扩展至：

| 角色 | 是否有审批豁免权限 |
|---|---|
| `HQ_ADMIN` | ✅ |
| `BASE_ADMIN` | ✅ |
| `TEAM_ADMIN` | ✅（新增） |
| `HALL_MANAGER` | ✅（新增） |

**业务限制**：权限打通后，后端 `canReviewDailyExemption` 函数仍会进行 `scopePath` 作用域校验：
- 团队管理员只能审批自己团队范围内主播的豁免申请
- 厅管只能审批自己所属厅范围内主播的豁免申请
- 越权操作返回 `EXEMPTION_REVIEW_FORBIDDEN`

**同步文件**：
- `backend/src/shared/sync-role-permissions.ts` — `TEAM_ADMIN` / `HALL_MANAGER` 的 `rolePermissions` 列表
- `backend/src/shared/task-assignment-schema-compat.ts` — `task:exemption:review` 的 `roleCodes` 列表

## 四、API 接口

所有接口均需要：
- `Authorization` Header（JWT Token）
- 当前身份（`identityRequired` 中间件）

### 4.1 模板管理

#### `GET /api/hall-daily/templates`

列出模板列表。

**Query 参数**

| 参数 | 必填 | 说明 |
|---|---|---|
| `teamOrgId` | 是 | 团队 OrgUnit ID |
| `status` | 否 | 过滤状态：`draft` / `archived` |
| `neverPublished` | 否 | `true` 时只返回从未被发布过的纯净草稿（即没有任何 `scheduled`/`active`/`ended` 状态 assignment 的模板），用于发布向导左栏草稿列 |
| `limit` | 否 | 每页条数 |
| `offset` | 否 | 偏移量 |

**响应示例**

```json
{
  "ok": true,
  "data": [
    {
      "id": "xxx",
      "title": "厅管日报模板",
      "status": "draft",
      "teamOrgId": "yyy",
      "version": 1,
      "createdAt": "2026-01-01T00:00:00.000Z",
      "_count": { "assignments": 3 },
      "items": [...]
    }
  ]
}
```

---

#### `GET /api/hall-daily/templates/:id`

查看单个模板（含完整题目和选项）。

**Query 参数**：`teamOrgId`（可选，用于额外校验归属）

---

#### `POST /api/hall-daily/templates`

新建模板。

**Body**

```json
{
  "title": "厅管日报",
  "description": "每日填报",
  "teamOrgId": "xxx",
  "items": [
    {
      "sortOrder": 0,
      "itemType": "FILL_BLANK",
      "title": "今日开播时长（小时）",
      "isRequired": true
    },
    {
      "sortOrder": 1,
      "itemType": "SINGLE_CHOICE",
      "title": "今日直播状态",
      "isRequired": true,
      "options": [
        { "sortOrder": 0, "label": "正常" },
        { "sortOrder": 1, "label": "异常" }
      ]
    },
    {
      "sortOrder": 2,
      "itemType": "LINK",
      "title": "观看培训视频",
      "isRequired": true,
      "linkUrl": "https://example.com/training"
    },
    {
      "sortOrder": 3,
      "itemType": "ATTACHMENT",
      "title": "上传今日直播截图",
      "isRequired": true
    }
  ]
}
```

> `LINK` 类型题目 `linkUrl` 为必填，必须是 `http://` 或 `https://` 协议的完整地址，否则返回 `HALL_TASK_LINK_URL_INVALID`。  
> 如不带协议前缀，系统会自动补充 `https://`。

---

#### `PATCH /api/hall-daily/templates/:id`

编辑模板标题、描述及题目。

**Body**（所有字段可选，`items` 若传则全量替换）

```json
{
  "title": "新标题",
  "description": "新描述",
  "items": [...]
}
```

**限制**：模板下存在 `active` 或 `scheduled` 状态的发布任务时，不允许修改，返回 `HALL_TASK_TEMPLATE_IN_USE`。

---

#### `DELETE /api/hall-daily/templates/:id`

删除模板（物理删除）。

**Query 参数**：`teamOrgId`（可选）

**删除规则**：

| 关联 assignment 状态 | 是否允许删除 |
|---|---|
| `draft` | 允许（级联清理 assignment 及 target） |
| `scheduled` | **拒绝**，返回 `HALL_TASK_TEMPLATE_HAS_ASSIGNMENTS` |
| `active` | **拒绝** |
| `ended` | **拒绝**（保留完成记录） |

---

#### `POST /api/hall-daily/templates/:id/copy`

复制模板（生成标题为"原标题（副本）"的新草稿，题目选项全部复制）。

---

#### `POST /api/hall-daily/templates/:id/archive`

归档模板（状态变为 `archived`，后续不可再用此模板发布新任务）。

### 4.2 发布任务管理

#### `GET /api/hall-daily/assignments`

列出发布任务。

**Query 参数**

| 参数 | 必填 | 说明 |
|---|---|---|
| `teamOrgId` | 是 | 团队 OrgUnit ID |
| `status` | 否 | 过滤状态 |
| `limit` | 否 | 每页条数 |
| `offset` | 否 | 偏移量 |

**响应字段**（每条）

```json
{
  "id": "xxx",
  "status": "active",
  "effectMode": "immediate",
  "effectiveAt": "2026-01-01T10:00:00.000Z",
  "publishedAt": "2026-01-01T10:00:00.000Z",
  "template": { "id": "...", "title": "厅管日报", "status": "draft" },
  "targets": [
    { "id": "...", "hallOrgId": "...", "hallOrg": { "id": "...", "name": "1号厅" } }
  ],
  "_count": { "records": 30 }
}
```

---

#### `POST /api/hall-daily/assignments/draft`

保存发布草稿（创建或更新）。

**Body**

```json
{
  "assignmentId": "xxx（更新时传入，创建时省略）",
  "templateId": "yyy",
  "teamOrgId": "zzz",
  "hallOrgIds": ["hall1", "hall2"],
  "effectMode": "immediate"
}
```

- `hallOrgIds` 中每个厅必须属于 `teamOrgId` 对应的团队（路径前缀校验）
- 更新时只能修改 `draft` 状态的草稿
- 每次调用会**全量覆盖**目标厅列表

---

#### `GET /api/hall-daily/assignments/:id/preview`

获取发布预览信息（发布前调用，用于展示冲突提示）。

**响应**

```json
{
  "assignmentId": "xxx",
  "templateId": "yyy",
  "templateTitle": "厅管日报",
  "effectMode": "immediate",
  "targetOrgCount": 3,
  "targetOrgs": [{ "id": "...", "name": "1号厅" }],
  "overlappingAssignments": [
    { "id": "...", "title": "旧模板", "status": "active" }
  ]
}
```

`overlappingAssignments` 为与当前目标厅有交集的其他 `active`/`scheduled` 任务，前端可用于提示用户"发布后将覆盖以下任务"。

---

#### `POST /api/hall-daily/assignments/:id/publish`

正式发布任务。

**Body**

```json
{
  "effectMode": "immediate",
  "teamOrgId": "xxx（可选，额外归属校验）"
}
```

**发布逻辑**

1. 当前发布任务必须处于 `draft` 状态
2. 必须至少有一个目标厅
3. 模板不能是 `archived` 状态
4. 同团队不能已有 `scheduled` 状态的待生效任务（防止排队混乱）
5. `effectMode = immediate`：直接将状态改为 `active`，同时将同团队其他 `active` 任务改为 `ended`
6. `effectMode = next_midnight`：状态改为 `scheduled`，`effectiveAt` 设为下一个北京时间零点；此时旧 `active` 任务**不提前结束**，等零点定时器激活新任务时一并结束

---

#### `POST /api/hall-daily/assignments/:id/close`

手动结束任务（`active` 或 `scheduled` → `ended`）。

**Body**：`teamOrgId`（可选）

---

#### `DELETE /api/hall-daily/assignments/:id`

删除发布草稿。

**删除规则**：

| 状态 | 是否允许删除 |
|---|---|
| `draft` | 允许 |
| `scheduled` | 允许（相当于取消排队） |
| `active` | **拒绝**，返回 `HALL_TASK_ASSIGNMENT_NOT_DRAFT` |
| `ended` | **拒绝** |

### 4.3 执行层（厅管填报）

> 权限：`task:record:submit`  
> 厅管只能操作自己持有活跃 `HALL_MANAGER` 身份所对应的厅的记录。

单题作答（保存或更新一条 HallTaskItemRecord）。

#### `POST /api/hall-daily/my-records/:id/leave-requests`

执行人申请请假。

**权限**：`task:record:submit`，且当前用户必须拥有该记录所属厅的活跃 `HALL_MANAGER` 身份。

**限制**：
- `reason` 必填，否则返回 `HALL_TASK_LEAVE_REASON_REQUIRED`

## 五、定时任务

由 `server.ts` 每分钟轮询调用两个函数：

### 5.1 `activateHallDailyScheduled()`

激活到期的 `scheduled` 任务：

1. 查找所有 `status = scheduled` 且 `effectiveAt <= now` 的任务
2. 对每条任务开启事务：
   - 再次确认状态（防并发）
   - 将同团队内其他 `active` 任务改为 `ended`（设置 `endedAt = now`）
   - 将本任务改为 `active`

### 5.2 `ensureHallDailyRecordsForToday()`

为所有生效中任务的目标厅创建当日记录：

1. 查找所有 `status = active` 的任务（含目标厅和模板题目）
2. 对每个目标厅，在当日北京日期下创建 `HallTaskRecord`（幂等）
3. 同时预建每道题目对应的 `HallTaskItemRecord`（`status = pending`）

> 北京时间计算通过 `formatBeijingDate()` / `makeBeijingDate()` 工具函数实现，避免服务器时区影响。

## 六、时间边界说明

| 时间点 | 说明 |
|---|---|
| 当天 23:59:59（北京） | `getDailyTaskDayEnd`，超过后 record 标记为 `overdue` |
| 次日 16:00:00（北京） | `getDailyTaskSupplementDeadline`，超过后补录窗口关闭，无法再修改昨日记录 |
| 次日零点（北京） | `next_midnight` 模式的 `effectiveAt`，定时器激活 `scheduled` 任务 |

请假窗口规则：

| 行为 | 时间限制 |
|---|---|
| 申请请假 | 任务日当天及次日 16:00 前 |
| 撤回请假 | 待审批状态，且任务日次日 16:00 前 |
| 审批请假 | 管理员可在次日 16:00 后继续审批已经提交的待审批申请 |
| 已批准请假后作答/提交 | 禁止 |

补录窗口判断（`GET /api/hall-daily/my-records`）：
- `canSupplementYesterday = now < 当天北京 16:00`
- 若为 `true`，接口同时返回今日和昨日记录；否则只返回今日记录

## 八、看板展示状态与统计口径

厅管日常任务看板不直接信任 `HallTaskRecord.status`，而是根据任务记录、当前时间、请假申请动态计算展示状态。

### 8.1 展示状态优先级

| 优先级 | 展示状态 | 文案 | 来源 |
|---|---|---|---|
| 1 | `submitted` | 已提交 | `HallTaskRecord.status = submitted` |
| 2 | `leave_approved` | 已请假 | 存在 `HallTaskLeaveRequest.status = approved` |
| 3 | `leave_pending` | 请假待审 / 请假审核中 | 存在 `HallTaskLeaveRequest.status = pending` |
| 4 | `overdue` | 已逾期 | 当前时间超过任务日 23:59:59 且未提交/未批准请假 |
| 5 | `in_progress` | 进行中 | 未逾期且 `doneItems > 0` |
| 6 | `pending` | 未开始 | 未逾期且 `doneItems = 0` |
| 7 | `null` | 暂无数据 / 无记录 | 当天没有任务记录 |

### 8.2 请假申请选择规则

当一条任务记录存在多条请假申请时，详情接口返回一个用于展示/审批的 `leaveRequest`：

```text
approved 优先，其次 pending，再其次 rejected，cancelled 不展示
```

这样可以避免历史拒绝记录覆盖当前待审批申请。

### 8.3 管理员统计口径

管理员总览和团队厅列表新增统计：

| 字段 | 说明 |
|---|---|
| `leaveApprovedCount` / `leaveApprovedHalls` | 已批准请假的厅数 |
| `leavePendingCount` / `leavePendingHalls` | 请假待审批的厅数 |

完成率口径：

```text
completionRate = (submittedCount + leaveApprovedCount) / assignedHalls
```

前端仍单独展示“已提交”和“已请假”，避免将请假误解为提交。

## 九、重要业务规则汇总

1. **同团队同一时刻只允许一个 `active` 任务**：发布新任务时，旧的 `active` 任务自动结束。
2. **同团队最多一个 `scheduled` 任务**：若已有待生效任务，发布任务时直接拦截，防止排队混乱。
3. **`next_midnight` 模式下旧任务不提前结束**：旧 `active` 任务在零点定时器激活新任务时才一并结束，两者在发布到零点之间会共存（但厅管只能看到 `active` 任务的记录）。
4. **模板有 `ended` 记录时不可删除**：`ended` 状态的 assignment 含有厅管完成记录，属于历史数据，只能归档模板，不可删除。
5. **模板有 `active`/`scheduled` 任务时不可编辑题目**：防止进行中的任务与模板不一致。
6. **目标厅归属校验**：发布草稿时，`hallOrgIds` 中所有厅必须通过路径前缀校验属于对应团队。
7. **每日记录幂等创建**：定时任务每分钟运行，双重检查（查询 + 事务内再查）防止并发创建重复记录。
8. **北京时间统一**：所有涉及"今天/次日零点"的计算均使用北京时间（UTC+8），不依赖服务器时区。
9. **发布向导左栏只展示纯净草稿**：`HallDailyIssuePage` 的"模板草稿"列请求时携带 `neverPublished=true`，一旦某模板被发布过任务（无论任务当前是什么状态），就不再出现在左栏，避免用户同时看到"模板草稿"和右边三列中同一模板的任务记录，消除心智负担。
10. **发布成功后向导自动归位**：`handlePublish` 成功后在调用 `onIssued()` 前重置向导状态（`step → 1`，清空已选模板、已选厅、草稿 ID），确保导航回同一页面时向导展示第一步，而非停留在第三步预览页。
11. **厅管权限边界**：执行层接口通过 `HALL_MANAGER` 角色 + `orgId` 双重校验，确保厅管只能操作自己负责的厅的记录，越权访问一律返回 `HALL_TASK_RECORD_NOT_FOUND`（不泄露记录是否存在）。
12. **逾期后仍可补录**：record 逾期（超过当天 23:59）后不会立即禁止作答，厅管仍可在次日 16:00 前完成补录并提交。
13. **厅管/团队管理可审批主播豁免申请**：自 2026-06-26 起，`TEAM_ADMIN` 和 `HALL_MANAGER` 拥有 `task:exemption:review` 权限，可在任务管理看板中对自己管辖范围内主播的豁免申请执行同意或驳回操作，后端 scope 作用域校验保证不能跨范围审批。
14. **主播日常任务看板切换厅/日期自动刷新**：自 2026-06-26 起，`DailyTaskDashboardPage` 的基地下拉框和日期选择器的 `onChange` 均直接调用 `load()`，切换厅或改变日期后无需再手动点击"查询看板"，数据自动刷新。

### 请假表生产 SQL（历史迁移链异常时可手动执行）

如果 `npx prisma migrate deploy` 因历史迁移链问题无法执行，可在确认生产库当前结构后手动执行以下 SQL 创建请假表：

```sql
CREATE TABLE `hall_task_leave_requests` (
  `id` VARCHAR(191) NOT NULL,
  `task_record_id` VARCHAR(191) NOT NULL,
  `applicant_user_id` VARCHAR(191) NOT NULL,
  `applicant_name` VARCHAR(191) NULL,
  `reason` TEXT NOT NULL,
  `status` ENUM('pending', 'approved', 'rejected', 'cancelled') NOT NULL DEFAULT 'pending',
  `reviewed_by` VARCHAR(191) NULL,
  `reviewed_at` DATETIME(3) NULL,
  `review_comment` TEXT NULL,
  `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  `updated_at` DATETIME(3) NOT NULL,
  PRIMARY KEY (`id`),
  INDEX `hall_task_leave_requests_task_record_id_idx` (`task_record_id`),
  INDEX `hall_task_leave_requests_status_idx` (`status`),
  INDEX `hall_task_leave_requests_applicant_user_id_idx` (`applicant_user_id`),
  CONSTRAINT `hall_task_leave_requests_task_record_id_fkey`
    FOREIGN KEY (`task_record_id`) REFERENCES `hall_task_records` (`id`)
    ON DELETE RESTRICT ON UPDATE CASCADE
) DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 回滚方案

Prisma 不提供自动回滚，如需回滚请手动执行反向 SQL：

```sql
-- 回滚 add_hall_task_leave_requests
ALTER TABLE `hall_task_leave_requests` DROP FOREIGN KEY `hall_task_leave_requests_task_record_id_fkey`;
DROP TABLE `hall_task_leave_requests`;

-- 回滚 add_hall_task_item_attachments
ALTER TABLE `hall_task_item_attachments` DROP FOREIGN KEY `hall_task_item_attachments_hall_task_item_record_id_fkey`;
DROP TABLE `hall_task_item_attachments`;

-- 回滚 add_hall_task_record_date_hall_idx
DROP INDEX `hall_task_records_record_date_hall_org_id_idx` ON `hall_task_records`;

-- 回滚 remove_hall_template_published_status（枚举无法自动回滚，需重新 ALTER）
ALTER TABLE `hall_task_templates` MODIFY COLUMN `status` ENUM('draft', 'published', 'archived') NOT NULL DEFAULT 'draft';
```
