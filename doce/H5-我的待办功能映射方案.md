# H5-我的待办功能映射方案

## 1. 目标
在**不干预现有 PC 项目文件**的前提下，独立开发一个面向手机端 H5 的“我的待办”应用。

目标不是重做一套简化版，而是：
- 尽量保留现有“我的待办”业务能力
- 对交互进行手机化重构
- 复用现有后端 API 与业务口径
- 避免影响现有桌面端页面与样式

---

## 2. 现有“我的待办”核心来源
当前核心交互文件：
- `frontend/src/pages/tasks/my-todos/TaskRecordCard.tsx`

从现有实现看，PC 端“我的待办”已经不仅仅是展示卡片，而是完整承载了：
- 待办展示
- 子任务填写
- 附件上传
- 日常补录
- 管理式协同信息展示
- 提交整条任务
- 豁免申请

因此 H5 设计应按“执行闭环迁移”来做，而不是只做列表页。

---

## 3. 现有功能块拆解

### 3.1 卡片级基础信息
当前卡片已覆盖：
- 任务状态
- 任务分类（DAILY / TEMPORARY）
- 临时任务模式（ACCOUNT / ANCHOR / MANAGER）
- 主体类型（USER / ORG）
- 主体名称
- 组织类型标签（BASE / TEAM / HALL）
- 截止时间
- 任务说明
- 发布人
- 协同人数 / 可见身份数
- 当前进度
- 是否逾期 / 紧急

### 3.2 子任务展开能力
当前支持：
- 卡片展开/收起
- 子任务按“未完成 / 已完成”分组
- 已完成子任务可再次查看
- 完成态展示摘要

### 3.3 子任务类型支持
当前已支持以下 6 类：
- `QA`
- `SINGLE_CHOICE`
- `MULTI_CHOICE`
- `FILL_BLANK`
- `LINK`
- `ATTACHMENT`

### 3.4 子任务填写行为
当前支持：
- 问答填写并提交
- 单选点击即提交
- 多选后确认提交
- 待办确认类一键完成
- 学习链接打开后确认完成
- 图片上传
- 上传附件后确认提交

### 3.5 日常任务特殊逻辑
当前包含：
- 今日任务识别
- 昨日补录识别
- 补录日期提示
- 日常任务状态口径
- 日常任务整体提交

### 3.6 临时任务特殊逻辑
当前包含：
- 触达式任务展示
- 主播式任务展示
- 管理式任务展示
- 临时任务整体提交
- 管理式组织主体补充能力

### 3.7 管理式任务特殊信息
当前 PC 待办已体现：
- 组织主体视角
- 协同维护提示
- 最近填写人
- 子任务完成人
- 协同填写记录 / 无记录提示

### 3.8 附件能力
当前包含：
- 图片选择
- 图片大小限制（1MB）
- 上传后预览
- 多附件显示
- 附件任务确认提交

### 3.9 豁免能力
当前包含：
- 申请豁免
- 输入豁免原因
- 查看豁免状态
- 取消 / 回撤相关处理

## 5. H5 信息架构建议

建议 H5 拆成 4 个核心页面：

### 5.1 登录页
职责：
- 手机号密码登录
- 后续可接飞书免登

### 5.2 我的待办列表页
职责：
- 展示全部待办入口
- 显示任务状态/进度/截止时间
- 提供简单筛选

### 5.3 任务详情页
职责：
- 查看任务完整信息
- 查看子任务分组
- 查看完成人 / 协同信息
- 承接整体提交按钮

### 5.4 子任务编辑页 / 底部抽屉
职责：
- 执行填写动作
- 上传图片
- 完成当前子任务

## 6. H5 页面功能映射

## 6.1 列表页映射
### 对应 PC 能力
PC 当前卡片包含：
- 状态
- 标题
- 截止时间
- 进度
- 发布人
- 主体信息

### H5 建议承接方式
每张卡片显示：
- 任务标题
- 任务标签（日常 / 临时 / 管理式等）
- 状态
- 截止时间
- 进度（doneItems / totalItems）
- 主体名称
- 主操作按钮：`继续处理` / `查看详情`

### H5 不建议直接照搬
- 不建议在列表页内直接展开复杂编辑器
- 不建议在列表页内展示过多协同说明
- 不建议在列表页中塞附件上传区域

---

## 6.2 详情页映射
### 对应 PC 能力
PC 展开卡片后展示：
- 子任务列表
- 已完成 / 未完成分组
- 各类题型交互
- 补录说明
- 完成人摘要

### H5 建议承接方式
详情页展示：
- 顶部任务摘要
- 主体信息
- 进度
- 发布人
- 截止时间
- 日常任务补录文案
- 子任务分组（未完成优先）
- 底部固定操作栏

---

## 6.3 子任务编辑映射
### QA
PC 现状：输入框 + 提交按钮
H5 建议：
- 单独输入区
- 大按钮提交

### SINGLE_CHOICE
PC 现状：点击选项即提交
H5 建议：
- 列表单选卡片
- 点击即完成

### MULTI_CHOICE
PC 现状：多选后确认
H5 建议：
- 选项列表 + 底部确认按钮

### FILL_BLANK
PC 现状：一键勾选确认
H5 建议：
- 直接保留为大按钮

### LINK
PC 现状：打开学习链接后确认提交
H5 建议：
- 一键打开链接
- 返回后点击“已完成学习”确认

### ATTACHMENT
PC 现状：上传图片 + 预览 + 确认
H5 建议：
- 支持拍照 / 相册
- 上传后即时预览
- 单独按钮确认附件提交

## 7. 管理式任务专项映射

### 7.1 当前正确业务口径
管理式任务的重点不是“所有人继续补填”，而是：
- 某个子任务由谁完成
- 其他管理进入后能看到是谁完成的
- 避免重复填写

### 7.2 H5 必须保留的管理式信息
建议在 H5 详情页保留：
- 主体名称（组织名）
- 组织类型
- 协同人数
- 最近填写人
- 每个子任务的完成人
- 完成时间
- 当前结果

### 7.3 H5 表现建议
每个子任务下显示：
- 状态：已完成 / 未完成
- 完成人：张三
- 完成时间
- 当前结果摘要

若无完成人数据：
- 不额外伪造展示
- 保留当前结果 + 状态

## 8. 日常任务专项映射

### 8.1 必须保留口径
日常任务不是 backlog，而是：
- 今天
- 昨日补录（若补录窗口仍有效）

### 8.2 H5 顶部提示建议
在日常任务详情页显示：
- 任务日期
- 是否补录
- 补录截止提示

### 8.3 提交逻辑
必须保留：
- 子任务逐项完成
- 必做项完成后整条任务提交

## 9. 上传与媒体交互建议

### 9.1 当前 PC 能力
PC 当前只允许：
- 图片上传
- 单张 ≤ 1MB

### 9.2 H5 建议增强
H5 可在不改协议的前提下优化体验：
- 拍照优先
- 相册选择
- 上传中 loading
- 上传成功后即时预览
- 上传失败明确提示

### 9.3 第一版是否改协议
不建议第一版改上传接口协议。
建议直接复用当前：
- `recordApi.submitItemRecord`
- `uploadApi.upload`

## 13. 风险与注意事项

### 13.1 不要误把 H5 做成“只读版”
当前 PC 我的待办本质是执行页，不是详情页。H5 第一版必须支持执行闭环。

### 13.2 不要遗漏管理式“完成人”语义
管理式任务最关键的是：
- 哪项是谁完成的
而不是仅展示任务整体已完成。

### 13.3 不要混淆日常任务与临时任务
两类任务在：
- 口径
- 状态
- 展示
- 提交
上都不同。
H5 必须按现有业务区分。

### 13.4 不要直接复制 PC 交互
PC 当前很多交互适合桌面端，但不适合手机：
- 行内输入
- 同卡片内堆太多控件
- 复杂展开结构
H5 需要重做交互承接。

---

## 14. 建议结论
如果你的目标是：
- 独立开发
- 不干预现有项目文件
- 尽量保留当前“我的待办”全部能力

那么最佳方案是：

1. 新建独立 H5 项目
2. 以现有 `TaskRecordCard.tsx` 为业务功能蓝本
3. 复用后端 API，不复用现有桌面页面组件
4. 第一版先完整保留“执行闭环”，再逐步优化交互

---

## 15. 下一步建议
在开始 H5 开发前，建议继续产出两份文档：

1. **H5 我的待办接口清单**
   - 需要哪些 API
   - 每个 API 承担什么功能

2. **H5 我的待办页面原型说明**
   - 列表页
   - 详情页
   - 子任务编辑页
   - 管理式任务详情表现

这样就可以直接进入独立项目搭建阶段。
