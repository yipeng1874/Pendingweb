# 临时任务-管理式开发者文档

## 1. 文档范围
本文档说明临时任务中的“管理式任务（MANAGER）”实现口径，覆盖：
- 业务定义
- 发布与预览
- 主体生成规则
- 我的待办与临时任务看板
- 无管理组织处理
- 子任务完成人记录
- 已知风险与维护建议

不包含：
- 触达式任务（ACCOUNT）
- 主播式任务（ANCHOR）
- 日常任务（DAILY）

---

## 2. 业务定义

### 2.1 管理式任务是什么
管理式任务对应临时任务模式：
- `temporaryMode = "MANAGER"`

其核心特征是：
- 面向管理账号投放
- 完成主体不是个人，而是组织主体
- 同一组织下多个管理身份共享一条主体任务记录
- 任一可见管理身份完成子任务或提交，都会推进该组织主体进度

### 2.2 管理式任务不面向主播
管理式任务固定只触达管理账号：
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

不会触达：
- `ANCHOR`

### 2.3 同一账号多组织多主体
如果一个账号同时管理多个组织，则会按多个组织分别生成主体任务。例如：
- 同一账号同时是两个厅的管理
- 则会看到两条 `ORG:<orgId>` 主体任务

---

## 3. 主体模型

### 3.1 完成主体是组织
管理式任务的主体不是 `USER`，而是 `ORG`。

典型主体：
- 基地主体
- 团队主体
- 厅主体

主体 key 形如：
- `ORG:<orgId>`

### 3.2 可见身份
一个组织主体下可以挂多个可见管理身份，例如：
- 同一个厅下有两个 `HALL_MANAGER`
- 两人都能看到并操作同一条主体记录

### 3.3 共享完成
管理式任务是共享主体进度：
- 任一可见管理身份提交
- 该组织主体即可进入已完成 / 进行中等状态

---

## 4. 发布链路

### 4.1 前端入口
相关页面：
- `frontend/src/pages/tasks/manage/TemporaryIssuePage.tsx`
- `frontend/src/pages/tasks/manage/components/TemporaryIssuePanel.tsx`

### 4.2 模式选择
管理式任务在第二步选择：
- `mode = "MANAGER"`

页面支持三个主体级别：
- `BASE`
- `TEAM`
- `HALL`

### 4.3 角色映射
当前固定映射为：
- `BASE -> BASE_ADMIN`
- `TEAM -> TEAM_ADMIN`
- `HALL -> HALL_MANAGER`

前端常量：
- `directManagerRoleByOrgType`

### 4.4 组织圈定
管理式任务第三步需要圈定目标组织。

规则：
- 只能在当前管理范围内选择组织
- 选择的组织类型由 `subjectOrgType` 决定
- 团队级管理式任务只能选团队
- 厅级管理式任务只能选厅

---

## 5. 后端标准化规则

### 5.1 草稿标准化
后端在 `normalizeTemporaryDraftInput` 中会把管理式任务标准化为：
- `temporaryMode = "MANAGER"`
- `temporarySubjectOrgType = BASE / TEAM / HALL`
- `targetRoleCodes = [对应管理角色]`

### 5.2 角色确定
如果 `subjectOrgType` 为：
- `BASE` -> `BASE_ADMIN`
- `TEAM` -> `TEAM_ADMIN`
- `HALL` -> `HALL_MANAGER`

---

## 6. 主体生成规则

### 6.1 主体来源
管理式任务最终主体生成逻辑位于：
- `backend/src/modules/task/assignment/temporary-assignment.utils.ts`

### 6.2 生成步骤
1. 从 assignment target 取出目标 path
2. 查找目标 path 下匹配角色的活跃身份
3. 按 `temporarySubjectOrgType` 找出对应组织主体
4. 对每个组织主体，收集与该组织 path 精确对应的管理身份
5. 只有存在可见管理身份时，才生成 `ORG:<orgId>` 主体

### 6.3 关键行为
如果某个组织下没有命中任何管理身份：
- **不会生成主体**
- **不会生成 taskRecord**
- **不会进入我的待办**
- **不会进入临时任务看板**

这是当前已确认口径。

---

## 7. 无管理组织处理

### 7.1 现状
以前系统会静默跳过无管理组织。

### 7.2 当前增强
现在第四步预览会明确展示：
- 哪些组织当前无可承接管理账号
- 本次发布不会为这些组织生成主体任务

前端位置：
- `TemporaryIssuePanel.tsx` 第四步预览区

后端新增字段：
- `missingManagerOrgs`

结构示例：
- `orgId`
- `orgName`
- `orgType`

### 7.3 厅额外信息
当无管理组织是 `HALL` 时，预览会额外显示：
- 抖音号 `douyinNo`
- 抖音 UID `douyinUid`

若未登记，则显示“未登记”。

---

## 8. 预览与发布

### 8.1 预览数据
管理式任务第四步预览会返回：
- `subjectCount`
- `orgSubjectCount`
- `visibleIdentityCount`
- `subjectSummaries`
- `missingManagerOrgs`

### 8.2 预览含义
- `subjectCount` 表示实际将生成的主体数
- 它不等于你选择的组织数
- 如果存在无管理组织，它们不会计入主体数

### 8.3 正式发布
正式发布时，系统会根据已生成的组织主体创建：
- `taskRecord`
- `visibleIdentityLinks`

---

## 9. taskRecord 与 itemRecord 设计

### 9.1 record 级别
管理式任务的主体 record：
- `subjectType = "ORG"`
- `subjectOrgId = 组织 id`
- `subjectOrgType = BASE / TEAM / HALL`

### 9.2 item 级别完成人
为了支持“这一项是谁做的”，当前已给 `TaskItemRecord` 增加：
- `completedByUserId`
- `completedByIdentityId`

Prisma Schema 文件：
- `backend/prisma/schema.prisma`

### 9.3 写入规则
当管理式任务某个子任务被完成时：
- 当前提交身份会写入该 itemRecord 的完成人字段
- 用于后续在看板详情 / 我的待办里展示“完成人”

---

## 10. 我的待办展示

### 10.1 待办主体
管理式任务在“我的待办”中展示的是组织主体任务。

### 10.2 展示重点
当前展示重点应包括：
- 子任务标题
- 完成状态
- 当前结果
- 完成人
- 完成时间

### 10.3 正确理解
对于管理式任务，其他管理进入后重点不是继续补填，而是知道：
- 这一项是不是已完成
- 是谁完成的
- 结果是什么

---

## 11. 临时任务看板

### 11.1 看板来源
临时任务看板展示的是：
- 已经生成出来的主体 record

不是：
- 单纯被发布页选中的组织列表

### 11.2 影响
如果某个组织无管理：
- 不会生成 record
- 就不会出现在临时任务看板中

### 11.3 管理式详情展示重点
管理式任务详情应重点展示：
- 主体名称
- 主体状态
- 完成进度
- 发布人
- 最后提交信息
- 每个子任务的完成人
- 每个子任务的当前结果

## 13. 已确认业务口径

1. 管理式任务只触达管理账号，不触达主播
2. 管理式任务主体是组织，不是用户
3. 同一组织可有多个可见管理身份，共享一条主体任务
4. 某组织无管理身份时，不生成主体 record
5. 无管理组织不会进入临时任务看板
6. 第四步预览必须能明确看到具体哪些组织无管理
7. 对于管理式子任务，重点是展示“谁完成了该项”，而不是强调协同补填流水

---

## 14. 风险与注意事项

### 14.1 静默漏发风险
如果不看第四步预览提示，发布人可能误以为：
- 所有选中组织都发到了

实际上：
- 无管理组织会被跳过

### 14.2 组织 path / org 归属要求一致
管理式任务匹配管理身份时，依赖：
- `identity.org.path`
- `identity.scopePath`
- `subjectOrg.path`

这些数据必须一致，否则可能出现组织存在管理但未命中。

### 14.3 item 完成人只对新写入数据生效
新增的 `completedByUserId / completedByIdentityId` 只会在字段落库之后，对后续写入生效。
历史旧数据不会自动补全。

---

## 15. 推荐排查顺序
如果管理式任务出现“组织没发到 / 看板没看到 / 子任务看不到完成人”，建议按以下顺序排查：

1. 发布第四步 `missingManagerOrgs` 是否命中该组织
2. 该组织是否存在对应角色的活跃身份
3. 身份的 `org.path / scopePath` 是否与组织 path 一致
4. 是否成功生成 `taskRecord(subjectType = ORG)`
5. 子任务完成后 `task_item_records` 是否写入：
   - `completed_by_user_id`
   - `completed_by_identity_id`
6. 前端详情页 / 待办页是否正确渲染完成人

---

## 16. 面向后续维护者的建议

1. 不要把管理式任务误改成按用户主体完成
2. 不要把“无管理组织”重新变回静默跳过且无提示
3. 不要把 item 级完成人逻辑退化成只看 record 级最后提交人
4. 如果未来需要更强审计能力，可再增加“子任务修改历史”，但不要替代“完成人”这一主语义

---

## 17. 总结
管理式任务的本质是：
- **按组织主体协同完成**
- **按管理身份承接**
- **按子任务记录具体完成人**

当前已形成的正确闭环是：
1. 发布页选择组织
2. 第四步预览提示无管理组织
3. 正式发布只为有管理的组织生成主体 record
4. 我的待办 / 临时任务看板按组织主体展示
5. 子任务层面可识别“是谁完成了这一项”
