# 临时任务-二次通知 开发者文档

## 1. 文档范围

本文档仅说明"临时任务"中的"二次通知（Pre-Deadline Reconfirm）"功能的开发口径，覆盖：
- 业务定义与设计决策
- 数据库变更
- 后端 API / 定时任务
- 前端交互与 UI 组件
- 生产环境部署步骤
- 风险与注意事项

不包含：
- 催办通知（`TemporaryNotifySchedule` 现有机制）
- 日常任务
- 厅管日常任务

---

## 2. 业务定义

### 2.1 二次通知是什么

发布者在创建临时任务时，可选开启「二次通知」功能。开启后：

```
发布日（7.18）→ 员工陆续完成 → 截止前一天（7.22）
  ├─ 系统自动扫描：status="submitted" && reconfirmStatus IS NULL
  ├─ 批量设置：reconfirmStatus = "pending" + reconfirmSentAt = NOW
  ├─ 发送飞书通知："「任务名」明天截止，请回顾确认你已提交的内容"
  └─ 我的待办：已完成任务卡片标记"今日重点关注"，可点击确认
```

### 2.2 与催办通知的差异

| 维度 | 催办通知（已有） | 二次通知（新增） |
|------|-----------------|------------------|
| 触发人群 | 未完成者（pending / in_progress） | 已完成者（submitted） |
| 触发时机 | 按档位梯度多次触发 | 仅截止前 1 天触发 1 次 |
| 开关位置 | 自动催办弹窗（用户级配置） | 任务发布第四步（任务级开关） |
| 通知渠道 | 飞书文字消息 | 飞书文字消息 + 我的待办确认按钮 |
| 是否可确认 | 无（只是提醒） | 有（点击确认即完成） |

### 2.3 核心设计决策

| 决策 | 说明 |
|------|------|
| **不新建表、不新建记录** | 复用 `taskRecord`，新增三个字段标记二次通知状态 |
| **确认不可修改数据** | 员工点击"确认内容无误"后，只改 `reconfirmStatus` 和 `reconfirmConfirmedAt`，不碰 answers / itemRecords |
| **幂等更新** | 定时 tick 用 `updateMany WHERE reconfirmStatus IS NULL` 保证同一条记录不重复投递 |
| **我的待办持续展示** | 已完成的任务原本就一直在"我的待办"列表中，二次通知仅叠加"今日重点关注"标签和确认按钮，不新增卡片 |
| **H5 暂时跳过** | 当前 H5 端不做确认按钮的适配，仅做兼容渲染（不崩溃即可） |

## 3. 数据库变更

### 3.1 新增字段

#### task_assignments

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `pre_deadline_confirm_enabled` | `BOOLEAN` | `false` | 发布者是否开启二次通知 |

#### task_records

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `reconfirm_status` | `VARCHAR(191)` | `NULL` | `NULL` = 未触发 / `pending` = 待确认 / `confirmed` = 已确认 |
| `reconfirm_sent_at` | `DATETIME(3)` | `NULL` | 定时任务触发时间 |
| `reconfirm_confirmed_at` | `DATETIME(3)` | `NULL` | 员工点击确认时间 |

### 3.2 Migration SQL

```sql
-- task_assignments
ALTER TABLE `task_assignments`
  ADD COLUMN `pre_deadline_confirm_enabled` BOOLEAN NOT NULL DEFAULT false;

-- task_records
ALTER TABLE `task_records`
  ADD COLUMN `reconfirm_status` VARCHAR(191) NULL,
  ADD COLUMN `reconfirm_sent_at` DATETIME(3) NULL,
  ADD COLUMN `reconfirm_confirmed_at` DATETIME(3) NULL;
```

### 3.3 防重机制

二次通知的防重复用现有 `temporary_notify_trigger_logs` 表，使用前缀 `reconfirm-` 区分：

```
slotKey 格式：
  催办通知：  "2026-07-22 13"
  二次通知：  "reconfirm-2026-07-22"

unique key: (assignment_id, slot_key)  → 天然防重
```

### 3.4 数据状态机

```
taskRecord.status = "submitted"
  reconfirmStatus = NULL
  ↓ 截止前 1 天 tick 触发
taskRecord.status = "submitted"
  reconfirmStatus = "pending"          ← 我的待办显示"今日重点关注" + 确认按钮
  reconfirmSentAt = <tick时间>
  ↓ 员工点击确认
taskRecord.status = "submitted"
  reconfirmStatus = "confirmed"        ← "今日重点关注"消失，显示"已确认"标签
  reconfirmConfirmedAt = <确认时间>
```

| `record.controller.ts` | 新增接口 | `reconfirmRecord` 处理确认请求，新增错误码 `RECONFIRM_NOT_PENDING` / `RECONFIRM_FORBIDDEN` |
| `record.service.ts` | 新增方法 | `reconfirmRecord` — 校验 `reconfirmStatus === "pending"` → 设为 `confirmed` |
| `record.routes.ts` | 新增路由 | `POST /tasks/my-records/:id/reconfirm` |
| `temporary-notify-schedule.routes.ts` | 核心逻辑 | `processPreDeadlineReconfirmIfNeeded` + `buildTemporaryReconfirmOpenIds` |

### 4.2 新增 API

#### POST `/api/tasks/my-records/:id/reconfirm`

**权限**：`task:record:submit`（与提交任务相同）

**请求体**：无

**成功响应**：
```json
{
  "success": true,
  "data": {
    "id": "record-id",
    "reconfirmStatus": "confirmed",
    "reconfirmConfirmedAt": "2026-07-22T12:00:00.000Z",
    ...
  }
}
```

**错误码**：
| 错误码 | HTTP | 含义 |
|--------|------|------|
| `RECONFIRM_NOT_PENDING` | 400 | 当前任务无需二次确认或已完成确认 |
| `RECONFIRM_FORBIDDEN` | 403 | 当前身份无权确认该任务 |
| `RECORD_NOT_FOUND` | 404 | 执行记录不存在 |

### 4.3 定时 tick 逻辑

仍在现有 `runTemporaryNotifyScheduleTick`（每分钟调用）中执行，在 `buildTemporaryNotifyOpenIds` 之前插入：

```
for assignment in assignments:
  daysLeft = ceil((deadline - now) / 86400000)
  
  ┌─ 二次投放分支 ──────────────────────────────────┐
  │ if assignment.preDeadlineConfirmEnabled          │
  │    && daysLeft === 1:                             │
  │   ① 重新查询 assignment status + isActive         │
  │      → 已关闭则跳过 skip                          │
  │   ② 检查 triggerLog(slotKey="reconfirm-<日期>")  │
  │      → 已有则跳过 skip（防重）                      │
  │   ③ updateMany:                                  │
  │      WHERE status = "submitted"                   │
  │        AND reconfirmStatus IS NULL                │
  │      SET reconfirmStatus = "pending",            │
  │          reconfirmSentAt = NOW()                  │
  │   ④ 发送飞书通知（复用 sendFeishuBatchMessage）    │
  │   ⑤ 写 triggerLog（即使失败也写，避免重复扫描）    │
  └─────────────────────────────────────────────────┘
  
  ┌─ 原有催办分支（不变）────────────────────────────┐
  │ dailyCount = getTierDailyCount(...)              │
  │ ...                                              │
  └─────────────────────────────────────────────────┘
```

关键函数：
- `processPreDeadlineReconfirmIfNeeded()` — 二次投放主逻辑
- `buildTemporaryReconfirmOpenIds()` — 查询二次投放受众的飞书 openId

### 6.5 回滚方案

本次变更均为增量修改，逻辑开关使用布尔型默认值 `false`，不影响现有流程。如有问题：

1. **功能回滚**：保持数据库字段不动（不影响任何查询），前端注释掉 checkbox 展示即可
2. **全量回滚**：执行以下 SQL 删除新增字段

```sql
ALTER TABLE `task_assignments` DROP COLUMN `pre_deadline_confirm_enabled`;
ALTER TABLE `task_records`
  DROP COLUMN `reconfirm_status`,
  DROP COLUMN `reconfirm_sent_at`,
  DROP COLUMN `reconfirm_confirmed_at`;
```

然后重新构建前端、重新生成 Prisma Client、重启后端服务。

## 7. 注意事项与已知限制

### 7.1 截止时间修改

如果在二次通知触发后发布者修改了 `deadlineAt`：
- 已标记为 `pending` 的 record 不受影响（员工仍可确认）
- `deadlineAt` 的变动由现有 `updateAssignment` 逻辑处理

### 7.2 任务提前关闭

定时 tick 在触发二次投放前会重新校验 assignment 的 `status` 和 `isActive`。如果任务已被关闭，不会触发二次通知。

### 7.3 H5 适配

当前 H5 端（`todo-h5/`）不会：
- 识别 `reconfirmStatus` 字段
- 渲染"今日重点关注"标签或确认按钮

H5 端的 `reconfirmStatus=pending` 记录将以普通"已完成"卡片展示，不会崩溃。H5 的确认功能计划在后续迭代中补齐。

### 7.4 并发安全

- `updateMany WHERE reconfirmStatus IS NULL` 保证同一条记录不会被重复设置为 `pending`
- `temporaryNotifyTriggerLog`（unique on `assignmentId + slotKey`）保证同一任务同一日期不会重复发送飞书通知
- 定时任务仅在每小时前 5 分钟执行（`getBeijingMinute(now) >= 5` 直接返回），进一步降低并发概率

### 7.5 性能考量

- 批量 `updateMany` 一次 SQL 完成所有 `submitted` 记录的 `reconfirmStatus` 切换，不会逐条遍历
- 飞书批量发送已有现成 `sendFeishuBatchMessage` 实现，无需额外处理限流
