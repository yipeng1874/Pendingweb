## 2. 功能概述

“主播账号管理”模块用于管理**审核通过后的主播账号与主播档案**，核心目标包括：

1. 按组织范围查看主播账号
2. 搜索主播昵称 / 抖音号 / UID / 绑定手机号
3. 查看主播档案详情
4. 编辑主播资料（昵称、抖音号、抖音 UID、归属厅）
5. 启用 / 停用主播账号
6. 删除主播档案
7. 对已绑定账号执行密码重置
8. 将注册审核通过的数据与主播档案建立映射关系

该模块不负责：
- 主播注册申请提交
- 注册审核动作本身
- 组织树维护本身
- 管理身份授权
- 登录鉴权本身

但它依赖以下模块输出：
- 组织树
- 用户账号
- 用户身份体系
- 主播注册审核
- 组织账号管理

## 3. 业务定位与当前规则

### 3.1 模块定位

该页面是“主播维度的账号后台”，和“组织账号管理”的管理对象不同：

- 组织账号管理：管理后台管理员身份
- 主播账号管理：管理主播档案与主播账号状态

### 3.2 当前查询规则

当前页面采用：

- 左侧组织树决定右侧查询范围
- 右侧列表按当前组织范围分页加载
- 每页默认 20 条
- 搜索和状态筛选在后端执行
- 详情抽屉独立请求，不再完全依赖列表行数据

### 3.3 当前组织范围语义

和“组织账号管理”的直属管理员模式不同，“主播账号管理”当前是**组织范围模式**：

- 选基地：看基地范围内的主播账号
- 选团队：看团队范围内的主播账号
- 选厅：看厅范围内的主播账号
- **不选组织：右侧不加载任何主播账号数据**

后端实现方式是：
- 先根据选中组织 `orgId` 查到该组织的 `path`
- 再按 `hallOrg.path startsWith selectedOrg.path` 过滤主播档案
- 前端 `useAnchorAccounts()` 在 `selectedOrgId` 为空时直接清空列表并停止请求

## 5. 页面功能拆解

### 5.1 页面入口

主播账号管理页面位于“主播管理”或等价业务分组下：

- 前端页面组件：`AnchorAccountsPage`
- 常见路由路径：通常挂在 `/anchors/accounts` 或等价路径（以实际路由配置为准）

### 5.2 页面组成

页面分为两块：

#### 5.2.1 左侧组织树

用途：
- 选择 HQ / BASE / TEAM / HALL 中的某个节点
- 决定右侧主播账号列表查询范围
- 懒加载展开子组织

#### 5.2.2 右侧主播账号区域

用途：
- 展示当前组织范围下的主播账号分页列表
- 支持搜索与状态筛选
- 打开详情抽屉
- 执行启用 / 停用 / 删除 / 重置密码 / 编辑资料

### 5.3 当前筛选项

主播账号管理页当前保留的筛选项：

- 关键词搜索：昵称 / 抖音号 / UID / 手机号
- 状态筛选：`bound / unbound / inactive`
- 手动刷新

说明：
- 右侧原先“归属厅下拉框”已移除
- 组织范围统一由左侧组织树控制

这条交互规则非常重要：

> 左侧组织树是唯一的组织范围控制入口，右侧不再保留重复的组织筛选器。

## 6. 页面权限与角色矩阵

### 6.1 页面访问权限

前端页面一般要求：
- 已登录
- 已选择身份
- 当前身份具备 `anchor:view`

### 6.2 页面内操作权限

页面内部根据权限拆分为：

#### 只读用户
满足：
- 有 `anchor:view`
- 但没有 `anchor:profile:create` 和 `anchor:profile:bind`

能力：
- 查看列表
- 搜索 / 翻页
- 打开抽屉查看详情
- 不能编辑、启停、删除、重置密码

#### 可管理用户
满足任一：
- 权限数组包含 `*`
- 或包含 `anchor:profile:create`
- 或包含 `anchor:profile:bind`

能力：
- 查看 + 编辑主播资料
- 启停主播账号
- 删除主播档案
- 重置密码

### 6.3 后端接口权限矩阵

| 接口 | 权限 | 说明 |
|---|---|---|
| GET `/anchors/org-children` | `org:view` | 懒加载组织树 |
| GET `/anchors/profiles` | `anchor:view` | 主播账号分页列表 |
| GET `/anchors/profiles/:id` | `anchor:view` | 主播账号详情 |
| POST `/anchors/profiles` | `anchor:profile:create` | 新建主播档案 |
| PATCH `/anchors/profiles/:id` | `anchor:profile:create` | 编辑主播资料 |
| POST `/anchors/profiles/:id/disable` | `anchor:profile:bind` | 停用主播账号 |
| POST `/anchors/profiles/:id/enable` | `anchor:profile:bind` | 启用主播账号 |
| DELETE `/anchors/profiles/:id` | `anchor:profile:bind` | 删除主播档案 |
| POST `/accounts/:id/reset-password` | `account:create` | 重置用户密码（由主播页调用） |
| GET `/anchors/applications` | `anchor:registration:review` | 查询审核通过申请，用于补充账号关联信息 |

### 6.4 HALL_MANAGER 的特殊限制

当前 `useAnchorAccounts` 中存在：
- `canReviewApplications = currentIdentity?.roleCode !== "HALL_MANAGER"`

含义：
- 厅管理通常可以查看主播账号列表
- 但不会额外加载审批通过申请列表用于重置密码映射

这意味着：
- HALL_MANAGER 页面能力通常会偏只读或弱管理
- 具体是否能操作，还要看它实际持有的权限码

## 7. 前端实现说明

### 7.1 页面核心状态

`useAnchorAccounts` 维护的核心状态包括：

1. `orgs`：当前已加载的组织节点平铺列表
2. `selectedOrgId`：当前左树选中组织
3. `anchors`：当前页主播档案列表
4. `apps`：审核通过申请列表（用于绑定用户信息、重置密码等）
5. `keyword`：搜索词
6. `status`：状态筛选
7. `anchorPage`：当前页码
8. `anchorTotal`：总条数
9. `editing`：当前抽屉编辑中的主播对象
10. `message / error`：页面提示消息
11. `loadedOrgIds`：已懒加载过的组织节点集合

### 7.2 页面初始化流程

页面初始化时：

1. `useEffect` 监听当前身份变化
2. 调用 `loadRootOrgs()`
3. 请求 `/anchors/org-children`
4. 加载首屏可见组织节点
5. 默认只展开身份路径，不再默认选中任何组织
6. 右侧主播账号列表保持空态，等待用户先选择左侧组织

### 7.3 当前组织选择规则

当前逻辑已经改为“先选左树再加载右侧”：

1. 首次进入页面不自动选中任何组织
2. 用户点击左侧组织节点后，才更新 `selectedOrgId`
3. `selectedOrgId` 为空时，`useAnchorAccounts()` 会直接清空 `anchors / anchorTotal / apps`
4. 只有选中组织后，才会请求 `/anchors/profiles` 和（非 `HALL_MANAGER` 的）审核通过申请补充数据

这样可以保证：
- `HQ_ADMIN / BASE_ADMIN / TEAM_ADMIN` 首次进入时不会误拉整段组织范围数据
- `HALL_MANAGER` 进入时也会先看到自己的厅节点，再手动选择后加载右侧内容

### 7.4 左树展开逻辑

点击左侧组织树折叠/展开时：

1. 若该节点未加载过，调用 `loadOrgChildren(orgId)`
2. 请求 `/anchors/org-children?parentId=xxx`
3. 将子节点合并进 `orgs`
4. 前端再通过 `buildOrgTree(orgs)` 重构树结构
5. 懒加载只负责补充节点数据，不再反向覆盖用户当前的折叠状态

点击左侧某个组织节点用于“选中”时：

1. 会先确保该节点的祖先链路处于展开状态
2. 再更新当前 `selectedOrgId`
3. 最后按既有逻辑加载该组织范围的数据

这样可以避免用户点到某个厅或团队后，左树因为异步数据补全而出现“自动折叠”的错觉。

### 7.5 右侧列表加载逻辑

调用 `load()` 时：

1. 组装查询参数：
   - `page`
   - `pageSize`
   - `keyword`
   - `orgId`
   - `status`
2. 调用 `/anchors/profiles`
3. 后端按组织范围 + 搜索条件 + 状态分页返回
4. 更新：
   - `anchors`
   - `anchorTotal`

### 7.6 详情抽屉逻辑

点击表格行时：

1. 先根据当前列表项打开抽屉
2. 再调用 `/anchors/profiles/:id`
3. 详情返回后用详情对象覆盖 `editing` 和 `activeAnchorDetail`

这样做的意义：
- 列表接口保持轻量
- 抽屉可以展示更完整信息
- 后续继续扩展详情字段更容易维护

### 7.7 操作逻辑

页面支持以下操作：

#### 保存主播资料
- 调用 `PATCH /anchors/profiles/:id`

#### 停用主播账号
- 调用 `POST /anchors/profiles/:id/disable`

#### 启用主播账号
- 调用 `POST /anchors/profiles/:id/enable`

#### 删除主播档案
- 调用 `DELETE /anchors/profiles/:id`

#### 重置密码
- 调用 `POST /accounts/:userId/reset-password`
- 默认密码重置为 `123456`

### 7.8 提示反馈机制

页面统一使用：
- `Toast` 成功提示
- `Toast` 错误提示
- 删除 / 重置密码等高风险操作使用浏览器 `confirm`

## 8. 后端处理逻辑

### 8.1 模块总入口

后端主播账号管理统一定义在：
- `backend/src/modules/anchor/routes.ts`

统一挂载：
1. `authRequired`
2. `identityRequired`
3. 接口级别 `permissionRequired`

### 8.2 组织范围控制

主播模块最关键的逻辑之一是：

- 当前身份 `scopePath` 控制可见范围
- 页面左树选中的 `orgId` 进一步收窄范围

核心处理方式：
1. 根据 `orgId` 查到组织 `path`
2. 生成 `effectiveScopePath = selectedOrg.path || currentIdentity.scopePath`
3. 用 `hallOrg.path startsWith effectiveScopePath` 过滤主播档案

### 8.3 组织树懒加载接口

后端 `getScopedOrgChildren()` 的职责：

- 根据当前身份与 `parentId` 返回下一层组织节点
- 支持 HQ / BASE / TEAM / HALL 分层懒加载
- 兼容开发管理员与总部管理员首屏加载逻辑

### 8.4 主播档案列表接口

`getProfiles()` 的职责：

1. 处理 keyword
2. 处理 status
3. 处理 orgId -> path 范围过滤
4. 使用 `skip/take` 分页
5. 关联 `hallOrg` 与 `identities`
6. 将绑定用户信息整合成 `boundUser`

### 8.5 详情接口

`getProfileDetail()` 的职责：
- 读取单个主播档案
- 校验当前身份范围权限
- 补充归属厅与绑定用户信息

### 8.6 启用 / 停用逻辑

`toggleProfileStatus()` 的职责：

#### 启用时
- 若已绑定用户，启用该用户账号
- 恢复主播身份 `UserIdentity`
- 恢复 `scopePath`
- 更新档案状态为 `bound` 或 `unbound`

#### 停用时
- 停用主播身份
- 写入 `expiredAt`
- 若存在绑定用户，则禁用该用户账号
- 更新主播档案状态为 `inactive`

### 8.7 删除逻辑

`deleteProfile()` 的职责：

1. 删除主播身份记录 `ANCHOR`
2. 删除与该绑定用户相关的注册申请记录
3. 若存在绑定用户，则禁用该用户
4. 删除主播档案

控制器层还有一个额外保护：
- 若该主播账号仍持有非主播类管理权限，禁止删除
- 需要先到“组织账号管理”中处理管理权限后再删

## 9. API 接口释义

以下接口均省略统一网关前缀 `/api`。

### 9.1 GET `/anchors/org-children`

#### 用途
按父节点懒加载组织树下一层节点。

#### 查询参数
| 参数 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| parentId | string | 否 | 父组织 ID；为空时按当前身份加载首屏组织 |
| includeVirtual | boolean | 否 | 是否包含虚拟组织 |

#### 权限
- `org:view`

#### 返回
- 当前身份可见范围内的下一层组织列表

#### 首屏规则
- `DEV_ADMIN / HQ_ADMIN`：返回 HQ / BASE
- `BASE_ADMIN`：返回 TEAM
- `TEAM_ADMIN`：返回 HALL
- `HALL_MANAGER`：返回当前厅

---

### 9.2 GET `/anchors/profiles`

#### 用途
分页查询主播账号列表。

#### 查询参数
| 参数 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| orgId | string | 否 | 当前选中组织 ID，用于确定范围 |
| hallOrgId | string | 否 | 额外按厅精确筛选（当前页面已不再主用） |
| keyword | string | 否 | 昵称 / 抖音号 / UID / 手机号模糊搜索 |
| status | string | 否 | `bound / unbound / inactive` |
| page | number | 否 | 页码，默认 1 |
| pageSize | number | 否 | 每页条数，默认 20 |

#### 权限
- `anchor:view`

#### 返回
```text
{
  items: Anchor[]
  total: number
  page: number
  pageSize: number
}
```

#### 备注
当前页面实际主要使用：
- `orgId`
- `keyword`
- `status`
- `page`
- `pageSize`

---

### 9.3 GET `/anchors/profiles/:id`

#### 用途
获取单个主播档案详情，供抽屉独立加载。

#### 权限
- `anchor:view`

#### 失败情况
- 档案不存在
- 当前身份无权查看该档案

---

### 9.4 PATCH `/anchors/profiles/:id`

#### 用途
更新主播资料。

#### 可更新字段
| 字段 | 说明 |
|---|---|
| nickname | 主播昵称 |
| douyinNo | 抖音号 |
| douyinUid | 抖音 UID |
| hallOrgId | 归属厅 |

#### 权限
- `anchor:profile:create`

---

### 9.5 POST `/anchors/profiles/:id/disable`

#### 用途
停用主播账号 / 档案。

#### 权限
- `anchor:profile:bind`

#### 效果
- 停用主播身份
- 绑定用户置为 disabled
- 档案状态改为 `inactive`

---

### 9.6 POST `/anchors/profiles/:id/enable`

#### 用途
启用主播账号 / 档案。

#### 权限
- `anchor:profile:bind`

#### 效果
- 恢复主播身份
- 如有绑定用户则恢复用户状态
- 档案状态恢复为 `bound` 或 `unbound`

---

### 9.7 DELETE `/anchors/profiles/:id`

#### 用途
删除主播档案。

#### 权限
- `anchor:profile:bind`

#### 保护逻辑

## 11. 数据加载（页面懒加载分级传输）与后端处理逻辑

> 这里的“懒加载分级传输”不是 React 代码分包，而是页面按使用阶段逐步拉取数据、避免一次性返回全量明细的分层传输策略。

### 11.1 当前总体策略

主播账号管理模块当前采用以下加载顺序：

1. 先加载左侧首屏组织节点
2. 用户展开组织节点时再加载下一级组织
3. 用户选中组织后，再加载该组织范围下的主播账号列表第一页
4. 用户修改筛选条件时重新请求分页列表
5. 用户点击某一行后，再独立拉取详情
6. 仅当当前身份允许时，才额外加载审核通过申请列表，用于补充绑定账号信息和密码重置能力

### 11.2 第一层：组织树首屏加载

接口：
- `GET /anchors/org-children`

特点：
- 首屏只返回当前身份需要看到的第一层节点
- 避免一次性全量加载 HQ / BASE / TEAM / HALL 整棵树

### 11.3 第二层：组织树展开懒加载

接口：
- `GET /anchors/org-children?parentId=...`

特点：
- 只在用户展开节点时请求
- 子节点合并到 `orgs` 后重建树
- 用 `loadedOrgIds` 防止重复请求
- 不再因为 `orgs` 补数而重新按默认展开规则覆盖用户已操作的树状态

### 11.4 第三层：主播账号分页列表

接口：
- `GET /anchors/profiles`

特点：
- 按组织范围分页
- 每页默认 20 条
- 搜索词和状态筛选都在后端执行
- 不再依赖前端对当前页做伪搜索

### 11.5 第四层：详情抽屉按需加载

接口：
- `GET /anchors/profiles/:id`

特点：
- 点击行才请求
- 列表和详情分离
- 列表接口可保持更轻
- 抽屉可以独立扩展更多字段

### 11.6 第五层：审核通过申请补充信息

接口：
- `GET /anchors/applications?status=approved&page=1&pageSize=100&targetOrgId=...`

用途：
- 构建 `detailsByUserId`
- 用于给绑定账号提供更多上下文
- 用于重置密码按钮匹配 `userId`

注意：
- 这部分不是主列表数据来源
- 只有在已选组织时才会加载，并跟随当前 `selectedOrgId` 收缩范围
- HALL_MANAGER 不会走这条加载链路

### 11.7 后端处理逻辑总结

后端对数据传输做了以下处理：

#### 11.7.1 范围收缩
- 身份 `scopePath` 限制可见组织和主播档案
- 选中组织 `orgId` 再进一步缩小范围

#### 11.7.2 条件搜索
- `keyword`
- `status`
- `orgId`
- `page`
- `pageSize`

#### 11.7.3 分页传输
- `normalizePagination` 统一处理分页参数
- 使用 `skip / take` 控制数据库返回量

#### 11.7.4 详情与列表分离
- 列表接口负责分页摘要数据
- 详情接口负责单条扩展信息

这是当前主播账号管理性能优化的核心落地点之一。

## 13. 当前风险点与改进建议

### 13.1 当前已知设计特点

1. 左侧组织树已懒加载，不再一次性拉完整树
2. 列表搜索和分页在后端执行
3. 抽屉详情已独立接口化
4. 删除主播档案会被管理身份保护规则拦截
5. 页面右侧组织筛选器已收敛为“左树决定范围”

### 13.2 当前仍可优化点

1. **组织树节点级 loading 还不细**
   - 当前主要是展开时触发请求，但节点级反馈还可以更强

2. **抽屉详情可增加 loading 态与缓存**
   - 当前已独立请求，但可进一步减少重复点击请求

3. **审核通过申请补充信息仍是固定加载 100 条**
   - 数据量继续增大时可能要继续拆分或分页

4. **密码重置能力依赖审核通过申请列表映射**
   - 后续可考虑更直接的账号详情绑定关系接口

5. **删除 / 启停 / 编辑缺少审计日志**
   - 建议后续补齐

### 13.3 推荐改造优先级

高优先级：
1. 关键操作审计日志化
2. 重置密码关联逻辑进一步明确化
3. 详情加载态与缓存增强

中优先级：
1. 组织树节点级 loading 提示
2. 抽屉字段扩展与结构优化
3. 审核通过申请补充信息改为按需查询

低优先级：
1. 列表列配置化
2. 批量启停 / 批量迁移归属厅
3. 操作历史时间线

### 14.6 组织树点击节点自动折叠问题修复

- 用户点击节点的语义现在被明确区分为“选中节点”，不再隐式触发“按默认规则重置树折叠状态”

本次修复覆盖页面：
- `AnchorAccountsPage.tsx`
- `AnchorReviewsPage.tsx`

修复后的行为规则：
1. 首次进入页面时，仍允许按身份范围设置默认展开状态
2. 用户手动展开 / 折叠后的树状态，应优先于默认展开规则
3. 点击某个组织节点时，会主动保证该节点的祖先链路处于展开状态
4. 懒加载子节点只负责补数据，不再覆盖用户当前折叠状态

这样做的产品意义是：
- 用户点击“某个厅”后，左树不会再看起来“自己收起来了”
- 组织定位更稳定
- 注册审核页与主播账号页的树交互保持一致

后续若继续改这两页的组织树逻辑，请重点注意：
- 不要在每次 `orgs` 数据变化后都重新全量计算折叠状态
- “默认展开”只适合作为初始化逻辑，不适合作为交互期逻辑
- 若继续引入新的懒加载层级，必须保证当前选中节点的祖先路径始终可见
