## 2. 功能概述

"主播账号管理"模块用于管理**审核通过后的主播账号与主播档案**，核心目标包括：

1. 按组织范围查看主播账号
2. 搜索主播昵称 / 抖音号 / UID / 绑定手机号
3. 查看主播档案详情
4. 编辑主播资料（昵称、抖音号、抖音 UID）
5. 通过专用迁移流程变更归属厅
6. 启用 / 停用主播账号
7. 删除主播档案
8. 对已绑定账号执行密码重置
9. 将注册审核通过的数据与主播档案建立映射关系
10. **异步导出主播账号为 CSV 文件（2026-06 新增）**
11. **区分当前身份 / 历史身份视图查询与导出（2026-06 新增）**

该模块不负责：
- 主播注册申请提交
- 注册审核动作本身
- 组织树维护本身
- 管理身份授权
- 登录鉴权本身

但它依赖以下模块输出：
- 组织树
- 用户账号
- 用户身份体系
- 主播注册审核
- 组织账号管理

## 3. 业务定位与当前规则

### 3.1 模块定位

该页面是"主播维度的账号后台"，和"组织账号管理"的管理对象不同：

- 组织账号管理：管理后台管理员身份
- 主播账号管理：管理主播档案与主播账号状态

### 3.2 当前查询规则

当前页面采用：

- 左侧组织树决定右侧查询范围
- 右侧列表按当前组织范围分页加载
- 每页默认 20 条
- 搜索和状态筛选在后端执行
- 详情抽屉独立请求，不再完全依赖列表行数据
- 页面默认进入"当前身份"视图
- 当前身份 / 历史身份采用互斥视图，不再在主列表中混排展示

### 3.3 当前 / 历史身份规则（2026-06 更新）

为降低运营心智压力，当前账号管理页已经调整为双视图模式：

- `current`：当前身份视图
- `history`：历史身份视图

规则如下：

#### 当前身份视图
- 默认进入页面时展示
- 用于日常运营、编辑、停用、迁移、重置密码等操作
- 默认隐藏迁移后封存的历史 `back` 档案

#### 历史身份视图
- 仅展示迁移后封存的历史主播身份
- 只用于历史归属、历史任务口径、迁移轨迹核对
- 详情抽屉前端强制只读
- 历史档案不可直接启用；如需回到原厅，必须重新发起迁移

#### 当前历史身份识别规则
当前代码里对"历史迁移档案"的识别是约定式实现，不是单独数据库字段：

- `status = inactive`
- 且昵称带有迁移后缀 `back-YYMMDD-HHmmssSSS`（兼容旧格式 `back-YYMMDD`）

注意：
- 这套规则目前在后端查询口径和前端展示口径中都已使用
- 它不是理想的长期建模方式，后续更推荐增加显式迁移流水或历史身份标记字段

### 3.4 迁移规则（2026-06 更新）

当前已不允许通过普通资料编辑直接修改 `hallOrgId`。

当前迁移规则为：

1. 普通资料编辑只允许修改：
   - 主播昵称
   - 抖音号
   - 抖音 UID
2. 变更归属厅必须调用专用迁移接口
3. 迁移时会：
   - 封存旧主播档案（改名为 `back-*` 历史身份）
   - 停用旧 `ANCHOR` 身份
   - 在目标厅创建新的主播档案
   - 为同一账号创建新的 active `ANCHOR` 身份
4. 历史任务保留在原厅，不应跟随迁移改归属
5. 回到原厅不是"启用旧 back 档案"，而是重新发起一次正式迁移

### 3.5 当前组织范围语义

和"组织账号管理"的直属管理员模式不同，"主播账号管理"当前是**组织范围模式**：

- 选基地：看基地范围内的主播账号
- 选团队：看团队范围内的主播账号
- 选厅：看厅范围内的主播账号
- **不选组织：右侧不加载任何主播账号数据**

后端实现方式是：
- 先根据选中组织 `orgId` 查到该组织的 `path`
- 再按 `hallOrg.path startsWith selectedOrg.path` 过滤主播档案
- 前端 `useAnchorAccounts()` 在 `selectedOrgId` 为空时直接清空列表并停止请求

## 5. 页面功能拆解

### 5.1 页面入口

主播账号管理页面位于"主播管理"或等价业务分组下：

- 前端页面组件：`AnchorAccountsPage`
- 常见路由路径：通常挂在 `/anchors/accounts` 或等价路径（以实际路由配置为准）

### 5.2 页面组成

页面分为两块：

#### 5.2.1 左侧组织树

用途：
- 选择 HQ / BASE / TEAM / HALL 中的某个节点
- 决定右侧主播账号列表查询范围
- 懒加载展开子组织

#### 5.2.2 右侧主播账号区域

用途：
- 展示当前组织范围下的主播账号分页列表
- 支持搜索与状态筛选
- 打开详情抽屉
- 执行启用 / 停用 / 删除 / 重置密码 / 编辑资料

### 5.3 当前筛选项

主播账号管理页当前保留的筛选项：

- 身份视图切换：`当前身份 / 历史身份`
- 关键词搜索：昵称 / 抖音号 / UID / 手机号
- 状态筛选：`bound / unbound / inactive`
- 手动刷新

说明：
- 右侧原先"归属厅下拉框"已移除
- 组织范围统一由左侧组织树控制

这条交互规则非常重要：

> 左侧组织树是唯一的组织范围控制入口，右侧不再保留重复的组织筛选器。

## 6. 页面权限与角色矩阵

### 6.1 页面访问权限

前端页面一般要求：
- 已登录
- 已选择身份
- 当前身份具备 `anchor:view`

### 6.2 页面内操作权限

页面内部根据权限拆分为：

#### 只读用户
满足：
- 有 `anchor:view`
- 但没有 `anchor:profile:create` 和 `anchor:profile:bind`

能力：
- 查看列表
- 搜索 / 翻页
- 打开抽屉查看详情
- 不能编辑、启停、删除、重置密码

#### 可管理用户
满足任一：
- 权限数组包含 `*`
- 或包含 `anchor:profile:create`
- 或包含 `anchor:profile:bind`

能力：
- 查看 + 编辑主播资料
- 启停主播账号
- 删除主播档案
- 重置密码

### 6.3 后端接口权限矩阵

| 接口 | 权限 | 说明 |
|---|---|---|
| GET `/anchors/org-children` | `org:view` | 懒加载组织树 |
| GET `/anchors/profiles` | `anchor:view` | 主播账号分页列表 |
| GET `/anchors/profiles/:id` | `anchor:view` | 主播账号详情 |
| POST `/anchors/profiles` | `anchor:profile:create` | 新建主播档案 |
| PATCH `/anchors/profiles/:id` | `anchor:profile:create` | 编辑主播资料 |
| POST `/anchors/profiles/:id/disable` | `anchor:profile:bind` | 停用主播账号 |
| POST `/anchors/profiles/:id/enable` | `anchor:profile:bind` | 启用主播账号 |
| DELETE `/anchors/profiles/:id` | `anchor:profile:bind` | 删除主播档案 |
| POST `/accounts/:id/reset-password` | `account:create` | 重置用户密码（由主播页调用） |
| GET `/anchors/applications` | `anchor:registration:review` | 查询审核通过申请，用于补充账号关联信息 |

### 6.4 HALL_MANAGER 的特殊限制

当前 `useAnchorAccounts` 中存在：
- `canReviewApplications = currentIdentity?.roleCode !== "HALL_MANAGER"`

含义：
- 厅管理通常可以查看主播账号列表
- 但不会额外加载审批通过申请列表用于重置密码映射

这意味着：
- HALL_MANAGER 页面能力通常会偏只读或弱管理
- 具体是否能操作，还要看它实际持有的权限码

## 7. 前端实现说明

### 7.1 页面核心状态

`useAnchorAccounts` 维护的核心状态包括：

1. `orgs`：当前已加载的组织节点平铺列表
2. `selectedOrgId`：当前左树选中组织
3. `anchors`：当前页主播档案列表
4. `apps`：审核通过申请列表（用于绑定用户信息、重置密码等）
5. `keyword`：搜索词
6. `status`：状态筛选
7. `viewMode`：身份视图模式（`current / history`）
8. `anchorPage`：当前页码
9. `anchorTotal`：总条数
10. `editing`：当前抽屉编辑中的主播对象
11. `message / error`：页面提示消息
12. `loadedOrgIds`：已懒加载过的组织节点集合

### 7.2 页面初始化流程

页面初始化时：

1. `useEffect` 监听当前身份变化
2. 调用 `loadRootOrgs()`
3. 请求 `/anchors/org-children`
4. 加载首屏可见组织节点
5. 默认只展开身份路径，不再默认选中任何组织
6. 右侧主播账号列表保持空态，等待用户先选择左侧组织

### 7.3 当前组织选择规则

当前逻辑已经改为"先选左树再加载右侧"：

1. 首次进入页面不自动选中任何组织
2. 用户点击左侧组织节点后，才更新 `selectedOrgId`
3. `selectedOrgId` 为空时，`useAnchorAccounts()` 会直接清空 `anchors / anchorTotal / apps`
4. 只有选中组织后，才会请求 `/anchors/profiles` 和（非 `HALL_MANAGER` 的）审核通过申请补充数据

这样可以保证：
- `HQ_ADMIN / BASE_ADMIN / TEAM_ADMIN` 首次进入时不会误拉整段组织范围数据
- `HALL_MANAGER` 进入时也会先看到自己的厅节点，再手动选择后加载右侧内容

### 7.4 左树展开逻辑

点击左侧组织树折叠/展开时：

1. 若该节点未加载过，调用 `loadOrgChildren(orgId)`
2. 请求 `/anchors/org-children?parentId=xxx`
3. 将子节点合并进 `orgs`
4. 前端再通过 `buildOrgTree(orgs)` 重构树结构
5. 懒加载只负责补充节点数据，不再反向覆盖用户当前的折叠状态

点击左侧某个组织节点用于"选中"时：

1. 会先确保该节点的祖先链路处于展开状态
2. 再更新当前 `selectedOrgId`
3. 最后按既有逻辑加载该组织范围的数据

这样可以避免用户点到某个厅或团队后，左树因为异步数据补全而出现"自动折叠"的错觉。

### 7.5 右侧列表加载逻辑

调用 `load()` 时：

1. 组装查询参数：
   - `page`
   - `pageSize`
   - `keyword`
   - `orgId`
   - `status`
   - `viewMode`
2. 调用 `/anchors/profiles`
3. 后端按组织范围 + 搜索条件 + 状态 + 身份视图分页返回
4. 更新：
   - `anchors`
   - `anchorTotal`

### 7.6 当前 / 历史视图交互规则

前端当前采用互斥视图：

- `current` 视图：只显示当前身份
- `history` 视图：只显示历史身份

行为规则：

1. 默认进入 `current`
2. 切换视图时自动回到第 1 页
3. 搜索、分页、总数、列表内容都跟随当前视图切换
4. 历史身份视图下，详情抽屉强制只读
5. 历史身份视图不应出现编辑、迁移、删除、启停、重置密码等运营动作心智

### 7.7 详情抽屉逻辑

点击表格行时：

1. 先根据当前列表项打开抽屉
2. 再调用 `/anchors/profiles/:id`
3. 详情返回后用详情对象覆盖 `editing` 和 `activeAnchorDetail`

这样做的意义：
- 列表接口保持轻量
- 抽屉可以展示更完整信息
- 后续继续扩展详情字段更容易维护

### 7.8 操作逻辑

页面支持以下操作：

#### 保存主播资料
- 调用 `PATCH /anchors/profiles/:id`
- 注意：普通保存已不允许直接修改 `hallOrgId`

#### 主播迁移
- 当前前端通过检测 `hallOrgId` 是否变化决定是否走迁移
- 调用 `POST /anchors/profiles/:id/migrate`
- 迁移前会弹出二次确认

#### 停用主播账号
- 调用 `POST /anchors/profiles/:id/disable`

#### 启用主播账号
- 调用 `POST /anchors/profiles/:id/enable`
- 注意：历史 `back` 档案不允许直接启用

#### 删除主播档案
- 调用 `DELETE /anchors/profiles/:id`

#### 重置密码
- 调用 `POST /accounts/:userId/reset-password`
- 默认密码重置为 `123456`

#### 历史视图约束
- 历史身份视图下详情抽屉前端强制只读
- 历史身份不可直接启用
- 历史身份如需回原厅，必须重新发起迁移

### 7.9 提示反馈机制

页面统一使用：
- `Toast` 成功提示
- `Toast` 错误提示
- 删除 / 重置密码等高风险操作使用浏览器 `confirm`

## 8. 后端处理逻辑

### 8.1 模块总入口

后端主播账号管理统一定义在：
- `backend/src/modules/anchor/routes.ts`

统一挂载：
1. `authRequired`
2. `identityRequired`
3. 接口级别 `permissionRequired`

### 8.2 组织范围控制

主播模块最关键的逻辑之一是：

- 当前身份 `scopePath` 控制可见范围
- 页面左树选中的 `orgId` 进一步收窄范围

核心处理方式：
1. 根据 `orgId` 查到组织 `path`
2. 生成 `effectiveScopePath = selectedOrg.path || currentIdentity.scopePath`
3. 用 `hallOrg.path startsWith effectiveScopePath` 过滤主播档案

### 8.3 组织树懒加载接口

后端 `getScopedOrgChildren()` 的职责：

- 根据当前身份与 `parentId` 返回下一层组织节点
- 支持 HQ / BASE / TEAM / HALL 分层懒加载
- 兼容开发管理员与总部管理员首屏加载逻辑

### 8.4 主播档案列表接口

`getProfiles()` 的职责：

1. 处理 keyword
2. 处理 status
3. 处理 orgId -> path 范围过滤
4. 处理 `viewMode=current/history` 身份视图过滤
5. 使用 `skip/take` 分页
6. 关联 `hallOrg` 与 `identities`
7. 将绑定用户信息整合成 `boundUser`

当前后端已加入统一视图口径：
- `current`：排除历史 `back` 档案
- `history`：只返回历史 `back` 档案

### 8.5 详情接口

`getProfileDetail()` 的职责：
- 读取单个主播档案
- 校验当前身份范围权限
- 补充归属厅与绑定用户信息

### 8.6 启用 / 停用逻辑

`toggleProfileStatus()` 的职责：

#### 启用时
- 若已绑定用户，启用该用户账号
- 恢复主播身份 `UserIdentity`
- 恢复 `scopePath`
- 更新档案状态为 `bound` 或 `unbound`
- **新增约束：历史 `back` 档案不允许直接启用**
- **新增约束：若该账号已存在其他 active 主播身份，不允许再启用第二条 active 主播身份**

#### 停用时
- 停用主播身份
- 写入 `expiredAt`
- **当前实现已移除“停用主播档案时顺带停整个用户账号”的旧逻辑**
- 更新主播档案状态为 `inactive`

### 8.7 迁移逻辑（2026-06 新增）

专用迁移接口：
- `POST /anchors/profiles/:id/migrate`

当前迁移逻辑不是"直接改归属厅"，而是：

1. 校验目标厅存在且有效
2. 校验当前档案已绑定账号
3. 校验当前账号不存在其他并存 active 主播身份
4. 将旧主播档案封存为历史 `back` 档案
5. 停用旧 `ANCHOR` 身份
6. 在目标厅创建新的主播档案
7. 为同一用户创建新的 active `ANCHOR` 身份
8. 保留原厅历史任务，不主动回写历史任务归属

当前迁移的业务定义：
- 迁移的是"主播身份"
- 不是"修改原档案归属厅"
- 回到原厅必须重新发起迁移，不可启用旧历史身份

### 8.8 删除逻辑

`deleteProfile()` 的职责：

1. 删除主播身份记录 `ANCHOR`
2. 删除与该绑定用户相关的注册申请记录
3. 若存在绑定用户，则禁用该用户
4. 删除主播档案

控制器层还有一个额外保护：
- 若该主播账号仍持有非主播类管理权限，禁止删除
- 需要先到"组织账号管理"中处理管理权限后再删

### 8.9 编辑资料与抖音号释放逻辑（2026-07-22 新增）

`updateProfile()` 的职责是更新主播资料（昵称、抖音号、抖音 UID）。

#### 抖音号唯一性冲突检测

更新抖音号时会调用 `assertNoAnchorRegistrationConflict`，该函数做**两重检查**：

1. **`anchorProfile` 表**：查是否已有其他档案使用该抖音号/UID（排除当前档案自身 `excludeProfileId`）
2. **`anchorRegistrationApplication` 表**：查是否已有 `pending` 或 `approved` 状态的注册申请使用该抖音号/UID（排除当前用户自己的申请 `excludeApplicationUserId`）

任一命中即抛出 `DOUYIN_NO_EXISTS` / `DOUYIN_UID_EXISTS` 错误。

#### 同步更新注册申请记录

当管理员修改主播的抖音号/UID 后，系统会同步更新该用户对应的 `anchorRegistrationApplication` 记录：

```typescript
if (identityChanged && current.boundUserId) {
    await tx.anchorRegistrationApplication.updateMany({
        where: {
            userId: current.boundUserId,
            status: { in: ["pending", "approved"] },
        },
        data: {
            douyinNo: nextDouyinNo,
            douyinUid: nextDouyinUid,
        },
    });
}
```

**业务意义**：

| 操作 | 效果 |
|---|---|
| A 注册 douyinNo=123456 | `anchorProfile.douyinNo=123456`，`anchorRegistrationApplication.douyinNo=123456` |
| 管理员改为 23456 | 两表都更新为 23456，旧值 `123456` 被释放 |
| B 注册 douyinNo=123456 | 冲突检测通过，正常注册 |

**设计要点**：
- 仅更新 `status` 为 `pending` 或 `approved` 的记录（参与冲突检测的记录），保留 `rejected`/`cancelled` 的历史原样
- 整个操作在 `prisma.$transaction` 事务中执行，失败自动回滚
- `AnchorRegistrationApplication` 表中 `douyinNo` 和 `douyinUid` 均无 `@unique` 约束，`updateMany` 不会触发唯一键冲突

## 11. 数据加载（页面懒加载分级传输）与后端处理逻辑

> 这里的"懒加载分级传输"不是 React 代码分包，而是页面按使用阶段逐步拉取数据、避免一次性返回全量明细的分层传输策略。

### 11.1 当前总体策略

主播账号管理模块当前采用以下加载顺序：

1. 先加载左侧首屏组织节点
2. 用户展开组织节点时再加载下一级组织
3. 用户选中组织后，再加载该组织范围下的主播账号列表第一页
4. 用户修改筛选条件时重新请求分页列表
5. 用户点击某一行后，再独立拉取详情
6. 仅当当前身份允许时，才额外加载审核通过申请列表，用于补充绑定账号信息和密码重置能力

### 11.2 第一层：组织树首屏加载

接口：
- `GET /anchors/org-children`

特点：
- 首屏只返回当前身份需要看到的第一层节点
- 避免一次性全量加载 HQ / BASE / TEAM / HALL 整棵树

### 11.3 第二层：组织树展开懒加载

接口：
- `GET /anchors/org-children?parentId=...`

特点：
- 只在用户展开节点时请求
- 子节点合并到 `orgs` 后重建树
- 用 `loadedOrgIds` 防止重复请求
- 不再因为 `orgs` 补数而重新按默认展开规则覆盖用户已操作的树状态

### 11.4 第三层：主播账号分页列表

接口：
- `GET /anchors/profiles`

特点：
- 按组织范围分页
- 每页默认 20 条
- 搜索词和状态筛选都在后端执行
- 不再依赖前端对当前页做伪搜索

### 11.5 第四层：详情抽屉按需加载

接口：
- `GET /anchors/profiles/:id`

特点：
- 点击行才请求
- 列表和详情分离
- 列表接口可保持更轻
- 抽屉可以独立扩展更多字段

### 11.6 第五层：审核通过申请补充信息

接口：
- `GET /anchors/applications?status=approved&page=1&pageSize=100&targetOrgId=...`

用途：
- 构建 `detailsByUserId`
- 用于给绑定账号提供更多上下文
- 用于重置密码按钮匹配 `userId`

注意：
- 这部分不是主列表数据来源
- 只有在已选组织时才会加载，并跟随当前 `selectedOrgId` 收缩范围
- HALL_MANAGER 不会走这条加载链路

### 11.7 后端处理逻辑总结

后端对数据传输做了以下处理：

#### 11.7.1 范围收缩
- 身份 `scopePath` 限制可见组织和主播档案
- 选中组织 `orgId` 再进一步缩小范围

#### 11.7.2 条件搜索
- `keyword`
- `status`
- `orgId`
- `page`
- `pageSize`

#### 11.7.3 分页传输
- `normalizePagination` 统一处理分页参数
- 使用 `skip / take` 控制数据库返回量

#### 11.7.4 详情与列表分离
- 列表接口负责分页摘要数据
- 详情接口负责单条扩展信息

这是当前主播账号管理性能优化的核心落地点之一。

### 12.3 排障建议

#### 问题 1：左侧点组织后右侧列表不变化
先检查：
- 是否成功触发 `/anchors/profiles?orgId=...`
- `selectedOrgId` 是否真的变化
- 后端 `getProfiles()` 是否正确把 `orgId` 转成 `path`
- 是否因为请求失败导致页面仍停留旧数据

#### 问题 2：搜索结果不正确
先检查：
- 请求是否带了 `keyword`
- 后端是否在 `nickname / douyinNo / douyinUid / bound user` 上查询
- 是否仍有前端本地过滤残留逻辑

#### 问题 3：点开抽屉详情不完整
先检查：
- `/anchors/profiles/:id` 是否返回了详情
- 当前身份是否有查看权限
- 前端是否正确用详情结果覆盖了列表对象

#### 问题 4：删除主播档案失败
先检查：
- 该绑定账号是否仍有非主播类管理身份
- 是否需要先去"组织账号管理"停用管理身份

#### 问题 5：密码重置按钮不显示
先检查：
- 当前档案是否有 `boundUserId`
- `approvedApps` 是否加载成功
- 是否能从审核通过申请列表里匹配到对应 `userId`
- 当前身份是否具备足够权限

## 15. 2026-06 异步导出功能（新增）

### 15.1 功能背景

原有导出方案为同步方式：前端调 `GET /anchors/profiles/export`，后端一次性查全量数据、生成 CSV 并直接作为响应流返回。

存在以下问题：
- 数据量大时请求超时
- 导出过程中用户无法知晓进度
- 浏览器 `<a href download>` 不携带 JWT Token，导致后端 401 拒绝、文件被存为 `file.json`

2026-06 改为**异步任务模式**：提交任务 → 后台生成 CSV → 前端轮询状态 → 完成后弹窗下载。

---

### 15.2 整体流程

```
用户点击"导出"
    │
    ▼
POST /anchors/export-tasks（提交任务，立即返回 taskId）
    │
    ▼
后台异步生成 CSV 并写入 uploads/exports/
    │
    ▼
前端"导出记录"弹窗轮询任务状态（每 3 秒）
    │
    ├─ pending / processing → 显示进度
    └─ done → 显示"下载"按钮
         │
         ▼
      点击"下载"
         │
         ▼
      fetch GET /anchors/export-tasks/:id/file
      （携带 Authorization header）
         │
         ▼
      Blob 转本地下载
```

---

### 15.3 数据库：`export_tasks` 表

新增模型 `ExportTask`，对应表 `export_tasks`。

| 字段 | 中文名 | 类型 | 说明 |
|---|---|---|---|
| id | 任务ID | UUID | 主键 |
| type | 任务类型 | string | 当前固定为 `anchor_profiles` |
| status | 任务状态 | enum | `pending / processing / done / failed` |
| params | 导出参数快照 | Json | 记录创建时的 `orgId / keyword / status` |
| file_path | CSV 文件路径 | string? | 相对路径，完成后写入 |
| row_count | 导出行数 | int? | 完成后写入 |
| error_msg | 错误信息 | string? | 失败时写入 |
| creator_id | 创建人ID | string | 外键 -> users.id |

### 15.4 后端：导出任务 Service

文件：`backend/src/modules/anchor/export-task.service.ts`

#### 核心方法

**`createTask(creatorId, params)`**
1. 检查是否已有同一创建人的 `pending / processing` 任务（防并发）
2. 写入 `export_tasks`，`status = pending`，`expiresAt = now + 7天`
3. 触发异步函数 `processTask(taskId)` — 不 await，立即返回 taskId
4. 懒清理（`setImmediate` 异步执行，不阻塞响应）：
   - `cleanExpiredTasks()`：删除所有用户已过期的任务记录和 CSV 文件
   - `trimUserTasks(creatorId)`：保留该用户最新 `MAX_TASKS_PER_USER`（10）条，超出的记录及文件一并删除

**`trimUserTasks(creatorId)`**（内部函数）
- 查询该用户全部 `anchor_profiles` 类型任务，按 `createdAt DESC` 排序
- 若总数 ≤ 10，不做任何操作
- 若超出，将第 11 条起的记录：先删除服务器 CSV 文件，再从数据库删除记录

**`processTask(taskId)`**（异步，不阻塞请求）
1. 将任务状态改为 `processing`
2. 按参数快照查询全量主播档案（复用 `getProfiles` 查询逻辑）
3. 组装 CSV（UTF-8 BOM，字段含中文引号转义）
4. 写入文件：`uploads/exports/主播账号导出_{orgId后6位}_{YYYYMMdd_HHmmss}_{taskId前8位}.csv`
5. 更新任务：`status = done`、`filePath`、`rowCount`
6. 任一步骤失败时：`status = failed`、写入 `errorMsg`

**`listMyTasks(creatorId)`**
- 查询当前用户创建的全部任务，按 `createdAt DESC`
- 过滤条件：`expiresAt > now`（仅返回未过期记录）
- **最多返回 10 条**（`take: 10`）

**`getTaskFile(taskId, creatorId)`**
- 鉴权：任务必须属于当前用户
- 状态必须为 `done`
- 未过期
- 返回文件绝对路径

### 15.7 前端：API 方法

文件：`frontend/src/features/anchors/api.ts`

新增三个方法：


**`downloadExportTaskFile` 实现要点**：

```ts
const res = await fetch(`/api/anchors/export-tasks/${id}/file`, {
  headers: {
    Authorization: `Bearer ${token}`,
    "X-Identity-Id": identityId,
  },
});
const blob = await res.blob();
const url = URL.createObjectURL(blob);
const a = document.createElement("a");
a.href = url;
a.download = filename;
a.click();
URL.revokeObjectURL(url);
```

> **重要**：必须用 `fetch` 携带 Token，不可用 `<a href download>`。
> `<a href download>` 是浏览器直接发起 GET，不带前端存储的 JWT，后端 401 后浏览器会把错误响应存为 `file.json`。

### 15.14 排障指南

#### 问题 1：点击"导出"后任务一直是 pending

- 后端日志是否有 `processTask` 报错
- `uploads/exports/` 目录是否存在且有写权限
- 数据库 `export_tasks` 表是否已创建（运行了 `migrate deploy`）

#### 问题 2：点击"下载"浏览器弹出 `file.json` 或提示无法下载

原因：使用了 `<a href download>` 不带 Token。

修复：确认前端调用的是 `anchorApi.downloadExportTaskFile()`，而不是 `<a>` 标签直接跳转。

#### 问题 3：下载文件在 Excel 中乱码

检查：CSV 文件是否以 UTF-8 BOM（`\uFEFF`）开头。`export-task.service.ts` 的 `processTask` 中已默认写入 BOM，若缺失说明文件生成逻辑被修改过。

#### 问题 4：`EXPORT_TASK_IN_PROGRESS` 错误一直出现

原因：上一个任务卡在 `processing` 状态（后端崩溃、服务重启等）。

临时处理：直接在数据库将卡住的任务 `status` 改为 `failed`：

```sql
UPDATE export_tasks
SET status = 'failed', error_msg = '手动修复：任务状态异常'
WHERE status IN ('pending', 'processing')
  AND created_at < NOW() - INTERVAL 10 MINUTE;
```

#### 问题 5：7 天后文件还没被清理

当前采用"懒清理"策略：只有在同一用户再次提交导出任务时才会触发清理。  
若要主动清理，可手动执行：

```bash
find /path/to/Pendingweb/uploads/exports -name "*.csv" -mtime +7 -delete
```

或配置 cron 定时清理。

#### 问题 6：导出记录弹窗最多只显示 10 条，旧记录看不到

这是设计行为：系统对每个用户只保留最新 10 条记录（7 天内有效）。  
每次提交新任务时，后台会异步触发 `trimUserTasks`，将超出 10 条的旧记录和对应 CSV 文件一并删除。  
若需要调整上限，修改 `export-task.service.ts` 中的常量：

```ts
const MAX_TASKS_PER_USER = 10; // 修改此值
```

## 16. 2026-07-22 管理员修改抖音号后释放旧值

### 16.1 问题背景

**场景**：用户 A 注册时抖音号填 `123456`，注册审核通过后发现自己填错了抖音号。管理员通过"主播账号管理 → 账号管理"将 A 的抖音号改为 `23456`。此时用户 B 想用 `123456` 注册，却发现被拦截。

**根因**：`updateProfile` 修改抖音号时只更新了 `anchorProfile` 表，没有同步更新 `anchorRegistrationApplication` 表。冲突检测逻辑 `assertNoAnchorRegistrationConflict` 会同时查询两张表，因此旧抖音号 `123456` 在注册申请表中仍然存在且状态为 `approved`，导致 B 无法注册。

### 16.2 修改内容

**文件**：`backend/src/modules/anchor/service.ts`

**修改位置**：`updateProfile()` 方法中，`anchorProfile.update` 成功后、`syncAnchorProfileSnapshots` 之前。

**修改逻辑**：当抖音号或 UID 发生变化且档案已绑定用户时，同步更新该用户 `pending`/`approved` 状态的注册申请记录。

### 16.3 影响评估

| 维度 | 评估 |
|---|---|
| **正确性** | 准确释放旧值，仅更新参与冲突检测的记录 |
| **安全性** | 事务保护，失败回滚 |
| **审计性** | 驳回/取消记录保持原样，不被污染 |
| **影响面** | 仅影响当前用户的申请记录 |
| **兼容性** | 纯增量逻辑，不改变现有接口契约 |

**风险等级**：低风险。`AnchorRegistrationApplication` 表无 `@unique` 约束，不会触发数据库冲突；事务内执行确保一致性。

### 16.4 验证方法

1. 用户 A 用 `douyinNo=123456` 注册并审核通过
2. 管理员在"账号管理"中将 A 的抖音号改为 `23456`
3. 用户 B 用 `douyinNo=123456` 注册 → 应正常通过，不再提示抖音号已存在
4. 查询 `anchor_registration_applications` 表：用户 A 的 `pending`/`approved` 记录中 `douyin_no` 应为 `23456`
