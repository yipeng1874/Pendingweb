# 飞书通知开发者文档

## 1. 文档目标

本文档用于系统性说明当前项目中的“飞书任务通知”能力，重点覆盖：

- 日常任务飞书通知的接口、筛选口径与页面入口
- 临时任务飞书通知的接口、筛选口径与页面入口
- 前后端代码目录、关键函数、关键字段与数据流
- 数据库表与字段如何支撑通知计算
- 后续开发者与 AI 工具接手时的推荐阅读顺序与维护注意事项

本文档的目标不是介绍飞书登录绑定，而是专门说明：

> 任务模块里，系统如何计算“谁该被通知”、如何调用飞书发送、以及前端在哪里触发通知。

---

## 2. 功能概述

当前项目中的飞书通知能力分为两条主线：

1. **日常任务飞书通知**
2. **临时任务飞书通知**

两者都基于：
- 用户表中的飞书绑定信息
- 飞书企业配置表中的 `appId / appSecret`
- 任务记录表中的任务完成状态

但两者在人群计算方式上有明显差异：

### 2.1 日常任务通知

日常任务通知更接近“按当天任务记录催办”。

主要特点：
- 面向某个基地当天的日常任务
- 只通知当天仍处于 `pending / in_progress` 的人
- 同一人按当天口径只统计一次
- 发送文案统一为：
  - `前缀，日常任务共 x 件，您还有 x 件没完成。`

### 2.2 临时任务通知

临时任务通知更接近“按某条进行中的 assignment 催办”。

主要特点：
- 只允许对**进行中的临时任务 assignment**发通知
- 只通知 `pending / in_progress` 的主体
- 人群计算复用临时任务现有主体分组逻辑
- 支持 `ACCOUNT / ANCHOR / MANAGER` 三种模式
- 业务规则当前按“主体统计不合并”实现预览与统计
- 发送文案当前统一为：
  - `前缀，临时任务共 x 件，您还有 x 件没完成。`

## 6. 权限与调用限制

## 6.1 通用中间件
所有通知接口统一挂在：
- `authRequired`
- `identityRequired`
- `permissionRequired("task:report:view")`

也就是说：
- 必须已登录
- 必须具备当前身份
- 必须拥有查看任务报表权限

## 6.2 日常任务权限规则
函数：`canManageDailyNotify(roleCode?: string)`

当前允许角色：
- `DEV_ADMIN`
- `HQ_ADMIN`
- `BASE_ADMIN`
- `TEAM_ADMIN`
- `HALL_MANAGER`

此外还会通过 `resolveBaseScopeOrg` 解析基地范围：
- HQ / DEV 可显式选基地
- 其他角色从当前身份组织向上回溯所在基地

## 6.3 临时任务权限规则
当前实现中临时任务通知除了角色校验外，还要求：
- 任务必须是 `TEMPORARY`
- 任务必须是 `active`
- 发送者必须满足任务拥有权校验

任务拥有权规则来自：
- `backend/src/modules/task/assignment/assignment.service.ts`
- 关键函数：`assertTemporaryOwnership`

当前口径：
- 如果 assignment 有 `createdByIdentityId`
  - 当前身份 `identityId` 必须与之相同
- 否则回退到 `createdBy` 与当前 `userId` 匹配

这意味着：
> 当前临时任务通知不是“看板里谁能看谁就能发”，而是“任务发起人 / 发起身份”才能发。

## 7. 日常任务通知的人群计算逻辑

## 7.1 核心函数
- `buildDailyNotifyAudience(taskDate, baseOrg)`
- 文件：`backend/src/modules/task/notify/notify.routes.ts`

## 7.2 计算步骤
1. 调用 `reconcileDailyAssignments(baseOrg.path)` 对日常任务状态做对账
2. 查询当前基地相关的日常任务 assignment
3. 过滤条件包括：
   - `category = DAILY`
   - `targets.some.orgId = baseOrg.id`
   - `status in (scheduled, active, ended)`
   - `deletedAt = null`
   - `effectiveAt <= 当天补录截止`
   - `endedAt` 为空或晚于当天结束时间
4. 对每个 assignment 调用：
   - `listAssignmentAudienceMembers(prisma, assignment)`
5. 以 `userId:taskDate` 去重
6. 查询对应 `TaskRecord`
7. 调用 `resolveTaskRecordStatus(...)`
8. 仅保留：
   - `pending`
   - `in_progress`
9. 拼出通知行 `NotifyAudienceRow`

## 7.3 关键口径说明
### 为什么日常任务是一人一天只统计一次
因为当前实现里：
- `refs` 的 key 是 `${userId}:${taskDate}`
- 同一用户当天重复命中时只保留第一次出现的 assignment 参考

### 为什么日常任务不支持“按 assignment 逐条通知”
因为日常通知本质是按“基地 + 当天”来汇总催办，而不是按单条 assignment。

## 8. 临时任务通知的人群计算逻辑

## 8.1 核心函数
- `buildTemporarySubjectGroups(...)`
- 文件：`backend/src/modules/task/assignment/temporary-assignment.utils.ts`

临时任务通知必须复用这套现有主体分组逻辑，不能自行重写组织树筛选。

## 8.2 三种模式

### A. `ACCOUNT`
主体类型通常按 `USER` 聚合。

来源包括：
- 明确设置的 `targetUserIds`
- 目标组织范围 `targets.orgPathSnapshot` 下扫描到的账号

### B. `ANCHOR`
主体类型按 `USER` 聚合。

来源包括：
- 目标组织范围下指定角色身份
- 排除组织与排除主播逻辑都会生效

### C. `MANAGER`
主体类型按 `ORG` 聚合。

来源包括：
- `temporarySubjectOrgType` 指定的组织层级
- 每个主体组织再反查其 `visibleIdentities`

## 8.3 当前通知口径
临时任务当前实现规则如下：
- 只允许通知 `assignment.status = active`
- 只通知 `TaskRecord.status in (pending, in_progress)`
- 统计上按主体计算，不做业务层合并
- 预览里显示：
  - 主体总数 `total`
  - 涉及账号数 `distinctUserCount`
  - 未开始数 `pendingCount`
  - 进行中数 `inProgressCount`
  - 未绑定飞书数 `unboundCount`

## 8.4 一个重要实现细节
### 业务统计“不合并”
你们当前定下的产品口径是：
- 一个人命中多条主体，不合并

因此在统计和预览上：
- `total` 统计的是主体数，不是用户数

### 发送层目前仍按 openId 去重批量发
后端当前复用飞书批量发送接口：
- `sendFeishuBatchMessage(config, openIds, text)`

因此同一飞书配置桶内：
- 相同 `openId` 会去重后再发

这意味着：
- 预览与统计口径：按主体
- 实际同一桶的飞书调用：按 `openId` 去重

如果未来要实现：
> 同一用户命中 3 条主体，就真的收到 3 条独立飞书

则需要把临时任务通知改造为：
- 不使用批量接口聚合 openIds
- 改为逐主体逐条发送

这是后续维护中最值得注意的一个点。

## 9. 飞书发送实现说明

## 9.1 获取企业 token
函数：`getFeishuTenantAccessToken(config)`

调用飞书接口：
- `POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal`

依赖字段：
- `config.appId`
- `config.appSecret`

## 9.2 批量发送消息
函数：`sendFeishuBatchMessage(config, openIds, text)`

调用飞书接口：
- `POST https://open.feishu.cn/open-apis/message/v4/batch_send/`

发送格式：
- `msg_type = text`
- `open_ids = [...]`
- `content.text = 文本`

## 9.3 为什么按飞书配置分桶
因为不同用户可能绑定在不同飞书企业配置下：
- 每个 `feishuConfigId` 对应不同 `appId / appSecret`
- 同一个企业配置共用一个 `tenant_access_token`

因此后端会：
1. 按 `feishuConfigId` 对通知对象分桶
2. 每桶单独换取 token
3. 每桶单独调用一次飞书批量发送

## 13. 当前维护注意事项

## 13.1 不要单独重写临时任务通知的人群算法
临时任务通知一定要复用：
- `buildTemporarySubjectGroups`

否则很容易丢失：
- `ACCOUNT / ANCHOR / MANAGER` 三模式差异
- 组织排除规则
- 主播排除规则
- 可见身份逻辑

## 13.2 日常与临时的通知粒度不同
- 日常任务：基地 + 日期
- 临时任务：单 assignment

不要把两者混成同一套总通知逻辑。

## 13.3 临时任务的“业务统计不合并”和“发送去重”并存
这是当前实现里最容易误解的一点：
- 预览上看，临时任务按主体统计
- 发送上看，飞书批量接口会按 `openId` 去重

后续修改文案、统计口径、消息数量时，必须先确认业务想改的是：
- 统计展示层
- 还是实际飞书发送次数

## 13.4 弹窗输入前缀是当前统一交互
目前通知前缀的交互已统一为：
- 页面主区域不再长期保留输入框
- 用户点击通知按钮后，在弹窗中填写前缀再确认发送

若后续开发者再次在页面主区域增加输入框，应先确认是否会破坏当前交互一致性。

## 13.5 推荐后续可继续优化的方向
后续如需继续扩展，建议优先考虑：

1. 将日常 / 临时通知弹窗抽成共用组件
2. 将通知预览 / 发送返回类型迁移到 `frontend/src/types.ts`
3. 引入通知发送日志表，记录：
   - 发送时间
   - 发送人
   - assignmentId / taskDate
   - 目标人数
   - 成功 / 失败人数
4. 若产品要求“一人命中多条主体必须收到多条消息”，改造临时任务发送为逐条单发

## 14. 面向 AI 工具的阅读提示

如果后续让 AI 继续维护飞书通知能力，建议在提示词里明确说明：

1. 当前项目存在两套通知：`daily-feishu` 与 `temporary-feishu`
2. 日常任务与临时任务的人群计算逻辑不同
3. 临时任务通知必须复用 `buildTemporarySubjectGroups`
4. 临时任务当前的产品规则是：
   - 只通知 active assignment
   - 只通知 pending / in_progress
   - 业务统计按主体不合并
   - 文案前缀由发送者决定
5. 当前弹窗交互为统一页面内弹层，不应回退为浏览器原生 `window.confirm`

这样 AI 在继续改功能时，更不容易误删规则或重写错误的人群计算路径。
