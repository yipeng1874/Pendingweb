# 部署开发者文档

## 1. 文档目标

本文档用于给当前 `D:\Pendingweb` 项目提供一份可长期复用的部署开发者文档，方便后续：

- 本机验证正式访问形态
- 公司电脑首次部署
- 后续版本更新
- Nginx / FRP / 后端运行维护

本文档面向开发者、实施人员、后续接手人员，重点强调“开发模式”和“正式访问模式”的区别，以及当前项目在公司电脑 + FRP 场景下的推荐部署方式。

---

## 2. 当前项目组成

项目当前主要由三部分组成：

1. `backend`：后端服务，负责 API、登录、权限、任务、提醒、上传等
2. `frontend`：PC Web 前端
3. `todo-h5`：H5 前端

此外还有：

- `doce/`：项目文档目录
- `uploads/`：后端上传目录
- `frpc.ini`：FRP 客户端配置

---

## 3. 两种运行模式

## 3.1 开发模式

开发模式用于：

- 改代码
- 热更新
- 本地调试
- 接口联调

当前开发模式端口：

- 后端：`4000`
- PC Vite：`5173`
- H5 Vite：`4173`

开发模式特征：

- 前端由 Vite dev server 提供
- 浏览器会看到很多 `.ts/.tsx` 模块请求
- 会有 websocket / HMR
- 不适合作为正式用户访问入口

---

## 3.2 正式访问模式

正式访问模式用于：

- 给真实用户访问
- 走 Nginx 提供 build 后静态资源
- 更节省 FRP 带宽
- 更接近后续公司电脑正式环境

当前推荐正式访问模式端口：

- 后端：`4000`
- PC Nginx：`8088`
- H5 Nginx：`8081`

正式访问模式特征：

- PC 使用 `frontend/dist`
- H5 使用 `todo-h5/dist`
- PC 与 H5 分别走独立端口
- `/api` 由各自 Nginx 转发到 `4000`
- 浏览器看到的是 `assets/*.js`、`assets/*.css`
- 没有 Vite websocket

---

## 4. 当前推荐正式访问架构

推荐访问链路：

```text
PC 浏览器
  -> FRP 29266
  -> Nginx 8088
     -> frontend/dist
     -> /api 代理到 backend 4000
     -> /uploads 映射上传目录

H5 浏览器
  -> FRP 29267
  -> Nginx 8081
     -> todo-h5/dist
     -> /api 代理到 backend 4000
     -> /uploads 映射上传目录
```

说明：

- 正式访问时不再依赖 `5173` 和 `4173`
- `5173` / `4173` 只保留给开发使用
- PC 和 H5 不再共用子路径部署
- PC 与 H5 分别使用独立入口，更稳定、更易维护

## 5. 必装软件与环境

## 5.1 必装软件

### 1. Git

用途：

- 拉取代码
- 更新代码
- 保持公司电脑与开发电脑同步

### 2. Node.js

建议版本：

- Node.js 18 LTS 或 20 LTS

用途：

- 安装依赖
- 构建 PC/H5/后端
- 运行后端

### 3. MySQL

建议版本：

- MySQL 8.x

用途：

- 项目业务数据库

### 4. Redis

建议版本：

- Redis 6.x / 7.x

用途：

- Redis 能力支持

### 5. Nginx

建议版本：

- Windows 稳定版 Nginx

当前验证版本示例：

- `C:\nginx-1.30.2`

用途：

- 提供正式静态访问入口
- 代理 API
- 开启 gzip 和缓存

### 6. PM2

安装命令：

```bash
npm install -g pm2
```

用途：

- 后端常驻运行
- 崩溃自动拉起
- 方便重启和看日志

### 7. FRP 客户端

用途：

- 在没有公网 IP 的情况下，把公司电脑服务暴露到固定外网地址

---

## 5.2 建议安装的软件

- 7-Zip：解压 nginx、frp 等工具
- DBeaver / Navicat / HeidiSQL：管理数据库
- Everything：快速查文件

## 6. Git 应该管理什么

## 6.1 应该提交到 Git 的内容

- `backend/src`
- `backend/prisma`
- `frontend/src`
- `todo-h5/src`
- `package.json`
- `vite.config.ts`
- `tsconfig.json`
- 文档与脚本
- `.env.example`

## 6.2 不应该提交到 Git 的内容

- `node_modules`
- `dist`
- `.env`

- 日志
- 本地缓存文件

当前项目已通过 `.gitignore` 做了基础规范。

## 7. 当前正式访问需要构建哪些内容

## 7.1 PC 前端

在项目根目录执行：

```bash
npm run build -w frontend
```

构建产物：

- `frontend/dist`

## 7.2 H5

进入 H5 目录执行：

```bash
cd todo-h5
npm run build
```

构建产物：

- `todo-h5/dist`

说明：

- H5 当前采用独立端口部署
- 不再依赖 `/h5/` 子路径
- H5 飞书回调页使用独立地址承接授权结果

## 7.3 后端

在项目根目录执行：

```bash
npm run build -w backend
```

构建产物：

- `backend/dist`

## 8. Nginx 负责什么

Nginx 在当前方案里负责：

1. 提供 PC build 后静态站点
2. 提供 H5 build 后静态站点
3. 代理 `/api` 到 `127.0.0.1:4000`
4. 暴露 `/uploads`
5. 开启 gzip 压缩
6. 为静态资源添加缓存策略
7. 处理前端路由刷新不 404

当前项目中建议重点关注的 Nginx 文件：

- 机器实际生效文件：`C:\nginx-1.30.2\conf\nginx.conf`
- 项目内备份示例：`deploy/nginx.conf.example`
- 说明文档：`doce/nginx-本机与公司电脑通用示例配置.md`

说明：

- `C:\nginx-1.30.2\conf` 目录下的 `mime.types`、`fastcgi_params`、`scgi_params`、`uwsgi_params` 等文件属于 Nginx 自带基础文件，通常不需要放进项目仓库维护。
- 真正和你项目部署强相关、建议备份到项目里的核心文件是 `nginx.conf`。
- 如果以后公司电脑路径从 `D:` 盘迁到 `C:` 盘，优先修改 `nginx.conf` 里的 `root` 和 `alias` 路径。

---

## 9. FRP 负责什么

FRP 在当前方案里负责：

- 把外部访问入口转发到本机 / 公司电脑 Nginx 端口

当前配置文件：

- `D:\Pendingweb\frpc.ini`

当前已有开发映射：

- `29263 -> 5173`（PC Vite）
- `29264 -> 4000`（后端）
- `29265 -> 4173`（H5 Vite）

当前推荐正式访问映射：

- `29266 -> 8088`（PC Nginx）
- `29267 -> 8081`（H5 Nginx）

推荐做法：

- 开发时保留旧映射
- 正式访问使用独立 Nginx 入口
- PC 和 H5 不再共用同一子路径入口，而是拆成两个独立端口，降低路由与缓存干扰

## 10. 本机验证正式访问形态的步骤

## 10.1 先构建

```bash
npm run build -w frontend
cd todo-h5
npm run build
cd ..
```

## 10.2 启动后端

确保后端 `4000` 正常运行。

## 10.3 启动 Nginx

当前本机验证使用目录示例：

- `C:\nginx-1.30.2`

先检查配置：

```powershell
.\nginx.exe -t
```

启动：

```powershell
.\nginx.exe
```

修改配置后重载：

```powershell
.\nginx.exe -s reload
```

## 10.4 本机验证地址

- PC：`http://localhost:8088/`
- H5：`http://localhost:8081/`

## 10.5 预期结果

浏览器应看到：

- `index.html`
- `assets/*.js`
- `assets/*.css`

不应再看到：

- `.ts/.tsx` 模块源码请求
- Vite websocket / HMR client

## 10.6 当前已验证成功的正式入口

当前建议正式访问入口为：

- PC：`http://frp7.ccszxc.site:29266/`
- H5：`http://frp7.ccszxc.site:29267/`

说明：

- PC 由 `8088` 提供服务
- H5 由 `8081` 提供服务
- FRP 分别映射到两个独立端口
- 比把 H5 挂在 `/h5/` 子路径更稳

## 11. 公司电脑正式部署建议目录

建议目录：

```text
C:\deploy
├─ source      # git 拉下来的源码
├─ pc-web      # frontend/dist
├─ h5-web      # todo-h5/dist
├─ backend     # 后端运行目录
├─ uploads     # 上传目录
└─ nginx       # nginx 目录（如需统一管理）
```

说明：

- 公司电脑上也可以继续保留源码目录
- 但正式访问建议指向部署目录，不直接指向开发源码目录

## 14. 哪些情况下需要重启或 reload

## 14.1 前端 / H5 代码更新

需要：

- 重新 build

通常不需要：

- 改 Nginx 配置

## 14.2 后端代码更新

需要：

- 重新 build（如果使用 build 运行）
- 重启后端进程 / PM2

## 14.3 Nginx 配置变化

例如：

- 静态目录变更
- `/api` 代理变化
- `/uploads` 变化
- gzip / cache 变化

需要执行：

```bash
nginx -s reload
```

## 14.4 FRP 配置变化

例如：

- 端口映射变化
- 从 dev 映射切到 Nginx 映射

需要：

- 重启 frpc

## 16. 推荐长期使用方式

推荐长期维护方式：

### 开发

- 开发电脑继续使用 `start-dev.bat`
- 用 Vite 热更新开发

### 正式访问

- 后端运行在 `4000`
- PC Nginx 运行在 `8088`
- H5 Nginx 运行在 `8081`
- FRP 映射：
  - `29266 -> 8088`（PC）
  - `29267 -> 8081`（H5）
- 用户分别访问 PC 与 H5 独立入口

这样可以：

1. 节省 FRP 带宽
2. 避免开发服务直接暴露给用户
3. 避免 PC 与 H5 共享子路径产生的路由干扰
4. 更接近真正可维护的正式部署模型
5. 让公司电脑部署更稳定
