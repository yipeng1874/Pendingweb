# 流转式任务开发者文档

## 1. 文档目标

本文档用于系统性说明当前项目中"流转式任务（Workflow Task）"从发布、流转、执行、提交到看板闭环的完整业务链路，方便：

- 新接手项目的前端开发快速理解页面入口、交互流程与状态机
- 新接手项目的后端开发快速定位各层职责、数据结构与接口逻辑
- 测试、产品人员统一理解流转规则与权限边界
- AI 工具在阅读代码时快速建立完整上下文，避免误改业务规则

本文档只覆盖"流转模式（workflow）"，协同模式与群发任务为后续规划，当前 `enabled: false`。

---

## 2. 业务定义与核心原则

### 2.1 什么是流转式任务

流转式任务是"协同任务"模块下的第一个子模式，类似审批流。任务由管理员创建，包含若干有序节点，每个节点指定一名执行人。节点按顺序逐个激活，前一节点完成提交后，后一节点自动激活，直至全部节点完成，任务整体结束。

**类比：** A 完成后交给 B，B 完成后交给 C，按顺序流转至结束。

### 2.2 核心业务规则

1. 任务由具备管理角色的人员在「任务发布 → 协同任务」页面创建
2. 每个任务至少包含 1 个节点，每个节点指定唯一执行人
3. 创建时仅第 1 个节点状态为 `active`，其余节点为 `pending`
4. 当前节点提交后，系统自动将下一节点从 `pending` 切换为 `active`
5. 全部节点 `completed` 后，任务整体状态变为 `completed`
6. 截止时间（`dueAt`）过后不可再提交
7. 执行人只能提交自己负责的、且处于 `active` 状态的节点
8. 发布权限：`DEV_ADMIN` / `HQ_ADMIN` / `BASE_ADMIN` / `TEAM_ADMIN` / `HALL_MANAGER`
9. 执行层（提交节点）：所有已登录用户，只需 `authRequired`，无需管理身份
10. 数据持久化使用 JSON 文件，不走 MySQL/Prisma

### 2.3 发布范围限制

- 非 `DEV_ADMIN` 角色只能在自己的 `scopePath` 范围内选择执行人
- 执行人必须是活跃身份（`status: active`）且为非主播角色（`roleCode != ANCHOR`）
- 同一账号多身份只算一个执行人（按 `userId` 去重）

## 5. 数据结构

### 5.1 任务主体（WorkflowTaskRecord）

```typescript
// backend/src/modules/task/collaboration/workflow.store.ts

type WorkflowTaskRecord = {
  id: string;                        // UUID，创建时自动生成
  mode: "workflow";                  // 固定值，当前只有流转模式
  title: string;                     // 任务标题（必填）
  description?: string;              // 任务描述（选填）
  targetOrgId: string;               // 目标组织 ID
  targetOrgName: string;             // 目标组织名称
  issuerOrgId?: string | null;       // 发布组织 ID
  issuerOrgName?: string | null;     // 发布组织名称
  issuerScopePath?: string | null;   // 发布者的组织路径（用于权限范围）
  createdByUserId: string;           // 发布人 userId
  createdByIdentityId: string;       // 发布人 identityId
  createdByName: string;             // 发布人昵称（快照）
  dueAt?: string | null;             // 截止时间（ISO 8601），可选
  status: WorkflowTaskStatus;        // 任务状态：draft | in_progress | completed
  currentStepOrder: number;          // 当前激活节点的 order（从 1 开始）
  createdAt: string;                 // 创建时间（ISO 8601）
  updatedAt: string;                 // 最后更新时间
  steps: WorkflowTaskStepRecord[];   // 节点列表（有序）
}
```

### 5.2 节点（WorkflowTaskStepRecord）

```typescript
type WorkflowTaskStepRecord = {
  id: string;                        // UUID
  order: number;                     // 节点序号（从 1 开始）
  title: string;                     // 节点标题
  requirement: string;               // 节点要求说明
  questions: WorkflowTaskQuestionRecord[];  // 题目列表
  assigneeUserId: string;            // 执行人 userId
  assigneeName: string;              // 执行人昵称（快照）
  assigneeOrgId?: string | null;     // 执行人所属组织 ID
  assigneeOrgName?: string | null;   // 执行人所属组织名称
  status: WorkflowStepStatus;        // pending | active | completed
  completedAt?: string | null;       // 完成时间
  stepAnswers?: WorkflowStepAnswer[]; // 提交的答案列表
  submittedAt?: string | null;       // 提交时间
  submittedByUserId?: string | null; // 实际提交人 userId
}
```

### 5.3 题目（WorkflowTaskQuestionRecord）

```typescript
type WorkflowTaskQuestionRecord = {
  id: string;                        // UUID
  title: string;                     // 题目内容
  itemType: WorkflowTaskQuestionType; // 题型（见下表）
  isRequired: boolean;               // 是否必填
  options?: string[];                // 选项列表（单选/多选题使用）
}

// 支持题型
type WorkflowTaskQuestionType =
  | "FILL_BLANK"     // 待办确认（勾选完成）
  | "SINGLE_CHOICE"  // 单选题
  | "MULTI_CHOICE"   // 多选题
  | "LINK"           // 链接确认
  | "ATTACHMENT"     // 附件上传
  | "QA"             // 问答（前端扩展类型）
```

### 5.4 答案（WorkflowStepAnswer）

```typescript
type WorkflowStepAnswer = {
  questionId: string;                // 对应题目 ID
  answerText?: string;               // 文本答案（FILL_BLANK / QA 使用）
  answerOptions?: string[];          // 选项答案（SINGLE_CHOICE / MULTI_CHOICE 使用）
  isLinkConfirmed?: boolean;         // 链接是否已确认（LINK 使用）
}
```

### 5.5 状态枚举

```typescript
// 任务状态
type WorkflowTaskStatus = "draft" | "in_progress" | "completed"

// 节点状态
type WorkflowStepStatus = "pending" | "active" | "completed"
```

| 状态 | 含义 |
|------|------|
| `draft` | 草稿（当前未使用，创建即 in_progress） |
| `in_progress` | 进行中（有节点尚未完成） |
| `completed` | 全部节点已完成 |
| `pending` | 节点等待中（前序节点未完成） |
| `active` | 节点当前可执行（轮到该执行人操作） |
| `completed` | 节点已提交完成 |

## 6. 状态机与流转规则

### 6.1 任务生命周期

```
创建任务
  └─ task.status = "in_progress"
     step[0].status = "active"      ← 第 1 节点立即激活
     step[1..N].status = "pending"  ← 其余节点等待

step[0] 执行人提交
  └─ step[0].status → "completed"
     step[0].completedAt = now
     step[1].status → "active"      ← 自动激活下一节点
     task.currentStepOrder → 2

step[1] 执行人提交
  └─ step[1].status → "completed"
     step[2].status → "active"      ← 继续流转
     ...

最后一个节点提交
  └─ step[N].status → "completed"
     allCompleted = true
     task.status → "completed"      ← 任务整体完成
```

### 6.2 关键流转代码（workflow.store.ts）

```typescript
// submitWorkflowStep() 核心逻辑
step.status = "completed";
step.completedAt = now;

// 自动激活下一节点
const nextStep = task.steps.find((s) => s.order === step.order + 1);
if (nextStep && nextStep.status === "pending") {
  nextStep.status = "active";
  task.currentStepOrder = nextStep.order;
}

// 全部完成则关闭任务
const allCompleted = task.steps.every((s) => s.status === "completed");
if (allCompleted) {
  task.status = "completed";
}
```

### 6.3 节点提交校验规则

提交节点时，后端依次校验：
1. 用户已登录（`userId` 存在）
2. 任务存在且该用户在任务的执行人列表中（`getWorkflowTasksForUser`）
3. 任务未超过截止时间（`dueAt`）
4. 节点存在（`stepId` 有效）
5. 节点的 `assigneeUserId === userId`（只能提交自己负责的节点）
6. 节点状态为 `active`（非 pending/completed 不可提交）

## 7. 后端接口清单

所有接口前缀：`/tasks/collaboration/workflows`

全局中间件：`authRequired`（所有路由需要登录）

### 7.1 接口总览

| 方法 | 路径 | 权限 | 功能 |
|------|------|------|------|
| `GET` | `/bootstrap` | `identityRequired` + 管理角色 | 初始化发布页数据 |
| `GET` | `/assignees/search` | `identityRequired` + 管理角色 | 搜索可选执行人 |
| `POST` | `/` | `identityRequired` + 管理角色 | 创建流转任务 |
| `GET` | `/mine` | `authRequired` 仅 | 获取我参与的流转任务（执行层） |
| `POST` | `/:taskId/steps/:stepId/submit` | `authRequired` 仅 | 提交节点答案 |
| `GET` | `/issued` | `identityRequired` + 管理角色 | 获取我发布的任务（看板用） |
| `GET` | `/:taskId` | `identityRequired` + 管理角色 | 获取单个任务完整详情（含答案） |

> **注意路由顺序**：`/mine`、`/issued`、`/bootstrap`、`/assignees/search` 等固定路径必须注册在 `/:taskId` 动态路由之前，否则会被误匹配。

### 7.2 接口详情

---

#### GET `/bootstrap`

**权限**：管理角色（`canManageWorkflow`）  
**功能**：返回发布页所需的全量初始化数据

**响应结构：**
```json
{
  "enabled": true,
  "mode": "workflow",
  "operator": {
    "identityId": "xxx",
    "orgId": "xxx",
    "roleCode": "TEAM_ADMIN",
    "scopePath": "/root/hq/base/team",
    "orgName": "某团队"
  },
  "rules": {
    "publishScope": "downstream_only",
    "notes": ["V1 仅实现流转模式..."]
  },
  "orgOptions": [...],           // 可选目标组织列表
  "assigneeOptions": [...],      // 可选执行人列表（按优先级排序）
  "tasks": [...]                 // 已发布任务列表（用于历史展示）
}
```

**执行人分类（AssigneeCategory）：**
| 值 | 含义 | 优先级 |
|----|------|-------|
| `subordinate_anchor` | 下级主播 | 最高 |
| `subordinate_manager` | 下级管理员 | 次之 |
| `peer_manager` | 同级管理员 | 最低 |

---

#### GET `/assignees/search?keyword=xxx`

**权限**：管理角色  
**功能**：关键词搜索可选执行人

**搜索字段**：昵称、手机号、组织名称、抖音号、主播昵称  
**最少输入长度**：非手机号 ≥ 3 字符，手机号 ≥ 5 位数字  
**返回**：最多 20 条，按分类 + 组织路径 + 昵称排序

---

#### POST `/`

**权限**：管理角色  
**功能**：创建流转任务

**请求体：**
```json
{
  "title": "任务标题",
  "description": "任务描述（选填）",
  "dueAt": "2026-06-30T18:00:00.000Z",
  "steps": [
    {
      "title": "节点一",
      "requirement": "完成 xxx",
      "assigneeUserId": "user_id_1",
      "questions": [
        {
          "title": "是否已完成准备工作？",
          "itemType": "FILL_BLANK",
          "isRequired": true
        },
        {
          "title": "选择完成方式",
          "itemType": "SINGLE_CHOICE",
          "isRequired": true,
          "options": ["线上", "线下"]
        }
      ]
    },
    {
      "title": "节点二",
      "requirement": "审核结果",
      "assigneeUserId": "user_id_2",
      "questions": [...]
    }
  ]
}
```

**校验规则：**
- `title` 不能为空
- `steps` 至少 1 个
- 每个节点的 `assigneeUserId` 必须有效且在管理范围内
- 每个节点至少 1 道题目，题目标题不能为空
- 单选/多选题选项不少于 2 个

**响应**：创建成功的 `WorkflowTaskRecord` 完整对象

---

#### GET `/mine`

**权限**：仅 `authRequired`（无需管理身份）  
**功能**：返回当前用户作为执行人参与的所有流转任务

**筛选逻辑（workflow.store.ts）：**
```typescript
tasks.filter(task =>
  task.steps.some(step => step.assigneeUserId === userId)
)
```

**说明**：只要任意节点的 `assigneeUserId` 等于当前用户，该任务就会出现在列表中（无论节点状态是 pending/active/completed）。执行人在前端通过节点状态判断当前是否轮到自己操作。

---

#### POST `/:taskId/steps/:stepId/submit`

**权限**：仅 `authRequired`  
**功能**：提交指定节点的答案

**请求体：**
```json
{
  "answers": [
    {
      "questionId": "question_uuid",
      "answerText": "已完成"
    },
    {
      "questionId": "question_uuid_2",
      "answerOptions": ["线上"]
    },
    {
      "questionId": "question_uuid_3",
      "isLinkConfirmed": true
    }
  ]
}
```

**错误码：**
| 错误码 | HTTP | 含义 |
|--------|------|------|
| `TASK_NOT_FOUND` | 404 | 任务不存在或无权访问 |
| `TASK_OVERDUE` | 400 | 任务已超截止时间 |
| `STEP_NOT_FOUND` | 400 | 节点不存在 |
| `FORBIDDEN` | 400 | 当前用户不是该节点执行人 |
| `STEP_NOT_ACTIVE` | 400 | 节点不处于可提交状态 |

**响应**：更新后的 `WorkflowTaskRecord`（含节点新状态）

---

#### GET `/issued`

**权限**：管理角色  
**功能**：返回当前发布者（按 userId + scopePath）发布的所有任务

**筛选逻辑：**
```typescript
// createdByUserId 匹配自己，或 issuerScopePath 在权限范围内
tasks.filter(task => {
  if (task.createdByUserId === userId) return true;
  if (scopePath && task.issuerScopePath) {
    return task.issuerScopePath.startsWith(scopePath)
      || scopePath.startsWith(task.issuerScopePath);
  }
  return false;
})
```

- `DEV_ADMIN` 传 `scopePath = null`，查全部任务
- 其他角色按 `scopePath` 前缀范围过滤

---

#### GET `/:taskId`

**权限**：管理角色  
**功能**：获取单个任务完整详情（含每个节点的 `stepAnswers`）

**说明**：`/issued` 接口返回的任务不含答案详情，点击「查看详情」时调用本接口获取完整数据。

### 8.4 前端服务层（workflowTask.ts）

```typescript

  // 初始化发布页数据
  bootstrap: () => api.get<WorkflowBootstrapPayload>(".../bootstrap"),

  // 搜索执行人
  searchAssignees: (keyword: string) => api.get<WorkflowAssigneeOption[]>(".../assignees/search?keyword=..."),

  // 创建流转任务
  create: (payload: WorkflowCreateInput) => api.post<WorkflowBootstrapTask>(".../workflows", payload),

  // 获取我参与的流转任务（执行层）
  myTasks: () => api.get<WorkflowMyTask[]>(".../workflows/mine"),

  // 提交节点答案
  submitStep: (taskId, stepId, answers) => api.post<WorkflowMyTask>(".../submit", { answers }),

  // 获取我发布的任务（看板）
  issuedTasks: () => api.get<WorkflowBootstrapTask[]>(".../workflows/issued"),

  // 获取单任务完整详情（含答案）
  getTaskDetail: (taskId) => api.get<WorkflowMyTask>(`.../${taskId}`),
}
```

## 10. 数据存储说明

### 10.1 存储方案

流转任务**不使用 MySQL/Prisma**，直接读写 JSON 文件：

```
backend/data/workflow-tasks.json
```

**文件结构：**
```json
{
  "tasks": [
    {
      "id": "uuid-xxx",
      "mode": "workflow",
      "title": "...",
      "steps": [...],
      ...
    }
  ]
}
```

### 10.2 存储层 API

| 函数 | 说明 |
|------|------|
| `listWorkflowTasks()` | 返回全部任务，按 `createdAt` 倒序 |
| `getWorkflowTasksForUser(userId)` | 返回该用户作为执行人的任务 |
| `getWorkflowTasksByIssuer(userId, scopePath)` | 返回发布者看板数据 |
| `getWorkflowTaskById(taskId)` | 按 ID 查单个任务（含完整答案） |
| `createWorkflowTask(input)` | 创建任务，初始化节点状态，写入文件 |
| `submitWorkflowStep(taskId, stepId, userId, answers)` | 提交节点，自动流转，写入文件 |

### 10.3 并发说明

当前 JSON 文件存储为**顺序读写**，不支持并发安全。生产环境建议：
- 迁移到 MySQL（新增 `WorkflowTask` 和 `WorkflowTaskStep` 表）
- 或引入文件锁 / Redis 缓存

## 11. 业务闭环全链路

```
┌─────────────────────────────────────────────────────────────────┐
│                        完整业务闭环                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 发布（管理员）                                               │
│     任务发布 → 协同任务 → 流转模式                               │
│     填写：标题/描述/截止时间 + 节点列表（执行人+题目）             │
│     POST /workflows → 写入 JSON，step[0] 自动激活               │
│                                                                 │
│  2. 执行（节点执行人，按顺序）                                    │
│     待办任务 → 我的待办 → 流转任务卡片                            │
│     GET /workflows/mine → 看到自己负责的任务                    │
│     active 节点：填写答案 → 提交                                 │
│     POST /steps/:stepId/submit → 节点完成，下一节点自动激活      │
│                                                                 │
│  3. 流转（自动）                                                 │
│     step[0] ✓ → step[1] active → step[1] ✓ → step[2] active   │
│     ... 直至最后一个节点完成 → task.status = "completed"        │
│                                                                 │
│  4. 看板（发布者）                                               │
│     任务看板 → 协同任务看板                                      │
│     GET /workflows/issued → 我发布的全部任务列表                 │
│     点击查看详情 → GET /workflows/:taskId → 节点答案详情          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 12. 常见问题与注意事项

### Q1：执行人为什么看不到任务？

- 检查该用户的 `userId` 是否在任意节点的 `assigneeUserId` 中
- 检查用户账号状态是否为 `active`
- 确认已登录（`authRequired`）

### Q2：节点提交后下一节点没有激活？

- 确认 `workflow.store.ts` 中 `submitWorkflowStep` 包含自动流转逻辑
- 检查下一节点的 `order === currentStep.order + 1`
- 检查下一节点初始状态是否为 `pending`（创建时 `index === 0 ? "active" : "pending"`）

### Q3：为什么协同任务看板不显示在导航？

- 检查 `AppLayout.tsx` 的 `TaskBoardNavGroup` 中，`visibleItems` 过滤函数是否包含 `/tasks/dashboard/workflow-board` 的判断分支
- 确认当前身份的 `roleCode` 在允许列表内

### Q4：发布任务时提示"执行账号不在管理范围内"？

- 非 `DEV_ADMIN` 的发布者只能选择 `scopePath` 下的执行人
- 目标执行人的 `userIdentity.scopePath` 必须以发布者的 `scopePath` 开头（或相等）

### Q5：如何扩展新题型？

1. 在 `workflow.store.ts` 的 `WorkflowTaskQuestionType` 中添加新类型
2. 在 `workflow.routes.ts` 的 `itemType` 字段校验处添加新类型
3. 在前端 `WorkflowTaskPage.tsx` 的题型选项配置中添加
4. 在 `WorkflowTaskCard.tsx` 的答题渲染逻辑中添加对应 UI
5. 在 `WorkflowBoardPage.tsx` 的 `answerDisplay` 函数中添加答案展示逻辑
