# 数据库自动备份开发者文档

> 项目：PendingWeb  
> 最后更新：2026-06-13  
> 维护人：开发团队

---

## 一、概述

本系统为 `D:\Pendingweb` 项目提供全自动双重备份能力，涵盖：

- **MySQL 数据库**（mysqldump 导出 .sql 文件）
- **用户上传文件**（`backend/uploads` 目录，robocopy 镜像同步）

每次备份同时写入 **D盘**（主备份）和 **C盘**（副备份），任意一个硬盘故障均不会丢失数据。

---

## 二、文件结构

```
D:\Pendingweb\
├── auto-backup.bat              # 主备份脚本（定时任务调用入口）
├── db-backup.bat                # 手动备份脚本（交互式，保留原有）
└── db-restore.bat               # 数据库恢复脚本

D:\Pendingweb-back-maysql\
├── db-backups\                  # D盘数据库备份目录
│   └── anchor_todo_YYYYMMDD_HHMM.sql
└── uploads-backups\             # D盘 uploads 备份目录

C:\Pendingweb-back-maysql\
├── db-backups\                  # C盘数据库备份目录
│   └── anchor_todo_YYYYMMDD_HHMM.sql
├── uploads-backups\             # C盘 uploads 备份目录
└── backup.log                   # 备份日志（所有执行记录）
```

## 三、备份配置

| 配置项 | 值 |
|--------|-----|
| 数据库主机 | localhost:3306 |
| 数据库用户 | root |
| 数据库名称 | anchor_todo |
| MySQL 路径 | `C:\Program Files\MySQL\MySQL Server 8.0\bin` |
| uploads 源目录 | `D:\Pendingweb\backend\uploads` |
| 每盘保留备份数 | 7 个（超出自动删除最旧的） |
| uploads 同步策略 | 镜像同步（/MIR，源删则备份同步删除） |

## 四、定时任务

系统注册了两个 Windows 任务计划程序任务：

| 任务名称 | 触发时间 | 状态 |
|----------|---------|------|
| PendingWeb备份-22点 | 每天 22:00 | Ready |
| PendingWeb备份-凌晨5点 | 每天 05:00 | Ready |

两个任务均调用同一脚本：`D:\Pendingweb\auto-backup.bat`

**任务属性：**
- 以最高权限运行（管理员级别）
- 如果错过计划时间，开机后尽快执行（`StartWhenAvailable`）
- 最长执行时间：30 分钟
- 失败后每 5 分钟自动重试，最多重试 3 次

---

## 五、备份脚本执行流程

```
auto-backup.bat 启动
│
├─ [1/4] mysqldump 导出数据库 → D:\Pendingweb\db-backups\anchor_todo_YYYYMMDD_HHMM.sql
│         ↓ 失败则写错误日志并退出
│
├─ [2/4] copy SQL 文件 → C:\ServerBackup\db-backups\（C盘副本）
│         ↓ 失败仅写警告，不中断流程
│
├─ [3/4] robocopy /MIR 同步 uploads → D:\Pendingweb\uploads-backups\
│
├─ [4/4] robocopy /MIR 同步 uploads → C:\ServerBackup\uploads-backups\
│
└─ 清理：D盘和C盘各自保留最新 7 个 SQL，超出的自动删除
```

## 七、手动执行备份

需要立即备份时，以管理员身份运行：

```powershell
& "D:\Pendingweb\auto-backup.bat"
```

或者双击 `db-backup.bat`（交互式，仅备份数据库到D盘，用于手动存档）。

---

## 八、数据库恢复

双击或运行 `db-restore.bat`，脚本会：

1. 列出 `db-backups\` 下所有可用备份（最新在上）
2. 输入编号选择要恢复的文件
3. 输入 `YES` 确认后执行恢复

> **警告：** 恢复操作会覆盖当前数据库所有数据，操作前请确认已有最新备份。

## 九、常见问题排查

### 备份失败：DB export FAILED

**可能原因：**
1. MySQL 服务未启动
2. mysqldump 路径不正确
3. 数据库密码变更

**检查步骤：**
```powershell
# 检查 MySQL 服务
Get-Service -Name "MySQL*"

# 手动测试 mysqldump
& "C:\Program Files\MySQL\MySQL Server 8.0\bin\mysqldump.exe" -uroot -padmin8888 anchor_todo > test.sql
```

### 备份失败：Copy to C drive FAILED

**可能原因：**
- C 盘空间不足
- `C:\ServerBackup\` 目录无写入权限

**检查步骤：**
```powershell
# 检查C盘剩余空间
Get-PSDrive C | Select-Object Used, Free

# 手动创建目录
mkdir C:\ServerBackup\db-backups
```

### 定时任务没有执行

```powershell
# 查看任务状态
Get-ScheduledTask -TaskName "PendingWeb备份-22点" | Select-Object TaskName, State
Get-ScheduledTask -TaskName "PendingWeb备份-凌晨5点" | Select-Object TaskName, State

# 查看最近执行历史
Get-ScheduledTaskInfo -TaskName "PendingWeb备份-22点"
```

### 重新注册定时任务

如果任务被误删，以管理员身份执行：

```powershell
$action = New-ScheduledTaskAction -Execute "D:\Pendingweb\auto-backup.bat"
$settings = New-ScheduledTaskSettingsSet -StartWhenAvailable -ExecutionTimeLimit (New-TimeSpan -Minutes 30) -RestartCount 3 -RestartInterval (New-TimeSpan -Minutes 5)
$trigger1 = New-ScheduledTaskTrigger -Daily -At "22:00"
Register-ScheduledTask -TaskName "PendingWeb备份-22点" -Action $action -Trigger $trigger1 -Settings $settings -RunLevel Highest -Force
$trigger2 = New-ScheduledTaskTrigger -Daily -At "05:00"
Register-ScheduledTask -TaskName "PendingWeb备份-凌晨5点" -Action $action -Trigger $trigger2 -Settings $settings -RunLevel Highest -Force
```

## 十、修改备份配置

所有配置集中在 `auto-backup.bat` 顶部的 `CONFIG` 区域：

```bat
:: ===== CONFIG =====
set DB_PASS=admin8888        # 修改数据库密码
set KEEP_COUNT=7             # 修改每盘保留备份数量
set UPLOADS_SRC=D:\Pendingweb\backend\uploads  # 修改源目录
set C_DB_DIR=C:\Pendingweb-back-maysql\db-backups  # 修改C盘备份路径
```

修改后无需重新注册定时任务，下次执行时自动生效。

---

## 十一、注意事项

1. **密码安全**：`auto-backup.bat` 中数据库密码为明文，请确保该文件不上传至公开代码仓库（已在 `.gitignore` 中排除）
2. **C盘空间**：uploads 备份使用镜像同步，C盘需预留与 D盘 uploads 同等大小的空间
3. **mysqldump Warning**：日志中出现 `Using a password on the command line interface can be insecure` 属于正常警告，不影响备份结果
4. **robocopy 退出码**：robocopy 返回 0-7 均为正常（0=无变化，1=有文件复制，7=包含额外信息），只有 ≥8 才表示出错
